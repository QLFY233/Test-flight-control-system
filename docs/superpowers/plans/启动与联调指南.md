# 试飞控制系统 — 启动与联调指南

> 本文档供 agent 或开发者按步骤启动完整的试飞控制系统。从零开始，每步可独立验证。

## 系统概览

```
┌─────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   前端 (浏览器)   │◄───►│  后端 A (FastAPI) │◄───►│  后端 B (ROS桥)   │◄───►│ 假无人机 (ROS) │
│   localhost:8000 │     │   venv-A 3.10+   │     │   venv-B 3.8     │     │   ROS Noetic   │
└─────────────────┘     └──────────────────┘     └──────────────────┘     └────────────────┘
```

4 个进程，必须在 WSL2 Ubuntu 20.04 内运行。

## 前提条件

| 项 | 要求 | 验证命令 |

| OS | WSL2 Ubuntu 20.04 | `lsb_release -a` |

## 第一步：创建虚拟环境

在项目根目录 (`flight-control-system/`) 下执行：

```bash
cd /path/to/flight-control-system

# ===== venv-A (Python 3.10+, 后端 A) =====
python3.10 -m venv .venv-A
source .venv-A/bin/activate
pip install -r venv-A-requirements.txt
deactivate

# ===== venv-B (Python 3.8 + ROS, 后端 B) =====
# 关键: 必须 --system-site-packages 才能访问 rospy/numpy/msgpack
python3 -m venv --system-site-packages .venv-B
source /opt/ros/noetic/setup.bash
source .venv-B/bin/activate
pip install -r venv-B-requirements.txt
deactivate
```

### 验证虚拟环境

```bash
# 验证 venv-A (Python 3.10+, 不含 rospy)
source .venv-A/bin/activate
python -c "import fastapi, pydantic_ai, sqlalchemy, msgpack; print('venv-A OK')"
deactivate

# 验证 venv-B (Python 3.8, 含 rospy + numpy)
source /opt/ros/noetic/setup.bash
source .venv-B/bin/activate
python -c "import rospy, numpy, msgpack; print('venv-B OK')"
deactivate
```

预期输出: `venv-A OK` 和 `venv-B OK`。

## 第二步：S0 — msgpack 跨版本兼容验证

**这是联调第 0 步，必须通过才能继续。**

A 侧和 B 侧的 msgpack 版本不同（A 侧 pip 最新版，B 侧 apt 0.6.2），`use_bin_type` 必须一致。

```bash
# A 侧
source .venv-A/bin/activate
python -c "
import msgpack
data = msgpack.packb(b'hello', use_bin_type=True)
result = msgpack.unpackb(data, raw=False, use_bin_type=True)
assert result == b'hello', f'Expected bytes, got {type(result)}'
print('A side msgpack OK:', result)
"
deactivate

# B 侧
source /opt/ros/noetic/setup.bash
source .venv-B/bin/activate
python -c "
import msgpack
data = msgpack.packb(b'hello', use_bin_type=True)
result = msgpack.unpackb(data, raw=False, use_bin_type=True)
assert result == b'hello', f'Expected bytes, got {type(result)}'
print('B side msgpack OK:', result)
"
deactivate
```

预期: 两侧都打印 `bytes` 类型 (`b'hello'`)，不是 `str`。

## 第四步：S1 — 假无人机单跑 (验证 ROS + sim-drone)

此步骤不依赖后端 A 或 B，只验证 ROS 环境和假无人机节点。

### 4.1 构建 catkin workspace

```bash
cd /path/to/flight-control-system

# 创建软链 (如果尚未创建)
mkdir -p ros_ws/src
ln -sf "$(pwd)/sim-drone" ros_ws/src/sim-drone

# 构建
cd ros_ws
source /opt/ros/noetic/setup.bash
catkin_make
source devel/setup.bash
cd ..
```

### 4.2 启动测试

需要 3 个终端，都在 WSL2 内：

**终端 1 — roscore:**

```bash
source /opt/ros/noetic/setup.bash
roscore
```

看到 `started core service [/rosout]` 即成功。保持运行。

**终端 2 — 假无人机:**

```bash
source /opt/ros/noetic/setup.bash
source /path/to/flight-control-system/ros_ws/devel/setup.bash
rosrun sim_drone fake_drone_node.py
```

看到 `[INFO] Fake drone initialized` 即成功。保持运行。

**终端 3 — 手动发 setpoint 验证:**

```bash
source /opt/ros/noetic/setup.bash

# 发一个目标点
rostopic pub /drone/setpoint_position/local geometry_msgs/PoseStamped "
header:
  frame_id: 'map'
pose:
  position: {x: 3.0, y: 2.0, z: 1.5}
  orientation: {x: 0.0, y: 0.0, z: 0.0, w: 1.0}
" -1

# 查看无人机当前位置
rostopic echo /drone/local_position/pose -n 5
```

验收标准:
- 假无人机位姿从 (0,0,0.5) 向 (3,2,1.5) 移动
- `rostopic echo` 能看到 position 值变化
- 停止 pub 超过 0.5 秒后，无人机自动悬停（位置不再变化）

## 第五步：S2 — 后端 B 单跑 (stub 模式，无 A)

此步骤验证后端 B 的求解器能独立工作，不依赖后端 A。

```bash
# 终端 1: roscore (若未运行)
source /opt/ros/noetic/setup.bash
roscore

# 终端 2: 假无人机
source /opt/ros/noetic/setup.bash
source /path/to/flight-control-system/ros_ws/devel/setup.bash
rosrun sim_drone fake_drone_node.py

# 终端 3: 后端 B (stub 模式)
cd /path/to/flight-control-system
source /opt/ros/noetic/setup.bash
source .venv-B/bin/activate
python -m backend_B.main --stub
```

验收标准:
- B 启动后 2 秒自动注入 hardcoded 测试轨迹 (0,0,1)→(3,2,1.5)→(4,3.5,1)
- 求解器生成连续轨迹 → setpoint 下发 ROS 话题
- 假无人机按轨迹移动
- B 日志中能看到 solver tick、collision check 输出

## 第六步：S3 — A↔B IPC 通联

此步骤打通后端 A 和后端 B 的 Unix socket 通信。

### 6.1 确保 socket 路径干净

```bash
rm -f /tmp/flight_control_AB.sock
```

### 6.2 启动

需要 4 个终端：

**终端 1 — roscore:**

```bash
source /opt/ros/noetic/setup.bash
roscore
```

**终端 2 — 假无人机:**

```bash
source /opt/ros/noetic/setup.bash
source /path/to/flight-control-system/ros_ws/devel/setup.bash
rosrun sim_drone fake_drone_node.py
```

**终端 3 — 后端 B (正常模式，连 A):**

```bash
cd /path/to/flight-control-system
source /opt/ros/noetic/setup.bash
source .venv-B/bin/activate
python -m backend_B.main
```

**终端 4 — 后端 A:**

```bash
cd /path/to/flight-control-system
source .venv-A/bin/activate
python -m backend_A.main
```

### 6.3 验收 S3

1. **B 侧日志**: `IPC client connected to /tmp/flight_control_AB.sock`
2. **A 侧日志**: `IPC server: client connected`
3. **心跳正常**: A 每 2s 发 ping，B 回 pong，A 日志无断连告警
4. **位姿上行**: A 侧日志出现 `pose received` (10Hz)
5. **浏览器** 打开 `http://localhost:8000`，前端能连上 A 的 WebSocket (StatusBar 显示绿点)

若心跳不通:
- 检查 socket 路径 `/tmp/flight_control_AB.sock` 是否存在
- 确认 B 启动时 A 已经在监听 (先启 A 再启 B，或 B 会 1s 恒定重连)
- 检查 `backend_B/bus/protocol.py` 和 `backend_A/bus/protocol.py` 的 `SCHEMA_VERSION` 都是 1

> S4 (碰撞校验 reject 回路) 可在 S5 之后验证。

### 7.1 启动 (同 S3 的 4 个终端)

### 7.2 验收 S5

2. 导航到 β 页面 (`/#/beta`)

## 后续联调步骤

### S4 — 碰撞校验 + reject 回路

在 β Chat 发一个会导致碰撞的指令（如 "飞到柱子里面"），验证：
- B 求解器碰撞校验失败 → 上行 `reject`
- A 收到 → WS 推前端 reject 消息
- α 收到 reject → 下轮重想

### S6 — 监控 alert 回路

人为制造异常（如在假无人机中改速度超限），验证：
- B 监控程序检测异常 → 上行 `alert`
- A monitor_trigger 收到 → WS 推前端 alert (β 对话框系统消息)
- β 被唤醒给出处置建议

### S7 — 断连安全

- 杀 A 进程 → B 应在 1s 内切 hover → 假无人机悬停
- 杀 B 进程 → 假无人机超时悬停 (sim-drone 自带 500ms 超时)
- A 检测 IPC 断连 → 前端 WS `link_status{state:down}`

## 常见问题

### `import rospy` 失败

症状: `ModuleNotFoundError: No module named 'rospy'`

原因: venv-B 未用 `--system-site-packages` 创建，或未 source ROS setup.bash。

解决:
```bash
# 重建 venv-B
rm -rf .venv-B
python3 -m venv --system-site-packages .venv-B
source /opt/ros/noetic/setup.bash
source .venv-B/bin/activate
pip install -r venv-B-requirements.txt
```

### `roscore` 未运行

症状: 假无人机或后端 B 启动时 `Unable to register with master`

解决: 先在一个终端运行 `source /opt/ros/noetic/setup.bash && roscore`

### A↔B socket 连不上

症状: B 侧日志持续打印 `IPC: connect failed, retrying in 1s`

检查:
1. A 是否已启动 (`python -m backend_A.main`)
2. socket 文件 `/tmp/flight_control_AB.sock` 是否存在 (`ls -la /tmp/flight_control_AB.sock`)
3. 若 socket 残留，删除后重启 A: `rm -f /tmp/flight_control_AB.sock`

### 前端 3D 不显示

症状: 浏览器打开 `http://localhost:8000`，3D 场景黑屏

检查:
1. 浏览器控制台是否有 Three.js CDN 加载失败 (检查网络)
2. WebGL 是否可用: 浏览器访问 `https://get.webgl.org/`
3. 若 CDN 不可达，修改 `frontend/index.html` 中的 Three.js/ECharts CDN 链接

### DeepSeek API 调用失败

症状: A 侧日志 `LLM API error` 或 SSE 返回 `error` 事件

检查:
1. `.env` 中 `DEEPSEEK_API_KEY` 是否有效
2. 网络能否访问 `https://api.deepseek.com`
3. 账户是否有可用额度

## 进程关停顺序

```bash
# 1. 停止后端 A (Ctrl+C)
# 2. 停止后端 B (Ctrl+C) — A 关停时会发 hover 给 B
# 3. 停止假无人机 (Ctrl+C)
# 4. 停止 roscore (Ctrl+C)
```

关停后清理:
```bash
rm -f /tmp/flight_control_AB.sock
```

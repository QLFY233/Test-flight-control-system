# 试飞控制系统 — 后端 A 设计(Agent 中枢)

> 子系统 spec,基于 `2026-07-01-总体架构-design.md`。后端 A 是 Agent 中枢,运行双 agent(α/β)、托管前端、对接讯飞语音、经 IPC 连后端 B。

## 2. 双 Agent — α/β 初始化与调用

### 2.1 Agent 工厂(多 provider 扩展)

```python
# agents/llm.py
PROVIDERS = {
    "deepseek": {"base_url": "https://api.deepseek.com", "api_key_env": "DEEPSEEK_API_KEY", "model": "deepseek-v4-flash"},
    # 未来加 OpenAI 兼容格式的 provider 在这里加一项
}
def make_agent(instructions, tools=None, output_type=None) -> Agent:
    provider = os.environ.get("LLM_PROVIDER", "deepseek")
    cfg = PROVIDERS[provider]
    model = OpenAIModel(cfg["model"], base_url=cfg["base_url"], api_key=os.environ[cfg["api_key_env"]])
    return Agent(model, instructions=instructions, tools=tools, output_type=output_type)
```

- 切 provider 只改环境变量。
- 只要目标 API 是 OpenAI 兼容格式,加一项配置即可。
- API key 一律环境变量,不入库不入 config 文件。

### 2.2 β Agent — 中枢大模型(龙虾模式)

- 启动时建单例,注册工具(来自 `tools/beta_tools.py`),`output_type=FlightPlan` 或结构化分析结果。
- **中枢调度**:β 是龙虾模式的中枢大模型,通过工具调度 α/analytics/历史查询等组件。
- **触发来源**:
  - 前端 `/api/chat/beta`(人类打字/语音发起)。
  - 异常监控触发器唤醒(收到 B 上行 alert → 注入异常上下文给 β,β 给处置建议)。
- 历史按需带(默认本次 session 全部,β 低频)。
- 产出:FlightPlan(结构化,前端 FlightPlanCard 展示)/ 数据分析结果 / 处置建议 / 转发指令给 α。
- 调用经 `run_agent_with_log`(§5.5)自动写 conversations。
- **β 工具集**(经组件消息总线调用):

| 工具类别 | 工具 | 说明 |
|---|---|---|
| 历史查询 | `query_sessions`/`query_telemetry`/`query_environment`/`query_conversations` | 见总体架构 §2.4 |
| 实时状态 | `get_field_map`/`get_current_pose`/`get_recent_telemetry(window_sec)`/`get_current_environment()` | 读 AppState 缓存;`get_current_environment()` 返回本次会话 `environment_id` 对应 `environments.data`(未设返回 null) |
| α 调度 | `propose_to_alpha(intent)` | β 提议发 α,**人审核**后才注入(前端展示待审核) |
| α 调度 | `forward_last_human_message()` | 转发人刚对 β 说的指令进 α 队列,**免审核**(人已发话) |
| 数据分析 | `analytics_fft`/`analytics_stats`/`analytics_filter` | 经 `analytics/` 组件,FFT/统计/滤波。可扩展注册表 |

> `propose_to_alpha` 安全路径:β 调 → 后端把 intent 暂存 `pending_proposal` → 前端展示待审核 → 人确认 → 注入 α 队列(经总线)。`forward_last_human_message`:直接进 α 队列(经总线)。两条路径都经总线发给 `alpha` 组件,α 翻译器不感知来源差异(只认指令)。

### 2.3 α Agent — 非对话翻译器(可注册组件)

> **α 不与人对话,不输出文字给人。** 先导用 LLM 翻译,未来换小参数非 LLM 小模型(研究内容 2),实现统一 `TrajectoryTranslator` 接口,接口不变。

- 启动时建单例,无对话输出,`output_type=TrajectorySpec`(只翻译,无 reply 字段)。
- 实现 `TrajectoryTranslator` 接口(组件注册表项 `alpha`),经消息总线接收指令。
- loop 周期处理 α 输入队列(β 转发 + 人确认的提议)+ 无人机数据(翻译上下文)。

**α loop 伪代码:**

```python
async def alpha_loop(app_state, bus):
    while True:
        await asyncio.sleep(app_state.config.alpha_loop_period)  # 1~5s 可配
        pose = app_state.current_pose
        alpha_inputs = app_state.drain_alpha_input_queue()  # β 转发 + 人确认提议
        # 仲裁
        if alpha_inputs:
            spec = await translate(alpha_agent, alpha_inputs, pose)  # 产出 TrajectorySpec
            app_state.current_trajectory_plan = spec
        elif app_state.current_trajectory_plan and app_state.current_trajectory_plan.has_remaining():
            spec = app_state.current_trajectory_plan.next_segment()
        else:
            spec = TrajectorySpec.hover(pose)  # 安全默认
        # 下发(经消息总线发 solver 组件)
        if spec:
            await bus.call(to="solver", tool="trajectory", args=spec.dict())
            await ws_push_alpha_output({"trajectory": spec})  # 完整 TrajectorySpec(前端规划层/3D 渲染,无对话文字)
        await repos.save_conversation(...)
```

**大模型非阻塞:** 翻译是 await,用 task 引用 + 周期检查;未完成则下发悬停或上次意图,不卡 loop。

**α 输入来源(三类,无人直接打字):**
1. β 的 `forward_last_human_message` 转发(人话指令,免审核,直接进队列)。
2. 人审核通过的 β 提议(`propose_to_alpha`,经人确认后进队列)。
3. 无人机数据(B 上行 pose/telemetry,loop 每周期读,作翻译上下文)。

> 前端已取消 α 对话框,人只与 β 对话;α 不产对话文字,只经 WS `alpha_output` 系统消息推送**完整 TrajectorySpec**(前端规划层/3D 渲染用,非 Chat)。

### 2.4 β 提议审核与 α 注入

- β 调 `propose_to_alpha(intent)` → 后端存 `app_state.pending_proposal` → 前端展示待审核 → 人确认 → `app_state.alpha_input_queue.append(intent)` → α loop 下周期翻译。
- β 调 `forward_last_human_message()` → 取 `app_state.last_human_message_to_beta` → 直接入 α 队列(免审核)。
- 两条路径都经消息总线发给 `alpha` 组件,α 翻译器只认指令,不区分来源(安全边界在总线层保证:propose 必须人确认才进队列)。

### 2.5 异常监控触发器(monitor-trigger,后端 A 侧桥接)

> 监控程序本身在后端 B(读 ROS 数据流),后端 A 侧是**触发器**:接收 B 上行的 alert 事件,转发前端 + 唤醒 β。

- 后端 A 新增 `monitor-trigger/` 模块,监听总线 `alert` 事件。
- 收到 alert → WS 推前端 `alert` 消息(前端作为**系统消息插 β 对话流**)+ 注入 β 上下文(唤醒 β 给处置建议)。
- β 收到异常上下文 → 分析(调 analytics 看近期数据)+ 给处置建议 → 经 WS/B.chat 推前端。
- **不自动执行飞行处置**(安全边界):β 的处置建议需人确认才(经 propose_to_alpha)执行。

### 2.6 数据分析组件(analytics,后端 A 侧)

- `analytics/` 模块,可注册组件集(FFT/统计/滤波等),实现 `AnalyticsTool` 接口。
- β 经总线 `call` 调用,如 `bus.call(to="fft_analyzer", tool="fft", args={...})`。
- 先导预定义工具,后期可扩(含代码执行模式,留接口)。
- 数据来源:β 调 `get_recent_telemetry` 或 `query_telemetry` 取数据 → 传给 analytics 工具。

## 3. 共享状态 AppState 与组件消息总线

### 3.1 AppState(`state.py`)

```python
@dataclass
class AppState:
    config: Config
    session_id: str | None = None
    current_pose: PoseData | None = None
    # α 输入队列(β forward + 人确认的 propose,经总线写入)
    alpha_input_queue: list[str] = field(default_factory=list)
    current_trajectory_plan: TrajectoryPlan | None = None
    # β 待审核提议(propose_to_alpha 产出,等人确认)
    pending_proposal: FlightPlan | None = None
    # 人最近对 β 说的话(forward_last_human_message 取用)
    last_human_message_to_beta: str | None = None
    pending_translation: asyncio.Task | None = None
    last_intent: TrajectorySpec | None = None
    ipc_connected: bool = False
    flight_status: str = "idle"  # idle/hovering/planned/executing/completed/aborted
```

并发保护:`asyncio.Lock`;高频位姿更新用单独锁,避免与低频队列操作互锁。

### 3.2 组件消息总线(`bus/`)

统一消息格式(总体架构 §3.7):`{schema_version, from, to, msg_type, call_id, tool, args, payload, ts}`。组件经总线发现与通信。

**`bus/router.py`:** 内存路由,`bus.call(to, tool, args)` → 查注册表 → 调对应组件 → 返回 result(同 call_id)。A 内组件(α/analytics/history_query)注册在此。

**`bus/bridge.py`:** A↔B 跨进程桥接。A 内组件要调 B 侧组件(solver/monitor)时,`bus.call(to="solver", ...)` 发现 `solver` 在 B → 经 IPC 桥接转发 → B 侧总线路由到 solver。反向同理(B→A 的 event 经桥接进 A 总线)。

**`bus/registry.py`:** 组件静态注册表(启动时注册),声明组件名+接受 tool+实现。

**`bus/protocol.py`:** 消息类型常量 + `SCHEMA_VERSION`。

### 3.3 A↔B IPC 桥接(`ipc/server.py` + `frames.py`)

A 作 server,接受 B 连接,传输统一总线消息(msgpack + 长度前缀帧):

```python
async def send_frame(writer, msg):
    payload = msgpack.packb(msg)  # 统一总线消息
    writer.write(struct.pack(">I", len(payload)) + payload)
    await writer.drain()
```

- B 侧恒定时间重连(非指数退避)。
- 心跳:A 每 2s 发 `ping` event,B 回 `pong`;A 超 5s 无 pong 视为断连。
- A 收到 B 的总线消息(event)→ 注入 A 内总线分发(pose→AppState+WS, alert→monitor-trigger, ...).

### 3.4 B→A 事件分发(A 侧总线处理)

| `tool`(msg_type=event) | 处理 |
|---|---|
| `pose` | 更新 `current_pose`(加锁)+ 转发前端 WS + 进遥测缓冲 |
| `telemetry` | 进遥测缓冲(每秒批量入库) |
| `status` | 更新 `flight_status` + 转发前端 WS |
| `reject` | 注入 α 上下文(下轮重想)+ 转发前端 WS |
| `pong` | 更新心跳时间戳 |
| `alert` | monitor-trigger 处理:WS 推前端 alert(β 系统消息)+ 注入 β 上下文唤醒 |

### 3.5 β→α 经总线调用(A 内内存)

- β 调 `propose_to_alpha(intent)` → `bus.call(to="alpha", tool="translate", args={intent})` 需人确认(总线层拦截:propose 先存 pending_proposal,人确认后才真正调 alpha).
- β 调 `forward_last_human_message()` → 直接 `bus.call(to="alpha", tool="translate", args={last_human_message})`(免审核).
- α 组件接收 `translate` 调用 → 进 `alpha_input_queue` → loop 下周期处理.

**`POST /api/sessions/{id}/abort` 内部处理**(高实时紧急中止,延迟预算 < 100ms,同机 IPC):
1. 经总线 `bus.call(to="solver", tool="abort")`(B 侧立即清当前轨迹缓存并切 hover,不下发新位置;优先于 `hover`,因 abort 同时清缓存与 α 输入)。
2. 同步 `app_state.alpha_input_queue.clear()`、`app_state.current_trajectory_plan=None`、`app_state.pending_proposal=None`。
3. `UPDATE flight_sessions SET status='aborted'`(§5.2)。
4. WS 广播 `status{flightStatus:'aborted'}`(BottomBar 紧急中止样式)。
5. 不经 α 翻译(abort 是控制层操作,不走 LLM);响应 `{status:'aborted'}`。

**`GET /api/overview` 状态探测**(G7):
- `backendA`:恒 `ok`(自身)。
- `backendB`:依 A↔B 心跳 `last_pong_at`,<5s → `ok`,否则 `down`(§3.6)。
- `drone`:B 上行 `pose` 最近时间 `current_pose.updated_at`,<1s(10Hz,缺 2 帧)→ `online`,否则 `offline`。
- `llm`:维护 `last_llm_call_ok` 标志,上次 DeepSeek 调用成功 → `ok`,失败 → `error`;首次未调用 → `ok`(惰性)。

### 4.2 SSE — Chat 流式(仅 β)

> α 不对话,无 Chat 端点。人类只与 β 对话。β 的 SSE 请求驱动流式(Pydantic AI UIAdapter)。

```python
@router.post("/api/chat/beta")
async def chat_beta(req: ChatRequest):
    # 人类语音转文字后也进这里(文字内容)
    app_state.last_human_message_to_beta = req.message  # forward 可引用
    async def stream():
        async for chunk in run_agent_with_log(beta_agent, req.session_id, 'beta', req.message):
            yield SSE_FORMAT(chunk)
    return StreamingResponse(stream(), media_type=SSE_CONTENT_TYPE)
```

- 人类打字/语音文字都走 `/api/chat/beta`。
- `last_human_message_to_beta` 存入 AppState,供 `forward_last_human_message` 取用。

**SSE 事件枚举**(`SSE_FORMAT` 输出事件,与前端详细设计 §4.2 逐字对齐):

| `type` | 字段 | 说明 |
|---|---|---|
| `text` | `content`(markdown 片段) | 流式文字片段 |
| `tool_call_start` | `toolName`, `toolArgs` | β 工具调用开始(如 `get_field_map` / `propose_to_alpha`) |
| `tool_call_result` | `toolName`, `result` | 工具返回(渲染为卡片;历史查询/analytics 结果) |
| `plan` | `segments`(FlightPlanSegment[]), `summary` | β 产出 FlightPlan(FlightPlanCard 渲染,对应 `pending_proposal`);人审走 `/api/proposals/{id}/approve` |
| `error` | `message` | 本回合失败(对齐 §6.3) |

> `plan` 事件 ≠ 已批准执行:仅 β 提议进入待审,前端 FlightPlanCard [批准] 调 `/api/proposals/{id}/approve`(原 `/api/plan/approve` 已废弃,统一走 propose 路径,C3)。

**语音自动发送模式**(A16,前端设置 "语音自动发送" 开关,`auto_send` 随首帧 `voice_frame` 上送):
- `auto_send=true`:STT 末帧后 A 自动注入 β 上下文并开启 `/api/chat/beta` SSE 流式回复(**前端无需再 POST**),前端仅订阅已有 SSE。
- `auto_send=false`:STT 末帧后 A 仅 WS `voice_stt_result(is_final=true)` 回填输入框,不触发 β;人按发送走常规 `POST /api/chat/beta`。

### 4.3 WebSocket(`/ws`)

**上行(前端→A):**

| 消息 | 说明 |
|---|---|
| `{type:"voice_frame", schema_version, data:<base64 PCM 16k/16bit/mono>, seq, ts}` | 语音输入,每帧 ~1280 字节(40ms) |
| `{type:"tts_request", schema_version, text}` | 请求朗读 |
| `{type:"sync", schema_version, ts}` | 重连后状态同步 |

**下行(A→前端):**

| 消息 | 说明 |
|---|---|
| `{type:"pose", schema_version, pos, quat, vel, accel, angularVel, ts}` | 位姿(10Hz) |
| `{type:"video_frame", schema_version, data, seq, ts}` | 视频 |
| `{type:"status", schema_version, flightStatus, mode, ...}` | 任务进度 |
| `{type:"reject", schema_version, reason, segmentIndex, ...}` | 求解器拒绝 |
| `{type:"link_status", schema_version, link, state, detail?}` | 链路告警 |
| `{type:"voice_tts", schema_version, data:<base64 mp3>, format:"mp3", seq}` | TTS 音频 |
| `{type:"voice_stt_result", schema_version, text, is_final}` | STT 识别结果(流式回填,支持动态修正) |
| `{type:"alpha_output", schema_version, trajectory:TrajectorySpec}` | α loop 系统消息(**完整 TrajectorySpec**,前端规划层/3D 渲染用,无对话文字) |
| `{type:"alert", schema_version, level, code, detail, suggestion?}` | 异常告警(前端作为**系统消息插 β 对话流**) |
| `{type:"sync_response", schema_version, pose, quat, status}` | 响应 sync |

> **`sync` 处理**(G5,前端重连后状态同步):A 收到 WS `{type:"sync"}` → 聚合 AppState `current_pose` + `flight.*` + `current_trajectory_plan` → 回 `sync_response{pose, quat, status}`(一次性补齐前端缺失状态,避免重连后 3D/进度空白)。

### 4.4 语音处理

**STT(`speech/stt_client.py`):**

```
前端按住语音按钮 → 采 PCM 16k/16bit/mono → 每 40ms base64 一帧 → WS voice_frame 上行
后端 A:
  1. 首个 voice_frame → 生成签名(auth.py) → 连 wss://iat.xf-yun.com/v1?...
  2. 发首帧 JSON(header.status=0, parameter.iat.domain=slm/language=zh_cn/dwa=wpgs, payload.audio.status=0)
  3. 后续 voice_frame → 转发(中间帧 status=1, seq 递增)
  4. 松开/60s 上限 → 发末帧(status=2, audio="")
  5. 收讯飞流式返回 → 解析 payload.result.text(base64→JSON: sn/ws/cw)
  6. 处理动态修正(dwa=wpgs):apd=追加, rpl=替换[rg]范围
  7. 拼接文本 → WS voice_stt_result(is_final=false) 实时回填前端(β 输入框)
  8. 末帧返回(status=2) → 最终文本 → voice_stt_result(is_final=true)
  9. 依首帧携带的 `auto_send` 开关(A16):
     - auto_send=true:最终文本注入 β 上下文,A 自动开启 `/api/chat/beta` SSE 流式回复(前端无需再 POST)
     - auto_send=false:仅 WS voice_stt_result(is_final=true) 回填输入框,不触发 β,等人 POST
```

**TTS(`speech/tts_client.py`):**

```
β 产出回复且前端开启朗读 / 前端发 tts_request
后端 A:
  1. 连 wss://cbm01.cn-huabei-1.xf-yun.com/v1/private/mcd9m97e6
  2. 鉴权方式一:header x-api-key: $XF_API_PASSWORD(免签名)
  3. 流式发文本(>10字可分段,status 0/1/2,seq 递增;parameter.tts.vcn=x6_lingxiaoxuan_flow, audio.encoding=lame, sample_rate=24000)
  4. 收流式 base64 mp3 帧 → WS voice_tts(format=mp3, seq) 下发
  5. 前端 AudioPlayer 解码 mp3 播放
```
> α 不产对话文字,故 TTS 来源为:β 回复(前端控制是否朗读)+ 前端主动 tts_request。异常告警文字也可触发 TTS。

**鉴权(`speech/auth.py`):**
- STT 必须签名:`signature_origin = "host: iat.xf-yun.com\ndate: {date}\nGET /v1 HTTP/1.1"`,hmac-sha256(apiSecret),base64,拼 authorization。date 用 RFC1123 UTC,偏差≤300s。
- TTS 用 `x-api-key: APIPassword` 方式一,免签名。

**密钥(`speech/xfyun_config.py`):** `XF_APP_ID`/`XF_API_KEY`/`XF_API_SECRET`/`XF_API_PASSWORD` 从环境变量读,不入库不入 config。

**音频格式契约:**

| 环节 | 格式 |
|---|---|
| 前端采集 | PCM 16k/16bit/单声道 |
| 前端→A 上行 | base64 PCM,每帧 ~1280 字节(40ms) |
| A→讯飞 STT | 原样转发(encoding=raw) |
| 讯飞→A TTS 返回 | base64 mp3(lame,24k) |
| A→前端 TTS 下行 | base64 mp3,format=mp3 |

## 6. 配置加载、生命周期、错误处理

### 6.1 配置加载(`config_loader.py`)

Pydantic 模型校验(`FieldConfig`/`DefaultConstraints`/`Config`),启动时加载 field.yaml + default_constraints.yaml 到内存,存入 AppState。校验失败立即报错退出。Config 含 `alpha_loop_period`(默认 2.0s)、`alpha_history_rounds`(默认 10)。

### 6.2 生命周期(`lifecycle.py`)

**启动 9 步(Uvicorn lifespan):**
1. 加载配置(Pydantic 校验)→ Config 单例
2. 初始化 SQLAlchemy engine + session factory
3. `create_all()` 建表
4. 初始化 AppState(dataclass + Lock)
5. 初始化 α/β Agent(读 prompts,注册 β 工具)
6. 启动 A↔B IPC server(监听 Unix socket)
7. 启动 α loop 后台 asyncio Task
8. 启动心跳(2s ping)
9. 开始接收前端请求

**关停:**
1. 停止接收新请求
2. cancel α loop task(等当前周期结束)
3. 经 IPC 给 B 发 hover(安全悬停)
4. 关闭 IPC(通知 B)
5. 关闭 SQLAlchemy engine
6. 清理资源

### 6.3 错误处理矩阵

| 场景 | 处理 |
|---|---|
| 配置文件缺失/校验失败 | 启动 raise,服务不启动 |
| LLM API 失败 | α 当前轮翻译失败→下发悬停(保安全)+ WS link_status 告警;β 当前回合失败→SSE 返回错误事件 |
| 讯飞 STT WS 断开 | 当前识别中止,前端 voice_stt_result 错误状态;不影响飞行 |
| 讯飞 TTS 失败 | 跳过本次朗读,记日志;不影响飞行 |
| A↔B IPC 断开 | ipc_connected=False;α loop 下发悬停(意图丢弃但不崩);前端 WS link_status state=down |
| B 心跳超时(5s) | 视为断开,同上 |
| 数据库写失败 | 记日志,重试 1 次;遥测丢失不致命,对话丢失告警 |
| WS 客户端断开 | 清理该连接订阅;不影响其他 |
| α loop 异常 | catch + 记日志 + **退避重启**(1~5s 退避,避免疯狂重启);期间 B 收不到意图维持悬停 |

**核心:** 任何失败不让服务崩,降级到安全悬停 + 告警。

# 试飞控制系统 — 开放式接口规范(组件接入标准)

> 研究产出 4:"一份定义组件接入标准的文档,指导后续新模型与程序的快速集成。"对应研究内容 5:"开放式架构设计:定义统一的组件注册、发现与通信协议,使中枢大模型能调度各类小模型与程序,实现功能扩展。"
>
> 本 spec 是总体架构 §3.7 + A-B 接口冻结 §1-§5 的**独立固化**,供"后续新模型/程序快速集成"使用。2026-07-07 本轮交付(对应甲方要求"完成开放式接口规范")。
>
> **定位**:给"想接入系统的新组件作者"看的接入手册。读完本文件,应能实现一个新组件(如新的 analytics 工具、新的小模型、新的检测器)并让 β 调度它,不动框架。

## 1. 架构总览(龙虾模式)

```
                ┌──────────────────────────────┐
                │   Agent β(中枢大模型)        │
                │   经消息总线调度各组件         │
                └──┬───────────────────────────┘
                   │ 统一消息总线(§2)
      ┌────────────┼────────────┬────────────┐
      ▼            ▼            ▼            ▼
  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
  │ alpha  │  │small_  │  │analytics│ │monitor │  ... 可注册组件
  │(动作   │  │model   │  │(FFT/   │  │(阈值/  │
  │ 翻译)  │  │(目标点)│  │ 统计)  │  │ 趋势) │
  └────────┘  └────────┘  └────────┘  └────────┘
      │           │
      │ ActionCommand
      ▼           ▼ 目标点
  ┌────────┐  ┌──────────┐
  │ ego-   │  │ 假无人机/ │
  │planner │  │ PX4       │
  │(阶段2/4)│  │           │
  └────────┘  └──────────┘
```

- **中枢**:Agent β(云端 LLM),唯一调度者;人类只与 β 对话。
- **组件**:α / 端侧小模型 / ego-planner / 监控程序 / analytics / 看板驱动器 / 历史查询 / 雷达(阶段2/4)。各实现统一接口,经总线通信。
- **传输分层**:A 内内存总线(asyncio)↔ A↔B IPC(msgpack)↔ B 内内存总线(threading)↔ 阶段4 机载↔云端。组件不感知传输差异(总线抽象)。

## 2. 统一消息格式(所有组件间通信)

```json
{
  "schema_version": 2,
  "from": "beta",
  "to": "fft_analyzer",
  "msg_type": "call",
  "call_id": "uuid",
  "tool": "fft",
  "args": {"data": [...], "options": {...}},
  "payload": {},
  "ts": 1234567890.0
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `schema_version` | int | ✅ | 协议版本(当前 `2`,跨进程一致;变更需两侧同步升级) |
| `from` | str | ✅ | 发送方组件名(或 `broadcast`/`human`/`frontend`) |
| `to` | str | ✅ | 接收方组件名(或 `broadcast`);跨进程时由对方进程总线路由 |
| `msg_type` | str | ✅ | `call`(请求)/ `result`(响应)/ `event`(事件)/ `error` |
| `call_id` | str | call/result 必填 | 请求-响应配对(call 与 result 同 id) |
| `tool` | str | ✅ | 工具/动作/事件名(见 §4 注册表) |
| `args` | object | call 携带 | 调用参数 |
| `payload` | object | event 携带 | 事件数据 |
| `ts` | float | ✅ | Unix 秒时间戳 |

**规则**:
- 未知 `schema_version` → 丢弃并告警(不崩)。
- 未知 `to`(无组件注册)→ 丢弃并记日志(协议错误,不崩)。
- 未知 `tool`(组件不认识)→ 组件回 `result:error` 或忽略并告警。
- `broadcast` 消息所有组件都收,但只处理认得的 `tool`。

## 3. 传输层(分层抽象,组件不感知)

| 传输段 | 实现 | 序列化 |
|---|---|---|
| A 内组件↔组件 | 内存总线(asyncio 队列,函数语义) | Python 对象直传(或 dict) |
| A↔B 跨进程 | Unix domain socket + 4 字节大端长度前缀帧 | msgpack(`use_bin_type=True`) |
| B 内组件↔组件 | 内存总线(threading 队列) | Python 对象直传 |
| A→前端 | Web 协议(SSE/WS/REST),非本总线 | JSON(字段名按前端 §3.3,非本格式) |
| 阶段4 机载↔云端 | 远期(机载↔云端链路) | 待定(远期 spec) |

**关键**:组件只调 `bus.call(to, tool, args)` / `bus.publish(event)`,不写传输细节。总线抽象层根据 `to` 查注册表决定走内存还是 IPC。

**A↔B IPC 帧格式**(冻结,见 [A-B 接口冻结](2026-07-05-A-B-接口冻结.md) §1):
- 4 字节大端 unsigned int 长度前缀 + msgpack 载荷(防粘包,上限 16 MiB)。
- `use_bin_type=True`(两侧一致,跨 0.6/1.x 版本硬契约)。
- socket 路径 `/tmp/flight_control_AB.sock`,A 作 server,B 作 client(恒定时间重连)。

## 4. 组件注册表(静态注册,启动时)

| 组件 | 位置 | 接受 `tool`(call) | 产出 `tool`(event) | 实现接口 |
|---|---|---|---|---|
| `alpha` | A | `translate` | — | `ActionTranslator` |
| `beta` | A | (由前端/异常唤醒,不被 call) | — | — |
| `small_model` | B | `generate_goal`/`abort`/`hover` | `status`/`reject` | `GoalGenerator` |
| `ego_planner`(阶段2/4) | B | `plan` | `status`/`reject` | `Planner` |
| `lidar`(阶段2/4) | B | (无) | `pointcloud` | `Sensor` |
| `monitor` | B | (无) | `alert` | `Detector` |
| `fft_analyzer`/`stats`/`filter` | A | `fft`/`stats`/`filter` | — | `AnalyticsTool` |
| `history_query` | A | `query_sessions`/`query_telemetry`/`query_environment`/`query_conversations` | — | `QueryTool` |
| `dashboard_driver` | A | `dashboard_configure`/`dashboard_set_filter`/`dashboard_list_panels` | — | `DashboardDriver` |

> **A↔B 跨进程 call.tool 全集**(冻结,见 A-B 接口冻结 §4):`action`/`abort`/`hover`/`ping`。A 内总线的 `alpha.translate` 不跨 A↔B。
> **B→A 跨进程 event.tool 全集**(冻结,见 A-B 接口冻结 §5):`pose`/`telemetry`/`status`/`reject`/`pong`/`alert`。

## 5. 组件接口契约(每个组件实现的统一接口)

每个组件实现一个 Python 抽象基类(ABC),接受总线消息、产出总线消息/事件。

### 5.1 `ActionTranslator`(α,动作翻译器)

```python
class ActionTranslator(ABC):
    @abstractmethod
    async def translate(self, intent: str, pose: dict, env: dict) -> dict:
        """输入: 指令文本 + 当前位姿 + 环境
        输出: ActionCommand dict(总体架构 §2.2: task_id/schema_version/actions/safety_constraints)
        失败抛 TranslateError, α loop 捕获后下发 hover(ActionCommand{code:hover})"""
        ...
```

- 先导:LLM 实现(DeepSeek-V4-Flash)。
- 远期:蒸馏小模型 α 实现([2026-07-05-小模型α-design.md](2026-07-05-小模型α-design.md))。
- 接入:实现本接口 → 注册为 `alpha` 组件 → β 经 `propose_to_alpha`/`forward_last_human_message` 触发 `translate`。

### 5.2 `GoalGenerator`(端侧小模型,目标点生成)

```python
class GoalGenerator(ABC):
    @abstractmethod
    def generate(self, action: dict, pose: dict, env: dict, safety: dict) -> dict:
        """输入: 单条 ActionCommand + 当前位姿 + 离散环境 + safety_constraints
        输出: {"goal":[x,y,z], "yaw":float, "speed_max":float}
        失败抛 GoalGenError(unknown_action_code / out_of_boundary_after_clamp)"""
        ...
```

- 先导:stub 规则占位。
- 阶段4:ONNX 推理([2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md))。
- 接入:实现本接口 → 注册为 `small_model` 组件 → α 经 A↔B IPC `call.action` 触发 `generate_goal`。

### 5.3 `Planner`(ego-planner,阶段2/4)

```python
class Planner(ABC):
    @abstractmethod
    def plan(self, goal: list, pointcloud: dict) -> dict:
        """输入: 目标点 [x,y,z] + 雷达点云
        输出: 连续轨迹(采样序列)
        失败抛 PlanError(ego_planner_unreachable)"""
        ...
```

- 阶段2/4 实现,先导不注册。
- 接入:实现本接口 → 注册为 `ego_planner` 组件 → small_model 产目标点后经它避障。

### 5.4 `AnalyticsTool`(数据分析工具)

```python
class AnalyticsTool(ABC):
    name: str  # 工具名, 如 "fft"
    @abstractmethod
    def run(self, data: list, options: dict) -> dict:
        """输入: 时序数据 + 参数
        输出: 分析结果(渲染为卡片)"""
        ...
```

- 接入:实现本接口 → 注册为 `fft_analyzer`/`stats`/`filter`/新名 → β 经 `analytics_*` 工具调用。

### 5.5 `Detector`(监控检测器)

```python
class Detector(ABC):
    name: str
    @abstractmethod
    def update(self, sample: dict) -> list:
        """输入: 一个遥测样本
        输出: 0~N 条 alert 事件"""
        ...
```

- 接入:实现本接口 → 注册到 `monitor` 的检测器注册表 → monitor 每周期跑所有检测器聚合 alert。

### 5.6 `DashboardDriver`(看板驱动器)

```python
class DashboardDriver(ABC):
    @abstractmethod
    def configure(self, panel_id: str, spec: dict) -> dict:
        """输入: 面板 ID + 展示规格(数据源/图表类型/筛选器)
        输出: 配置回执;经 WS 推前端 dashboard_config 渲染"""
        ...
    @abstractmethod
    def set_filter(self, panel_id: str, filter: dict) -> dict: ...
    @abstractmethod
    def list_panels(self) -> list: ...
```

- 接入:实现本接口 → 注册为 `dashboard_driver` 组件 → β 经 `dashboard_*` 工具调用。

## 6. 新组件接入步骤(给新组件作者)

### 6.1 接入一个 A 侧组件(如新的 analytics 工具 `analytics_wavelet`)

1. **实现接口**:在 `backend-A/analytics/wavelet.py` 实现 `AnalyticsTool`(`name="wavelet"`,`run(data, options)` 返回小波分析结果)。
2. **静态注册**:在 `backend-A/bus/registry.py` 启动时 `register("wavelet_analyzer", ["wavelet"], WaveletTool())`。
3. **加 β 工具**:在 `backend-A/tools/beta_tools.py` 加 `analytics_wavelet(data, options)` 工具函数,内部 `bus.call(to="wavelet_analyzer", tool="wavelet", args={...})`。
4. **更新 β prompt**:在 `prompts/beta.md` 告知 β 有新工具 `analytics_wavelet` 可用。
5. **前端渲染**:`analytics_wavelet` 结果经 SSE `tool_call_result` 事件回前端,ChatMessage 渲染为卡片(若新数据类型,前端加卡片渲染分支)。
6. **测试**:β 在对话中调 `analytics_wavelet` → 总线路由到新组件 → 结果回 β → 前端渲染。

**不动框架**:不改总线、不改协议、不改其他组件。只加注册表一行 + 工具函数 + prompt。

### 6.2 接入一个 B 侧组件(如新的检测器 `GeofenceDetector`)

1. **实现接口**:在 `backend-B/monitor/geofence.py` 实现 `Detector`(`name="geofence"`,`update(sample)` 检查地理围栏)。
2. **注册到检测器注册表**:在 `backend-B/monitor/detector.py` `DETECTORS` 加 `GeofenceDetector()`。
3. **(若需独立组件而非检测器)**:在 `backend-B/bus/registry.py` `register("geofence", [...], GeofenceComponent())`,产 `event:alert`(code=`geofence_breach`)。
4. **测试**:monitor 每周期跑新检测器 → alert 上行 A → 前端 β 系统消息。

### 6.3 接入一个新的"小模型"(替代端侧小模型或新增)

1. **实现 `GoalGenerator`**:在 `backend-B/small_model/my_model.py` 实现 `generate(action, pose, env, safety)`。
2. **注册路由**:在 `backend-B/small_model/goal_gen.py` `make_goal_generator()` 加 `elif backend == "my": return MyGoalGenerator()`;环境变量 `SMALL_MODEL_BACKEND=my` 切换。
3. **(若新动作编码)**:在 `action_codes.py` 加编码 + 总体架构 §2.2 动作编码表登记 + α prompt 更新;`schema_version` 不动(向后兼容,旧模型遇未知编码兜底 `reject`)。
4. **测试**:`SMALL_MODEL_BACKEND=my` 跑 S2~S5 联调。

## 7. 动态发现/热加载演进路径(先导不做,留接口)

**先导**:静态注册(代码/配置列出组件),不做动态发现。

**远期演进**:
- **动态注册**:组件启动时向总线发 `event:register`(含组件名+接受 tool 列表),总线维护动态注册表。
- **服务发现**:组件可发 `call.discover`(查询某 tool 由谁提供),总线回注册表。
- **热加载**:analytics 工具支持运行时加载 Python 模块(限定目录 + 沙箱),不需重启 A。
- **版本协商**:动态注册带组件版本,总线拒绝不兼容版本。

本 spec 预留 `event:register`/`call.discover` 的 `tool` 名空间(先导不用),远期 spec 扩展。

# 试飞控制系统 — 小模型 α 设计(试飞动作小模型,蒸馏 LLM α)

> 子系统 spec,基于 `2026-07-01-总体架构-design.md`(2026-07-07 重构后)、`2026-07-02-后端A-design.md` §2.3、研究内容 2(`docs/设计目标.md`)。
>
> 本 spec 把总体架构 §2.3 末"未来换小参数非 LLM 小模型"具体化为可执行设计:**用先导 LLM α + 人审核的 `ActionCommand` 作标的,行为克隆训练一个轻量 encoder-decoder,替换 LLM α,实现 `ActionTranslator` 接口不变**。研究内容 2 字面"操纵杆/油门控制指令序列"由 PX4 内置控制器接管(总体架构 §1.3 安全边界、原始设计稿四级控制器),小模型升一层输出 `ActionCommand`(动作编码+数值),**不碰操纵杆/油门**。
>
> **2026-07-07 重构同步**:α 输出从 `TrajectorySpec` 改为 `ActionCommand`(动作编码+数值,总体架构 §2.2);接口名 `TrajectoryTranslator`→`ActionTranslator`(A spec §2.3、开放式接口规范 §5);下游由 `solver.trajectory` 改 `small_model.generate_goal`(求解器已废弃,B 侧端侧小模型翻译动作→目标点);物理安全由 B 侧 small_model 夹紧 + ego-planner(阶段2/4)在线避障兜底,不再有"求解器碰撞校验"。`schema_version` 2。详见 [2026-07-07-架构重构-执行总报告.md](../../过程文件/2026-07-07-架构重构-执行总报告.md)。
>
> 阶段定位:阶段二(核心组件开发)末启动数据积累 → 阶段三(联调)训练 → 阶段四(外场演示)前上线。**不阻塞当前后端 B/A/前端编码**,与先导 LLM α 并行不互斥(β 经总线调 `alpha.translate`,不感知后端是 LLM 还是小模型)。

---

> ⚠️ **两类"小模型"必须区分(2026-07-07 重构后尤其重要)**
>
> | 维度 | **小模型 α(本 spec)** | **端侧小模型**([2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md)) |
> |---|---|---|
> | 定位 | 蒸馏的 α:替换 LLM α,把"试飞意图文本"翻译成 `ActionCommand` | B 侧/机载组件:把 `ActionCommand`+离散环境+无人机状态翻译成**目标点** |
> | 实现接口 | `ActionTranslator`(A 内,A spec §2.3、开放式接口规范 §5) | `GoalGenerator`(B 内,B spec §2.2/§4、端侧小模型 spec §4) |
> | 在系统中的位置 | 原来是 α loop 后端,产 `ActionCommand`(`actions`/`safety_constraints`) | α 的**下游**,产目标点 `[x,y,z]`+`yaw`+`speed_max` |
> | 运行宿主 | A 侧 venv-A(Py3.10+),远期可机载 | 先导 B 侧 venv-B(Py3.8)stub;阶段4 机载 ONNX |
> | 模型规模 | T5-small 级(60M,~100~250MB ONNX) | 数 MB 的 MLP(动作编码+环境+状态→目标点) |
> | 训练标的 | (试飞意图文本, 人审 `ActionCommand`) | (`ActionCommand`+环境+状态, 小模型目标点) |
> | 训练数据来源 | conversations 表 α 的 `tool_call`(intent)/`tool_result`(ActionCommand) | conversations 表 small_model 的 `tool_call`(action/pose/env)/`tool_result`(goal) |
> | 阶段4 是否机载 | 可机载(替代云端 LLM α) | 必机载(B 内组件) |
> | 是否阻塞先导 | 否(先导用 LLM α) | 否(先导用 stub 规则) |
>
> 两者都是"小模型",但**完全不同组件、不同接口、不同宿主、不同训练标的**。本项目"机载部署"研究内容指向**端侧小模型**;本 spec 的"小模型 α"是替换云侧 LLM α 的蒸馏产物,可机载但非必须。任何把两者混为一谈(如误以为换上小模型 α 就能去掉 LLM、就能在机载跑端侧推理)都会导致架构错位。

> 与其他 spec 接口对齐:A spec §2.3(`ActionTranslator` 接口、α loop、α 输入队列)、§3.5(总线 `to="alpha", tool="translate"`)、§5.5(`run_agent_with_log` 写 conversations);总体架构 §2.2(`ActionCommand` 格式)、§3.7(组件注册表 `alpha`);A-B 接口冻结 §4 注("A 内 `alpha.translate` 不跨 A↔B")。

## 1. 定位与安全边界

### 1.1 为何要小模型(LLM 已能翻译,为何再训练)

| 价值 | 说明 |
|---|---|
| **试飞术语专精** | LLM 通才不懂"横滚 30° 保持 2 秒改平"该拆成哪些动作编码、`speed_max`/`ceiling` 该设多少、动作序列怎么分。试飞数据训出的小模型在专业动作上更准、更稳。 |
| **可机载部署/离线** | 研究内容 2 明文"为后续机载部署提供算法基础"。LLM 依赖云端 API,小模型本地推理,断网可飞——这是阶段四外场演示的前提。 |
| **可复现可审计** | 飞行安全要可审计:同样任务单应出同样 `ActionCommand`。LLM 不确定性高(温度/采样),小模型权重固定,同输入同输出,可回归测试。 |
| **延迟可控** | LLM 1~5s/次(A spec §2.3 loop 周期),小模型本地推理 50~200ms,α loop 周期可缩到 0.5s,提升外场响应。 |

### 1.2 不做什么(安全边界,对齐总体架构 §1.3)

- **不输出操纵杆/油门/电机控制量**。那是 PX4 内置控制器(位置→速度→姿态→角速度→电机)的职责,小模型**升一层**只产 `ActionCommand`(动作编码+数值 + `safety_constraints`),交给 B 侧端侧小模型翻译成目标点、再经 ego-planner(阶段2/4)/setpoint(先导)下发给飞控。
- **不绕过 B 侧端侧小模型/ego-planner**。小模型 α 产出的 `ActionCommand` 仍交 B 侧组件做夹紧与避障;B 侧 stub 遇未知动作编码或夹紧后仍越界 → 回 `reject` 上行 α 重想(总体架构 §3.6、B spec §4.1/§8)。LLM α 与小模型 α 在安全边界上一视同仁:**α 只产意图,不直飞**。
- **不直接与人对话**。α 是非对话翻译器(总体架构 §2.3、A spec §2.3),小模型 α 同样不产对话文字,只经 WS `alpha_output` 系统消息推送 `ActionCommand` 摘要(动作编码+数值+目标点)。
- **不自动执行飞行处置**。β 提议经 `propose_to_alpha` 仍需人审核才注入 α(总体架构 §1.3、A spec §2.4)。

### 1.3 与 LLM α 的并存与切换(β 无感)

```
β 经总线 bus.call(to="alpha", tool="translate", args={intent, pose, env?})
   │
   ▼
A 侧 agents/alpha.py 路由
   ├─ alpha_backend="llm"     → LLMTranslator(先导, Pydantic AI Agent)
   └─ alpha_backend="small"   → SmallModelTranslator(本 spec, ONNX 推理)
   │
   ▼
产出 ActionCommand → bus.call(to="small_model", tool="generate_goal", args={action,pose,env})
                     (经 A↔B IPC 下发 B 侧端侧小模型)
```

- `alpha_backend` 由环境变量 `ALPHA_BACKEND=llm|small` 选(A spec §6.4 风格),启动时读一次。
- 切换不改 β 工具(`propose_to_alpha`/`forward_last_human_message`)、不改总线协议、不改 B 侧接口——`ActionTranslator` 接口统一,β 与 B 侧端侧小模型都不感知 α 后端是 LLM 还是小模型。
- 阶段二全程用 `llm`(先导 + 攒数据);阶段四外场前切 `small`(已验证精度达标)。

## 4. ActionTranslator 统一接口(对齐 A spec §2.3、开放式接口规范 §5)

```python
# backend-A/agents/translator_base.py
from abc import ABC, abstractmethod

class ActionTranslator(ABC):
    """α 动作翻译器统一接口(总体架构 §3.7 组件契约, A spec §2.3, 开放式接口规范 §5)。
    输入: 指令文本 + 当前位姿(翻译上下文) + 可选环境(env, 透传给下游端侧小模型作特征)
    输出: ActionCommand(总体架构 §2.2 格式: task_id/schema_version=2/actions/safety_constraints)
    不与人对话, 不产 reply 文字。"""
    @abstractmethod
    async def translate(self, intent: str, pose: PoseData | None, env: dict | None = None) -> dict:
        """返回 ActionCommand dict(task_id/schema_version/actions[/safety_constraints])。
        失败抛 TranslateError, α loop 捕获后下发悬停(A spec §2.3 大模型非阻塞)。"""
        ...

class TranslateError(Exception): ...
```

```python
# backend-A/agents/alpha.py 路由
import os
from .alpha_llm import LLMTranslator
from .alpha_small import SmallModelTranslator

def make_translator() -> ActionTranslator:
    backend = os.environ.get("ALPHA_BACKEND", "llm")
    if backend == "small":
        return SmallModelTranslator(model_dir="models/alpha_small")
    return LLMTranslator()  # 先导
```

α loop(A spec §2.3 伪代码)调 `translator.translate(intent, pose, env)`,**不感知后端**。切换 `ALPHA_BACKEND` 不改 loop、不改总线、不改 β 工具、不改 B 侧接口。

## 5. 数据集 schema(飞行动作数据集)

### 5.1 标的来源(A spec §5.5 自动写)

A spec §5.5 `run_agent_with_log` 把每条 α 调用写进 `conversations` 表(总体架构 §5.1):
- `agent='alpha'`、`role='tool_call'`:β 转发/人确认的指令文本(intent)
- `role='tool_result'`:α 产出的 `ActionCommand` JSON

**人审核通过的 `ActionCommand` 才是正标的**(总体架构 §1.3 安全边界:`propose_to_alpha` 经人确认才注入 α;`forward_last_human_message` 免审核因人已发话)。筛选条件:
```sql
-- 训练标的: 人确认路径(propose 经审) + 免审路径(forward 人话), α 产出的 ActionCommand
SELECT c_intent.content AS intent,
       c_result.content AS action_command_json,
       c_intent.session_id, c_intent.created_at
FROM conversations c_intent
JOIN conversations c_result
  ON c_result.session_id = c_intent.session_id
 AND c_result.agent = 'alpha' AND c_result.role = 'tool_result'
 AND c_result.created_at >= c_intent.created_at
 AND c_result.created_at < c_intent.created_at + INTERVAL '10 second'
WHERE c_intent.agent = 'alpha' AND c_intent.role = 'tool_call'
  AND c_intent.metadata->'approved' = 'true'   -- 人审核通过(propose 路径)或 forward 标志
ORDER BY c_intent.created_at;
```

> `metadata.approved` 由 `run_agent_with_log` 在写入时带:`propose_to_alpha` 路径人确认后 = `true`;`forward_last_human_message` 路径 = `true`(免审=人已发话);被驳回的 = `false`/不写。α 调用写库时 `metadata` 含 `approved` 标志——**本反作用已于 2026-07-06 落地到 A spec §5.5**(A7,见 §12.1);2026-07-07 重构后 `tool_result` 内容为 `ActionCommand` JSON(原 TrajectorySpec JSON 废弃),`metadata` 另含 `action_code`/`schema_version`(对齐端侧小模型训练标的抽取需要,见端侧小模型 spec §5)。

### 5.1a 训练数据快照隔离(G6)

> A 侧运行时用 aiosqlite **异步写** `data/flight_control.db`(A spec §5.2/§5.4,每秒批量 flush 遥测、对话实时写);训练脚本(venv-A,同步 SQLAlchemy)**不应直接读运行中的同一文件**,以免读半写/锁冲突。

流程:
1. 导出前先对热库做一致性快照:`sqlite3 data/flight_control.db ".backup data/alpha_dataset/snapshot_<ts>.db"`(在线一致备份,避免拷到 WAL 未 checkpoint 的半页);或停服后 `cp`。
2. 训练/导出脚本只连快照 `data/alpha_dataset/snapshot_<ts>.db`(只读 `uri=file:...?mode=ro`),执行 §5.1 SQL → 产出 `train/val/test.jsonl`。
3. 仅 `data/alpha_dataset/` 入数据集版本(随 `meta.json` 记快照时间戳与源库 size);生产库 `flight_control.db` 不被训练侧持有写句柄。

> SQL 示例(`INTERVAL '10 second'`)为 Postgres 语法;SQLite 等价:`AND c_result.created_at <= datetime(c_intent.created_at, '+10 seconds')`。训练脚本按目标方言切换(先导 SQLite)。

### 5.2 数据集样本格式(JSONL)

```json
{"intent":"飞到 (3,2,1) 后悬停 2 秒再返回起点","action":{"task_id":"20260630143012","schema_version":2,"actions":[{"code":"goto","target":[3.0,2.0,1.0],"units":"m"},{"code":"hover","value":2.0,"units":"s"},{"code":"return_home"}],"safety_constraints":{"speed_max":1.5,"ceiling":2.5,"floor":0.3,"boundary":[[0,0,0],[5,4,3]]}},"session_id":"20260630143012","approved":true}
```

字段:
- `intent`:任务单文本(β 转发/人确认的指令)
- `action`:人审通过的 `ActionCommand`(总体架构 §2.2 逐字格式,`schema_version=2`;`actions` 是动作编码序列,`safety_constraints` 含 `speed_max`/`ceiling`/`floor`/`boundary`,不再含 `keep_clear_distance`——碰撞相关约束废弃,交阶段2/4 ego-planner 在线避障)
- `session_id`/`approved`:溯源到 conversations 记录,可审计

### 5.3 数据量目标

| 阶段 | 数据量 | 来源 |
|---|---|---|
| 先导(阶段二) | 100~300 条 | 实际试飞 + 人审 `ActionCommand`(`run_agent_with_log` 自动落库) |
| 扩量(阶段二末) | 500~2000 条 | 离线用 LLM α 批量生成 + 人抽样审(原始设计稿"约 500 条试飞") |
| 增强后(阶段三) | 2000~10000 条 | 动作参数微扰(`target` ±0.1m、`speed_max` ±10%)、同义改写("飞到"↔"移动到") |

### 5.4 数据增强(`augment.py`)

- **数值微扰**:动作 `target` ±5%、`safety_constraints` 值 ±10%(保语义不变)
- **同义改写**:LLM 离线把 `intent` 改写 3~5 种说法,`action` 不变(教模型对同一动作序列的多种表述鲁棒)
- **动作组合**:把"起飞到 1.0m"+"goto (3,2,1)"+"返航"拼成一条长任务单,`action` 是序列拼接

## 6. 模型与训练

### 6.1 模型选型(T5-small 级,encoder-decoder)

- **架构**:encoder-decoder(适合"文本 → 结构化 JSON"翻译任务,比 decoder-only 更可控)
- **基座**:T5-small(60M)/T5-base(220M) 或 BART-small;先导用 T5-small(CPU 可训可推)
- **输入**:task 单文本 + 当前位姿摘要(如 `[POS:1.2,0.5,1.0 YAW:0] 飞到 (3,2,1) 后悬停 2 秒返航`)
- **输出**:`ActionCommand` JSON(逐 token 生成,`schema_version=2`)

### 6.2 训练目标(行为克隆 + 结构化约束)

```
L = L_token + λ_safe * L_safe + λ_valid * L_valid

L_token  : 标准 cross-entropy(token 级, T5 默认)
L_safe   : B 侧安全校验惩罚(soft): 输出 ActionCommand 经 B 侧 small_model stub 夹紧后越界 /
           未知动作编码 → 加惩罚(不可微, 用 REINFORCE 估梯度或作 hard filter 丢弃)
L_valid  : JSON 语法合法率(括号/字段名/类型), 不合法 → 大惩罚
```

- 先导只用 `L_token`(纯行为克隆,复现人审标的);阶段三加 `L_safe` + `L_valid`(强化学习+约束)。
- `λ_safe`/`λ_valid` 默认 0.1,可调。
- **注意**:不再依赖求解器碰撞校验作奖励信号(求解器已废弃)。`L_safe` 改用"B 侧 small_model stub 夹紧 + 动作编码合法性"作可调用的奖励环境(对齐 B spec §4.2 stub 规则);阶段2/4 接 ego-planner 后再加"ego-planner 规划成功率"信号。

### 6.3 训练循环(`train.py` 伪代码)

```python
def train(config):
    model = T5ForConditionalGeneration.from_pretrained(config.base_model)
    tok = T5Tokenizer.from_pretrained(config.base_model)
    ds = load_dataset("data/alpha_dataset/train.jsonl", tokenize=tok)   # labels = action JSON tokens
    optim = AdamW(model.parameters(), lr=config.lr)
    for epoch in range(config.epochs):
        for batch in ds.iter(config.batch_size):
            labels = batch["action_tokens"]     # ActionCommand JSON tokens
            inputs = batch["intent_tokens"]
            out = model(input_ids=inputs, labels=labels)
            loss = out.loss
            # 阶段三加: + λ_safe * safe_loss(out, batch) + λ_valid * valid_loss(out)
            loss.backward(); optim.step(); optim.zero_grad()
        val_score = evaluate(model, "data/alpha_dataset/val.jsonl")
        if val_score > best: save_checkpoint(model)
    export_onnx(model, "models/alpha_small/model.onnx")
```

### 6.4 评测(`eval.py`)

| 指标 | 定义 | 通过阈值(先导) |
|---|---|---|
| **JSON 合法率** | 输出能 parse 成 dict 且含必填字段(`task_id`/`schema_version`/`actions`) | ≥ 95% |
| **B 侧安全校验通过率** | 输出 `ActionCommand` 经 B 侧 small_model stub 夹紧并通过动作编码合法性(未知编码 reject) | ≥ 90% |
| **exact-match** | 输出 `ActionCommand` 与人审标的逐字段一致 | ≥ 60%(先导),≥ 80%(阶段四) |
| **结构相似度** | 动作数/动作编码分布/`safety_constraints` key 集合一致(BLEU 级) | ≥ 80% |
| **延迟** | ONNX 推理 p95 | < 200ms |

exact-match 不强求 100%——同一任务单可有多个合法 `ActionCommand`(动作粒度不同但都安全),B 侧安全校验通过即合格。

## 7. ONNX 导出与推理

### 7.1 导出(`export/to_onnx.py`)

```python
def export_onnx(model, tok, out_dir):
    model.eval()
    dummy = tok("示例任务单", return_tensors="pt")
    torch.onnx.export(
        model, (dummy.input_ids, dummy.attention_mask, dummy.input_ids),
        f"{out_dir}/model.onnx",
        input_names=["input_ids", "attention_mask", "decoder_input_ids"],
        output_names=["logits"],
        dynamic_axes={"input_ids": {0: "batch", 1: "seq"}, ...},
        opset=14,
    )
    tok.save_pretrained(f"{out_dir}/tokenizer")
```

### 7.2 推理(`backend-A/agents/alpha_small.py`)

```python
class SmallModelTranslator(ActionTranslator):
    def __init__(self, model_dir="models/alpha_small"):
        self.sess = ort.InferenceSession(f"{model_dir}/model.onnx")
        self.tok = AutoTokenizer.from_pretrained(f"{model_dir}/tokenizer")

    async def translate(self, intent: str, pose: PoseData | None, env: dict | None = None) -> dict:
        prompt = self._format_prompt(intent, pose)     # env 透传, 不作为 α 输入特征(下游端侧小模型才用)
        ids = self.tok(prompt, return_tensors="np").input_ids
        # 自回归生成(ONNX: 逐 token 循环, 或用 onnxruntime gen-ai)
        tokens = self._generate(ids, max_len=512)
        cmd_json = self.tok.decode(tokens, skip_special_tokens=True)
        try:
            cmd = json.loads(cmd_json)
            self._validate_schema(cmd)        # 总体架构 §2.2 字段校验(schema_version=2, actions, safety_constraints)
            return cmd
        except (json.JSONDecodeError, SchemaError) as e:
            raise TranslateError(f"small model bad output: {e}") from None
```

- `_generate`:自回归解码(贪心或 beam search)。ONNX 不直接支持 HuggingFace `generate`,逐 token 循环调 session,或用 `onnxruntime-genai`(context7 核实,见 §11)。
- 失败抛 `TranslateError`,α loop(A spec §2.3)捕获后下发悬停(大模型非阻塞逻辑同样适用)。

### 7.3 推理延迟与资源

- T5-small ONNX CPU:prompt 50 token + 输出 200 token,~150~300ms(p95 < 500ms 满足)。
- 内存:< 500 MB(可机载)。
- 不加载 torch/onnxruntime-training,只 onnxruntime(部署轻)。

## 8. α loop 适配(小模型非阻塞,对齐 A spec §2.3)

A spec §2.3 α loop 已是 `await translate(...)` 产出 `ActionCommand`,LLM 与小模型同一接口。小模型比 LLM 快,但仍是异步(避免阻塞 event loop):

```python
# alpha loop 内(A spec §2.3 伪代码补一行)
cmd = await translator.translate(alpha_inputs, pose, env)   # env 透传给下游端侧小模型
# 小模型:onnxruntime 调用包在 asyncio.to_thread, 不阻塞 event loop
# LLM: await Pydantic AI agent.run(已是 async)
# 产出 ActionCommand → bus.call(to="small_model", tool="generate_goal", args={action:cmd, pose, env}) 经 IPC 下发 B 侧
```

- 小模型推理用 `asyncio.to_thread`(Python 3.10+),把同步 ONNX 调用包成 async。
- 超时:小模型设 1s 超时(比 LLM 5s 短),超时抛 `TranslateError` → loop 下发悬停。

## 9. 安全校验链(小模型 α 输出绝不直飞)

```
小模型 α 产出 ActionCommand dict
   │
   ├─ schema 校验(字段名/类型, schema_version=2, 总体架构 §2.2)
   │   失败 → TranslateError → α loop 下发悬停
   │
   ├─ bus.call(to="small_model", tool="generate_goal", args={action,pose,env})
   │   (经 A↔B IPC 下发 B 侧端侧小模型)
   │
   ▼
B 侧端侧小模型(B spec §4.1-§4.3,先导 stub;阶段4 真模型) + ego-planner(阶段2/4)
   ├─ stub 规则映射动作编码→目标点, 夹紧 safety_constraints.boundary/ceiling/floor/speed_max
   │   未知动作编码 → event:reject(reason="unknown_action_code") 上行 α → α 上下文注入(下轮重想)→ α loop 下发悬停
   │   夹紧后仍越界 → event:reject(reason="out_of_boundary_after_clamp") 上行 α
   ├─ 先导(阶段1): stub 产出的目标点直发 setpoint(无在线碰撞校验,先导不验避障,B spec §4.5)
   └─ 阶段2/4: 目标点交 ego-planner + 雷达点云在线避障
       规划失败(目标被障碍包围不可达) → event:reject(reason="ego_planner_unreachable") 上行 α
       成功 → 连续轨迹 setpoint 下发无人机
```

**双层防线**:小模型 α 产出经 schema 校验(A 内)+ B 侧端侧小模型夹紧/动作合法性(B 内)+ ego-planner(阶段2/4)在线避障 + 假无人机边界自保(先导)。**任一层失败都不下发非安全 setpoint,转悬停**(总体架构 §3.6)。

> **与重构前对比**:原"求解器 offline 校验整条轨迹 + solver tick 在线重校"已废弃(求解器删除)。新链路里 α 不产航点序列,B 侧 small_model 直接产目标点;碰撞不再是上层下发约束,而是在线由 ego-planner(阶段2/4)处理。先导期"安全"由软约束 + 假无人机边界自保兜底。

### 12.1 对 A spec 的反作用(需补一句)

A spec §5.5 `run_agent_with_log` 写 conversations 时,`metadata` 需含 `approved` 标志,以区分**人审核通过的 α 产出**(可作训练标的)与**被驳回/未审的**(不可作标的)。具体:
- `propose_to_alpha` 路径:人确认后注入 α → 写 `tool_call` 时 `metadata={"path":"propose","approved":true,"proposal_id":...}`
- `forward_last_human_message` 路径:人已发话 → 写 `tool_call` 时 `metadata={"path":"forward","approved":true}`(免审=人已发话)
- β 提议被人驳回 → 不进 α,无 `tool_result`,不算标的

**已落地(2026-07-06)**:A spec §5.5 末尾已补 — `run_agent_with_log` 写 α 的 `tool_call` 时 `metadata` 必带 `approved` 布尔与 `path`(`propose`/`forward`),供小模型 α 训练标的筛选(A7)。**2026-07-07 重构后追加**:`tool_result` 内容由 `TrajectorySpec JSON` 改为 `ActionCommand JSON`;`metadata` 另含 `action_code`(首条动作编码)与 `schema_version=2`(供端侧小模型训练标的抽取,见端侧小模型 spec §5)。

## 13. 不在本 spec 范围

- **操纵杆/油门/电机控制量**:PX4 内置控制器职责(原始设计稿四级控制器),小模型不碰。
- **机载部署硬件选型**:远期,本 spec 只产 ONNX 文件 + 推理代码,硬件部署另 spec。
- **小模型替代 β**:β 是中枢大模型,语义理解/调度能力远超小模型能耐,**不替代**。小模型只替 α(纯翻译)。
- **端侧小模型(B 侧动作→目标点)**:由 [2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md) 独立 spec 承担,与本项目"机载部署"研究内容直接对应;本 spec 不重复定义(见 §0 区分框)。
- **实时训练/在线学习**:小模型离线训练,权重固定可审计;在线学习会破坏可复现性,不做。
- **多模态(图像/视频)**:小模型只处理文本任务单 + 位姿数值,不接视频/图像。

## 14. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 数据量不足(< 500 条),小模型欠拟合 | LLM 离线扩量 + 数值微扰/同义改写增强(§5.4);先导用 T5-small 不 T5-base 减少参数需求 |
| 小模型输出 JSON 语法错(token 生成跑偏) | `L_valid` 大惩罚(§6.2)+ 运行时 schema 校验 + 失败转悬停(§9) |

| exact-match 期望过高(同任务多合法解) | 评测以 B 侧安全校验通过率为主,exact-match 仅参考(§6.4) |

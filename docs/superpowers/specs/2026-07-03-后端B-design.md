# 试飞控制系统 — 后端 B 设计(实时飞控桥)

> 子系统 spec,基于 `2026-07-01-总体架构-design.md`。后端 B 是实时飞控桥,跑在 **WSL2 Ubuntu 20.04 + ROS Noetic + Python 3.8** venv,经 **A↔B Unix domain socket(msgpack 长度前缀帧,统一总线消息)** 接收后端 A 下发的 TrajectorySpec,内部求解器生成连续轨迹并做场地防碰撞校验,经 ROS 话题下发"假无人机"(阶段1)或 PX4 SITL + mavros offboard(阶段2);订阅无人机位姿/IMU,以 10Hz 上行 A;监控程序读 ROS 数据流做异常检测,异常经总线上行 alert。
>
> 与后端 A 的接口契约见总体架构 §3.2(L5 链路、A↔B 消息契约)、§3.7(统一消息总线格式)、§3.4(ROS 话题契约)、§3.6(失败/断连处理)。后端 B 是 B 侧组件(`solver`/`monitor`)的宿主,经 B 内内存总线路由到实现,uplink 事件(`pose`/`telemetry`/`status`/`reject`/`pong`/`alert`)经 IPC 桥接上行 A。

## 1. 运行环境与版本锁定

### 1.1 运行宿主

- **OS**:Ubuntu 20.04 LTS,跑在 **WSL2**(Windows 11 22H2+,`%UserProfile%\.wslconfig` 含 `[wsl2] networkingMode=mirrored`)。
- **Python**:系统 Python 3.8(ROS Noetic 绑定,不允许升 3.10)。
- **ROS**:Noetic,安装在 `/opt/ros/noetic`(`source /opt/ros/noetic/setup.bash`)。
- **进程身份**:后端 B 是一个**独立的 OS 进**(与后端 A 同机,不同 venv,经 Unix socket 通信)。它本身既可作为一个 ROS 节点(内嵌 `rospy` 节点)跑,也可拆成"B 主进程 + 独立 ROS 节点";先导采用**单进程内嵌 rospy**(详见 §3.1)。

### 1.2 venv-B 依赖(对外接口契约与总体架构 §4.1/§4.2 一致)

```

  rospy / rospkg                   # 经 apt 装(ros-noetic-rospy 等), 不能 pip — 见下方 venv-B 创建方式
  mavros 1.20.1                    # apt (ros-noetic-mavros), 阶段2
  mavlink 2025.5.5                 # apt python3-msgpack 不提供, 阶段2 改 pip pymavlink — 见 §12
  msgpack 0.6.2                    # apt (python3-msgpack), 与 A 侧版本对齐 — 本机 Candidate 0.6.2-1, 未装, 需 sudo apt install (见 §12)
  numpy 1.17.4 (apt 已装,见 §12 核实回填表)  # apt python3-numpy, 系统 site-packages

> **关键约束:msgpack 跨进程一致**。B 侧 msgpack 0.6.2(apt)必须能与 A 侧 msgpack 互通。msgpack 跨大版本(0.6 vs 1.x)在 bin 类型默认行为上有差异,**IPC 帧编解码双方必须 use_bin_type 一致**(见 §4.2 协议常量)。A 侧若用 pip 最新版,**A 侧建 Packer 时显式 `use_bin_type=True`** 与 B 侧对齐;否则退化到 B 侧 `use_bin_type=False`。这是 A↔B 互通的硬性契约,联调第一步验证项。

> **venv-B 创建方式(ROS+venv 经典坑,已本机实测)**:`rospy`/`rospkg`/`std_msgs`/`geometry_msgs`/`sensor_msgs` 等随 ROS apt 装在 **`/opt/ros/noetic/lib/python3/dist-packages`**(系统级,不进 venv site-packages),**不能 pip 装进 venv**;`numpy` 同理在系统 `/usr/lib/python3/dist-packages`。故 venv-B 必须以 `--system-site-packages` 创建,且**运行时先 source ROS 再 activate venv**,让 ROS 的 dist-packages 经 PYTHONPATH 可见。
> ```bash
> # 在 WSL2 Ubuntu 20.04
> # 1) 一次性补装系统 apt 依赖(若未装,见 §12 实测:rospy 随 ROS 已装,numpy 已装;msgpack/scipy 待装)
> sudo apt install python3-msgpack python3-scipy          # 进系统 site-packages, venv 可见
> # 2) 创建 venv-B(允许访问系统 site-packages)
> python3 -m venv --system-site-packages .venv-B
> # 3) 启动顺序(也是 run_backend_b.sh 的顺序,见 §1.5):先 source ROS 再 activate venv
> source /opt/ros/noetic/setup.bash
> source .venv-B/bin/activate
> pip install pyyaml       # 其余纯 Python 依赖按 venv-B-requirements.txt 装(venv 内, 不影响系统)
> ```
> **验证(本机实测已通过)**:`source /opt/ros/noetic/setup.bash && source .venv-B/bin/activate && python -c "import rospy, numpy, msgpack; print('OK')"` 应全部成功——`rospy` 来自 `/opt/ros/noetic/lib/python3/dist-packages`(经 ROS PYTHONPATH),`numpy` 1.17.4 来自系统 site-packages,`msgpack` 0.6.2 来自系统 apt `python3-msgpack`(或 venv pip 装,二选一,版本须与 A 侧对齐 `use_bin_type=True`,见冻结 §1)。若 venv 不加 `--system-site-packages`,rospy/numpy 不可见,会全线失败。

### 1.3 与 venv-A 的隔离

- B 侧 venv 不共享 A 侧 site-packages,只经 Unix socket 传 msgpack 二进制帧。
- B 侧**禁止**(无法)使用 A 侧任何库:FastAPI / Pydantic AI / SQLAlchemy / aiosqlite 均不出现在 B 侧。
- B 侧**不**直接读写后端 A 的 SQLite 文件;历史落库由 A 侧负责(A 把 B 上行 `pose`/`telemetry` 缓冲后批量入库,总体架构 §5.2)。B 只产上行事件,不接触数据库。

```bash

## 2. B 侧组件注册表与消息总线

### 2.1 B 内内存总线(`bus/`)

后端 B 复用总体架构 §3.7 的**统一消息总线协议**:`{schema_version, from, to, msg_type, call_id, tool, args, payload, ts}`。B 内组件经内存总线通信,B 主进程的 IPC 桥把"A 发来的 call/event"注入 B 总线,把"B 组件产出的 event"上行 A。

- **`bus/protocol.py`**:`SCHEMA_VERSION = 1`(与 A 一致),消息类型常量(`call`/`result`/`event`/`error`),`MSGPACK_USE_BIN_TYPE`(默认 True,与 A 对齐,见 §4.2 联调验证),`to` 名常量(`solver`/`monitor`/`alpha`/`beta`/`broadcast`)。
- **`bus/registry.py`**:B 侧组件静态注册。启动时注册 `solver` 和 `monitor`,各声明接受 `tool` 列表(见总体架构 §3.7 表):
  - `solver` 接受 `trajectory` / `abort` / `hover`。
  - `monitor` 不接受 call(它只持续产出 `event:alert`)。
- **`bus/router.py`**:`bus.call(to, tool, args)` → 查注册表 → 同步调用组件 → 返回 `result`(同 `call_id`)。B 内是同步函数语义(solver 调用快,monitor 不被 call)。

### 2.2 B 侧组件接口契约(对齐总体架构 §3.7 "组件接口契约")

每个组件实现统一接口:接受总线消息、产出总线消息/事件。

**solver 组件(`solver/component.py`):**

| 接受 `tool` | `args` | 产出 | 说明 |
|---|---|---|---|
| `trajectory` | TrajectorySpec(dict,总体架构 §2.2 格式) | `result`(成功/拒绝)+ 内部下发 ROS | 翻译→连续轨迹→碰撞校验→下发;校验失败回 `event:reject` |
| `abort` | `{}` | `result` | 中止当前轨迹,求解器切到悬停 |
| `hover` | `{pose?}` | `result` | 安全悬停(用当前位姿或 home) |

solver 还可在执行中产出上行事件:`status`(段进度)、`reject`(段被拒)。

**monitor 组件(`monitor/component.py`):**

- 不接受 call;持续读 ROS 数据流(BState),每个检测周期产出 0~N 条 `event:alert`(经总线上行 A)。
- 实现 `Detector` 接口(总体架构 §3.7):输入数据流 → 输出 alert 事件。检测器注册表:阈值/趋势先导,后期可加 ML 检测器(开放式架构)。

### 2.3 A↔B 消息契约(对齐总体架构 §3.2,逐字对齐)

| 方向 | `msg_type` | `tool` | 含义 | B 侧处理 |
|---|---|---|---|---|
| A→B | `call` | `trajectory` | TrajectorySpec 下发求解器 | `bus.call(to="solver", tool="trajectory", args=spec)` |
| A→B | `call` | `abort` | 中止当前轨迹 | `bus.call(to="solver", tool="abort")` |
| A→B | `call` | `hover` | 安全悬停 | `bus.call(to="solver", tool="hover")` |
| A→B | `event` | `ping` | 心跳(2s) | 回 `pong` |
| B→A | `event` | `pose` | 当前位姿(10Hz) | `rosbridge/subscriber` 周期产出 |
| B→A | `event` | `telemetry` | 速度/加速度/IMU(10Hz) | 同上,可与 pose 合并或分帧 |
| B→A | `event` | `status` | 求解器/无人机状态(低频) | solver 产出 |
| B→A | `event` | `reject` | 求解器拒绝某段 | solver 产出(回告 α 重想) |
| B→A | `event` | `pong` | 心跳回应 | IPC 桥回 |
| B→A | `event` | `alert` | 监控程序检测到异常 | monitor 产出 |

- **重连**:B 侧**恒定时间重连**(非指数退避,总体架构 §3.2),socket 断开每 1s 重连一次。
- **心跳**:A 每 2s 发 `ping`,B 回 `pong`;A 超 5s 无 pong 视为断连。B 侧不主动判定 A 断连(由 A 判定,A 断连后 B 收不到新意图 → B 自降级悬停,见 §8)。

### 2.4 上行事件分发(B→A → 总线 → A 侧处理,对齐后端 A spec §3.4)

B 上行的 event 在 A 侧被分发:`pose` → AppState + WS + 遥测缓冲;`alert` → monitor-trigger → WS alert + 唤醒 β;`reject` → α 上下文 + WS。B 侧只管"按频率产 event 并推 IPC",不感知 A 侧如何用。

## 3. 进程模型与线程架构

### 3.1 单进程多线程

后端 B 在**单进程**内,因 rospy 是基于线程的回调模型,而求解器/IPC 是 asyncio 或同步阻塞,采用**混合线程模型**:

- **主线程**:rospy spin(订阅回调在此线程触发,更新 BState)。
- **求解器线程**:接收 A 下发 `trajectory` 调用,生成连续轨迹,以定频(10~50Hz)产出 setpoint 推到 ROS publisher 队列。
- **上行线程**:以 10Hz 从 BState 读位姿/遥测,封 msgpack 帧发 IPC。
- **监控线程**:以检测周期(默认 10Hz)读 BState,跑检测器,产 alert 推 IPC。
- **IPC 线程**:socket 收发循环 + 心跳响应。

> 用 `threading` + 线程安全队列(`queue.Queue`)做线程间消息传递,BState 用 `threading.Lock` 保护高频字段(位姿)。不引入 asyncio(避免在 ROS Noetic + Python 3.8 下与 rospy 事件循环冲突)。

### 3.2 线程职责表

| 线程 | 触发 | 职责 | 写 BState? |
|---|---|---|---|
| rospy 主线程 | rospy 回调 | 订阅位姿/IMU → 更新 BState | 是(加锁) |
| solver 线程 | A 的 `trajectory` call + 定频 tick | 翻译/插值/碰撞校验/下发 setpoint | 读 BState(当前位姿) |
| uplink 线程 | 10Hz 定时 | BState → IPC 帧 | 读 |
| monitor 线程 | 检测周期定时 | 跑检测器 → alert | 读 |
| ipc 线程 | socket 可读 | 解帧 → B 总线;心跳 | 读/写 ipc_connected |

### 3.3 关停顺序

1. 停止接收新 A 消息(IPC 线程停)。
2. solver 切 `hover`(下发最后意图为安全悬停)。
3. 关闭 ROS publisher(让假无人机/PX4 收到末帧后维持)。
4. rospy signal_shutdown(等回调退出)。
5. 关 IPC socket。
6. 清理资源。

> 任何关停/异常路径,确保**最后下发的 setpoint 是悬停**(总体架构 §3.6:"任何环节失联/失败,无人机进入安全悬停")。

## 4. 求解器(`solver/`)

> 求解器是 B 内最核心组件:输入 TrajectorySpec(航点序列 + 逐段约束),输出连续轨迹(位置/速度/角速度/姿态时变曲线),做场地防碰撞校验,再以定频把 setpoint 推到 ROS。

### 4.1 TrajectorySpec → 内部轨迹表示(`translator.py`)

输入(总体架构 §2.2 格式):
```json
{
  "task_id": "20260630-001",
  "schema_version": 1,
  "global_constraints": {"keep_clear_distance": 0.5, "ceiling": 2.5, "floor": 0.3},
  "segments": [ { "waypoints": [...], "segment_constraints": {...} } ]
}
```

`translator` 把它转成 `TrajectoryPlan`(内部桩数据结构):
- 一串 `Segment`,每段含起止航点、段约束、点约束。
- 约束层级解析(§4.2):点 > 段 > 全局 > 求解器默认(`default_constraints.yaml`)。
- 标记 `return_home` 段。

### 4.2 约束层级解析(`constraints.py`)

对齐总体架构 §2.2 末尾:"约束层级:点约束 > 段约束 > 全局约束 > 求解器默认。可扩展:加新约束类型就是加新 key"。

```python
# 伪代码
def resolve_constraints(point_c, segment_c, global_c, default_c):
    # 后者覆盖前者:default < global < segment < point
    merged = dict(default_c)
    merged.update(global_c or {})
    merged.update(segment_c or {})
    merged.update(point_c or {})
    return merged
```

- 求解器用**约束注册表**:认得的 key(`speed_max`/`accel_max`/`angular_velocity_max`/`keep_clear_distance`/`ceiling`/`floor`)执行;不认得的 key **忽略并告警**(上行 `status` 或 `reject`,提示未知约束)。
- 加新约束类型(未来 `jerk_max`/`attitude_limit`/`time_window`)= 加 key + 求解器加对应执行,**TrajectorySpec/总线格式不变**(开放式)。

### 4.3 连续轨迹生成(`trajectory.py`)

> 先导策略(不依赖外部库,先用 numpy/scipy 实现;若需更优轨迹后期换 NLopt):
>
> 1. **段间插值**:航点之间用线性位置插值 + 梯形速度 profile(加速段 + 匀速段 + 减速段),受 `speed_max`/`accel_max` 约束。
> 2. **时间分配**:每段距离 / `speed_max` 得最小时间;`accel_max` 限制加减速段时长。
> 3. **采样**:以 solver tick 频率(默认 20Hz,可配)采样出 `(t, pos, vel, accel, yaw)` 序列。yaw 在航点间线性插值(或 shortest-angle)。
> 4. **角速度约束**:`angular_velocity_max` 限制 yaw 变化率,超限则延长该段时间。
> 5. **返航段**:`return_home` 标记的段,目标点取场地 `home.position`。

输出 `ContinuousTrajectory`:
```python
@dataclass
class ContinuousTrajectory:
    task_id: str
    samples: list[TrajectorySample]   # (t, pos[3], vel[3], accel[3], yaw)
    total_duration: float
    segment_boundaries: list[float]   # 各段在时间轴上的分界
```

**安全保底**:`ContinuousTrajectory.hover(pose)` 产出一个"原地保持、零速度"的单点轨迹,作为无任务/A 断连时的下发。

### 4.4 场地防碰撞校验(`collision.py` + `field.py`)

对齐总体架构 §5.3 场地配置与碰撞检测表。

`field.py` 把 `field.yaml` 加载成 `Field(boundary, obstacles, home)`,obstacle 按 `type` 分发:

| `type` | 参数 | 碰撞检测方法 |
|---|---|---|
| `box` | center, size | 轨迹采样点到 AABB 距离 ≤ keep_clear_distance |
| `cylinder` | center, radius, height | 采样点到圆柱轴心 XY 距离 ≤ radius + keep_clear_distance(且 z 在 [0,height]) |
| `sphere` | center, radius | 采样点到球心距离 ≤ radius + keep_clear_distance |
| `plane` | point, normal | 采样点是否在法向禁止侧(+ keep_clear_distance 余量) |
| `capsule` | p1, p2, radius | 采样点到胶囊体轴线距离 ≤ radius + keep_clear_distance |

- 先导实现 box/cylinder/sphere/plane;**capsule 后补**(总体架构 §5.3:"先导实现 box/cylinder/sphere/plane,capsule 后补")。
- 校验范围:整条采样轨迹(每段采样点逐一校),任一点碰撞 → 该段 `reject`。
- 边界校验:采样点必须在 `boundary` x/y/z 范围内(± keep_clear_distance)。

### 4.5 solver 组件执行流程(`component.py`)

```
收到 A 的 call(trajectory, TrajectorySpec)
  ├─ translator: TrajectorySpec → TrajectoryPlan(约束层级解析)
  ├─ trajectory: 生成 ContinuousTrajectory(梯形速度 + 采样)
  ├─ collision: 逐采样点校验场地(含 boundary + obstacles + keep_clear_distance)
  │   ├ 失败 → 产 event:reject(reason, segmentIndex) 上行 A,回 result:rejected
  │   └ 成功 → 连续轨迹入"当前执行轨迹"缓存,solver 线程开始定频 tick 下发
  └─ result:ok

solver 线程 tick(20Hz):
  ├ 若有执行中轨迹且未结束:
  │   按 t 取下一个 sample → ROS publisher 下发 setpoint
  │   段切换时产 event:status(currentSegment/totalSegments)
  │   轨迹结束产 event:status(completed)
  └ 若无轨迹 / abort / hover:
      下发"当前位姿悬停"setpoint
```

- **abort**:清当前执行轨迹缓存,切悬停。
- **hover**:若 args 给了 pose 用之,否则用 BState 当前位姿;无位姿时用场地 home。

### 4.6 求解器对外不值得依赖第三方库的判断

> 先导用 numpy + scipy(若 apt 提供)手写梯形速度 profile + 线性插值,代码量小、可控、易调试。**不**在先导引入重型轨迹优化库(casadi/NLopt),除非手写方案在实跑时出现明显抖动/超调。这一选择与"先导打通链路"的目标一致(架构概要:"先做一个最简单的假无人机,跑通了再接 PX4")。

class Phase2Adapter(SetpointAdapter):       # PX4 + mavros(阶段2)

### 5.3 publisher(`publisher.py`)

- 接收 solver 线程的 sample(20Hz),转 ROS 消息发 setpoint。
- **首帧前填一个当前位置 setpoint**,避免无人机跳变(从 0/0 起跳)。
- 阶段2(PX4 offboard)需先发若干秒 setpoint 流再切 offboard 模式(标准做法),此逻辑放 adapter 内,solver 不感知。

### 5.4 subscriber(`subscriber.py`)

- 订阅位姿/速度/IMU。回调加锁写 BState。
- 同时维护"最近一次更新时间",监控线程据此判"无人机数据是否超时停产"(产 alert:drone_data_stale)。

### 5.5 10Hz 上行节流

- 上行线程以 10Hz 读 BState,产 `event:pose` 和 `event:telemetry`,封 msgpack 发 IPC。
- 节流实现:上行线程 `while True: read; pack; send; sleep(0.1)`。不依赖 ROS 的话频率,解耦 ROS 抖动与上行频率。

## 6. 监控程序(`monitor/`)

> 对齐总体架构 §1 末:"异常监控程序(monitor/):读 ROS 数据流,阈值+趋势检测,异常经总线上行 alert(B 是飞行数据源头)";研究内容 3:"开发数据监控程序,接入飞行数据流,依据预设阈值与趋势分析逻辑检测异常,并触发大模型调度,生成告警信息与处置建议。"

### 6.1 Detector 接口与注册表(`detector.py`)

```python
class Detector(Protocol):
    name: str
    def update(self, sample: TelemetrySample) -> list[Alert]: ...

DETECTORS: list[Detector] = [ThresholdDetector(), TrendDetector()]
```

- 检测器注册表:阈值/趋势先导,后期可加 ML 检测器(开放式)。
- monitor 组件每个周期跑所有检测器,聚合 alert 上行。

### 6.2 阈值检测(`thresholds.py`)

预设阈值(读 `default_constraints.yaml` 或单独 `monitor_thresholds.yaml`,先放全局默认,后期可按段覆盖):

| 检测项 | 阈值来源 | alert code |
|---|---|---|
| 位置超出 boundary + keep_clear_distance | 场地配置 + 约束 | `out_of_boundary` |
| 高度超 ceiling / 低于 floor | 约束 | `ceiling_breach`/`floor_breach` |
| 速度超 speed_max | 约束 | `overspeed` |
| 加速度超 accel_max | 约束 | `overaccel` |
| 角速度超 angular_velocity_max | 约束 | `over_angular` |
| 无人机数据停产(>500ms 无更新) | 固定 | `drone_data_stale` |
| 求解器下发的 setpoint 与实际位姿持续偏差大 | 趋势 | `tracking_error` |

### 6.3 趋势检测(`trends.py`)

- **突变**:相邻样本二阶差分超阈值(抖动)。
- **持续偏离**:滑窗内 setpoint vs actual 偏差均值超阈值(轨迹跟不住)。
- 先导用 numpy 滑窗统计;后期可换频域(与 A 侧 analytics/FFT 联动,但 B 侧只产原始 alert,FFT 分析在 A 侧)。

### 6.4 alert 事件格式(对齐前端 AlertMessage,见前端详细设计 §3.3)

```python
# B 上行 event:alert 的 payload
{
  "level": "warn",         # info/warn/critical
  "code": "overspeed",
  "detail": "速度 2.1 m/s 超过 1.5 m/s",
  "suggestion": None,        # B 不给建议,β 给(被唤醒后)
  "ts": 1234567890,
  "segment_index": 2        # 可选, 若执行中
}
```

- **B 只告警不建议直接执行**(总体架构 §1.3 安全边界):alert 上行 A → A 转发前端(β 对话框系统消息) + 唤醒 β 给处置建议 → 处置需人确认才执行。
- monitor 产 alert 的节流:同一 `code` 在 N 秒内只上告一次(防刷屏,N 可配,默认 2s),critical 级别不节流。

### 6.5 监控程序本身故障(对齐总体架构 §3.6)

- monitor 线程异常 → catch + 记日志 + 告警前端(上行一条 `code=monitor_down` alert)+ 退避重启(1~5s)。
- 不影响飞行(降级:无监控期间只靠阈值硬终止由求解器内置的"超 boundary 直接 reject/abort"保底)。

## 7. 共享状态 BState(`state.py`)

```python
@dataclass
class BState:
    # 高频(ROS 回调写, 加锁)
    current_pose: PoseData | None         # 来自 /drone/local_position/pose
    current_velocity: VelocityData | None # 来自 /drone/local_position/velocity
    current_imu: ImuData | None           # 来自 /drone/imu/data
    last_data_ts: float                    # 最近一次 ROS 数据时间(判停产)
    # 低频(solver / monitor 写)
    current_trajectory: ContinuousTrajectory | None
    current_segment_index: int
    # 连接
    ipc_connected: bool
    # 锁
    pose_lock: threading.Lock
    # 求解器执行状态
    solver_status: str                    # idle/executing/hovering/aborted
    # 配置(启动加载, 只读)
    field: Field
    default_constraints: dict
```

- 高频位姿更新用 `pose_lock`,与低频队列/状态操作互锁分开。
- BState 不进数据库,不跨进程,只在本进程内。

## 8. 失败/断连处理(对齐总体架构 §3.6)

| 场景 | B 侧处理 |
|---|---|
| A↔B socket 断开 | `ipc_connected=False`;solver 切 `hover`(用 BState 当前位姿);记录上次位姿维持;等待重连(1s 恒定) |
| A 崩溃(收不到新意图) | 同上:solver 维持悬停,无新意图即悬停 |
| B 自己崩溃 | rospy 节点独立可停;假无人机/PX4 收不到 setpoint → 假无人机继续积分最后速度?需 sim-drone 侧"超时无 setpoint 自动悬停"(见 §9 sim-drone 契约) |
| 求解器 reject 某段 | 上行 `event:reject`(reason/segmentIndex);α 下轮重想 |
| 监控异常 alert | 上行 `event:alert`;A 转发前端 + 唤醒 β(不自动执行) |
| 监控程序本身故障 | 记日志,告警 `monitor_down`;退避 1~5s 重启;不影响飞行 |
| ROS 数据停产(无人机离线) | monitor 产 `drone_data_stale` alert;solver 维持最后 setpoint(不下发新位置) |
| msgpack 解帧失败 | 记日志,丢弃该帧,不断连(可能是版本不一致,见 §4.2 联调) |

**核心原则**:任何失败让 B 切到安全悬停,无人机总有合法 setpoint;异常只上行告警,不自动做飞行处置(人确认)。

### 8.1 solver 线程的安全保底逻辑

```
solver tick 结构图(伪代码):
  if not ipc_connected or no current_trajectory:
      publish_hover(current_pose)        # 安全悬停
      return
  if current_trajectory finished:
      publish_hover(current_pose)
      emit status: completed
      return
  sample = current_trajectory.next()
  if collision_check(sample) fails:        # 在线下发的二次校验(防场地动态变化/位姿漂移)
      publish_hover(current_pose)
      emit reject
      return
  adapter.publish_position(sample.pos, sample.yaw)
```

- 每次下发前**再做一次**当前 sample 的碰撞校验(场地若动态变化、或位姿漂出预期,在线校验兜底)。
- 任何异常分支都落到 `publish_hover`,保证不下发非安全 setpoint。

## 10. 生命周期与启动顺序(`lifecycle.py`)

**启动 N 步:**
1. 加载配置(`field.yaml` + `default_constraints.yaml`,Pydantic 或纯 yaml 校验,失败即退出)。
2. 初始化 BState(dataclass + Lock)。
3. 初始化 ROS 节点(`rospy.init_node("backend_b", anonymous=False)`),创建 publisher/subscriber(adapter 按 phase 选)。
4. 初始化 B 内总线(注册 solver/monitor 组件)。
5. 初始化 IPC client(连 A 的 Unix socket,恒定时间重连循环起)。
6. 启动 solver 线程(默认 tick 20Hz)。
7. 启动 monitor 线程(检测周期 10Hz)。
8. 启动 uplink 线程(10Hz)。
9. rospy.spin()(主线程,等关停信号)。

**关停**(信号处理 SIGINT/SIGTERM):
1. solver 切 `hover`(下发最后意图)。
2. 关 publisher(发末帧悬停)。
3. `rospy.signal_shutdown()`。
4. 关 IPC。
5. 清理。

> **关键**:B 启动时**先连 A 再启 solver 线程**,确保 A 断连感知已就绪后才接受 setpoint 流;若 A 未连上,solver 直接维持悬停(无轨迹可执行)。

### S1 — 假无人机单跑(无 B)
- `roscore` + `rosrun sim_drone fake_drone_node.py`。
- 手动 `rostopic pub /drone/setpoint_position/local` 发目标点,假无人机移动,`rostopic echo /drone/local_position/pose` 看位姿。
- 验收:假无人机收 setpoint 即动、发回位姿、超时悬停。

### S2 — B 单跑(无 A,内部 stub)
- B 启动,用一段 hardcoded TrajectorySpec(不走 IPC,直接调 `bus.call(to="solver", tool="trajectory", args=stub_spec)`)。
- 验收:求解器生成连续轨迹、碰撞校验通过、setpoint 话题有数据、假无人机按轨迹动、上行 `pose` 帧能在 B 内打印(msgpack 自解自验)。

### S3 — A↔B IPC 通
- A、B 互联(同机 Unix socket,路径配 `/tmp/flight_control_AB.sock` 或项目内)。
- A 发 `ping`,B 回 `pong`(心跳通)。
- A 发一段 `trajectory`(可先在 A 侧 hardcoded 不经 α),B 收到 solver 执行,假无人机动。
- 验收:A 收到 B 上行 `pose`(10Hz)、`status`、轨迹执行完。

### S4 — 碰撞校验 + reject 回路
- A 下发一个会撞墙的 TrajectorySpec,B solver 碰撞校验失败,上行 `reject`。
- 验收:A 收到 reject,WS 推前端 reject,α 上下文收到(下轮重想)。

### S5 — 经 α 翻译链路打通
- 前端 β Chat 发"飞到 (3,2,1)",β → α 翻译 TrajectorySpec → B solver → 假无人机。
- 验收:前 3D 看到无人机移动、轨迹线、α_output 系统消息。

### S6 — 监控 alert 回路
- 人为让假无人机超速(发超过 speed_max 的 setpoint,或直接改假无人机速度),monitor 产 `overspeed` alert 上行。
- 验收:前端 β 对话框出现 alert 系统消息,β 被唤醒给处置建议。

### S7 — 断连安全
- 杀 A 进程,B 应在 1s 内切 hover,假无人机悬停。
- 杀 B 进程,假无人机应超时悬停(依靠 §9.1 sim-drone 超时逻辑),A 收到 IPC 断连后前端 link_status=down。
- 验收:任何环节断,无人机不乱飞。

### S8(阶段2)— 切 PX4 SITL
- 装 PX4 SITL + mavros 1.20.1(阶段2),改 `topics.py` phase=2。
- PX4 SITL 在 WSL2 跑法已核实(见 §12):`make px4_sitl jmavsim` 启 SITL+iris(GUI,WSLg 可用);`HEADLESS=1 make px4_sitl gz_x500` 跑 headless Gazebo;`make px4_sitl none_iris` 接自定义仿真器。话题切 `/mavros/...`,setpoint 用 `mavros/setpoint_raw/local`(`mavros_msgs/PositionTarget` + `type_mask`,见 §5.2)。

## 14. 模块依赖关系(无环)

```
main → lifecycle → {config_loader, state, bus, ipc, rosbridge, solver, monitor}
bus → {ipc, registry}
solver → {constraints, collision, field, trajectory, translator, rosbridge(adapter)}
monitor → {detector, thresholds, trends, state}
rosbridge → {node, topics, publisher, subscriber, adapter}
ipc → {frames, dispatch, bus}
state, config_loader → (叶子)
```

`state`、`config_loader` 是叶子。无循环依赖。B 不依赖 A 的任何模块(只经 IPC 协议常量对齐 `SCHEMA_VERSION` 与 `use_bin_type`,这些是常量不是模块)。

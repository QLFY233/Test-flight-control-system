# 试飞控制系统 — 后端 B 设计(实时飞控桥)

> 子系统 spec,基于 `2026-07-01-总体架构-design.md`。后端 B 是实时飞控桥,跑在 **WSL2 Ubuntu 20.04 + ROS Noetic + Python 3.8** venv,经 **A↔B Unix domain socket(msgpack 长度前缀帧,统一总线消息)** 接收后端 A 下发的 **`ActionCommand`**(动作编码+数值,替代旧 TrajectorySpec),交 **端侧小模型**(B 内组件,先导 stub 占位;阶段4 迁机载)翻译成目标点;**阶段2/4** 接 ego-planner + 雷达在线避障生成连续轨迹;经 ROS 话题下发"假无人机"(阶段1)或 PX4 SITL + mavros offboard(阶段2);订阅无人机位姿/IMU,以 10Hz 上行 A;监控程序读 ROS 数据流做异常检测,异常经总线上行 alert。
>
> **2026-07-07 架构重构**:求解器(`solver/`)+场地碰撞检测整章**删除**;改为端侧小模型(`small_model/`,先导 stub)+ ego-planner 桥(`ego_planner_bridge/`,阶段2/4)+ 雷达感知(`perception/`,阶段2/4)。A↔B `call.tool` `trajectory`→`action`;`reject` 语义改为"小模型/ego-planner 无法生成可达目标"。`schema_version` 升 1→2。详见 [修改方案-2026-07-07-落地记录.md](../../过程文件/修改方案-2026-07-07-落地记录.md)。
>
> 与后端 A 的接口契约见总体架构 §3.2(L5 链路、A↔B 消息契约)、§3.7(统一消息总线格式)、§3.4(ROS 话题契约)、§3.6(失败/断连处理)。后端 B 是 B 侧组件(`small_model`/`monitor`/`ego_planner`/`lidar`)的宿主,经 B 内内存总线路由到实现,uplink 事件(`pose`/`telemetry`/`status`/`reject`/`pong`/`alert`)经 IPC 桥接上行 A。

## 1. 运行环境与版本锁定

### 1.1 运行宿主

- **OS**:Ubuntu 20.04 LTS,跑在 **WSL2**(Windows 11 22H2+,`%UserProfile%\.wslconfig` 含 `[wsl2] networkingMode=mirrored`)。
- **Python**:系统 Python 3.8(ROS Noetic 绑定,不允许升 3.10)。
- **ROS**:Noetic,安装在 `/opt/ros/noetic`(`source /opt/ros/noetic/setup.bash`)。
- **进程身份**:后端 B 是一个**独立的 OS 进**(与后端 A 同机,不同 venv,经 Unix socket 通信)。它本身既可作为一个 ROS 节点(内嵌 `rospy` 节点)跑,也可拆成"B 主进程 + 独立 ROS 节点";先导采用**单进程内嵌 rospy**(详见 §3.1)。

### 1.2 venv-B 依赖(对外接口契约与总体架构 §4.1/§4.2 一致)

```

  rospy / rospkg                   # 经 apt 装(ros-noetic-rospy 等), 不能 pip — 见下方 venv-B 创建方式
  mavros 1.20.1                    # apt (ros-noetic-mavros), 阶段2
  mavlink 2025.5.5                 # apt python3-msgpack 不提供, 阶段2 改 pip pymavlink — 见 §12
  msgpack 0.6.2  ⏳待装              # apt (python3-msgpack), 与 A 侧版本对齐 — 本机 Candidate 0.6.2-1, **未装**, 需 sudo apt install python3-msgpack (见 §12, A17)
  numpy 1.17.4 (apt 已装,见 §12 核实回填表)  # apt python3-numpy, 系统 site-packages

```

> **关键约束:msgpack 跨进程一致**。B 侧 msgpack 0.6.2(apt)必须能与 A 侧 msgpack 互通。msgpack 跨大版本(0.6 vs 1.x)在 bin 类型默认行为上有差异,**IPC 帧编解码双方必须 use_bin_type 一致**(见 §4.2 协议常量)。A 侧若用 pip 最新版,**A 侧建 Packer 时显式 `use_bin_type=True`** 与 B 侧对齐;否则退化到 B 侧 `use_bin_type=False`。这是 A↔B 互通的硬性契约,联调第一步验证项。

> **venv-B 创建方式(ROS+venv 经典坑,已本机实测)**:`rospy`/`rospkg`/`std_msgs`/`geometry_msgs`/`sensor_msgs` 等随 ROS apt 装在 **`/opt/ros/noetic/lib/python3/dist-packages`**(系统级,不进 venv site-packages),**不能 pip 装进 venv**;`numpy` 同理在系统 `/usr/lib/python3/dist-packages`。故 venv-B 必须以 `--system-site-packages` 创建,且**运行时先 source ROS 再 activate venv**,让 ROS 的 dist-packages 经 PYTHONPATH 可见。
> ```bash
> # 在 WSL2 Ubuntu 20.04
> # 1) 一次性补装系统 apt 依赖(若未装,见 §12 实测:rospy 随 ROS 已装,numpy 已装;msgpack/scipy 待装)
> sudo apt install python3-msgpack python3-scipy          # 进系统 site-packages, venv 可见
> # 2) 创建 venv-B(允许访问系统 site-packages)
> python3 -m venv --system-site-packages .venv-B
> # 3) 启动顺序(也是 run_backend_b.sh 的顺序,见 §1.5):先 source ROS 再 activate venv
> source /opt/ros/noetic/setup.bash
> source .venv-B/bin/activate
> pip install pyyaml       # 其余纯 Python 依赖按 venv-B-requirements.txt 装(venv 内, 不影响系统)
> ```
> **验证(本机实测 rospy/numpy 已通过;msgpack 待装后补验证)**:`source /opt/ros/noetic/setup.bash && source .venv-B/bin/activate && python -c "import rospy, numpy, msgpack; print('OK')"` 应全部成功(msgpack 需先 `sudo apt install python3-msgpack`,A17)——`rospy` 来自 `/opt/ros/noetic/lib/python3/dist-packages`(经 ROS PYTHONPATH),`numpy` 1.17.4 来自系统 site-packages,`msgpack` 0.6.2 来自系统 apt `python3-msgpack`(或 venv pip 装,二选一,版本须与 A 侧对齐 `use_bin_type=True`,见冻结 §1)。若 venv 不加 `--system-site-packages`,rospy/numpy 不可见,会全线失败。

### 1.3 与 venv-A 的隔离

- B 侧 venv 不共享 A 侧 site-packages,只经 Unix socket 传 msgpack 二进制帧。
- B 侧**禁止**(无法)使用 A 侧任何库:FastAPI / Pydantic AI / SQLAlchemy / aiosqlite 均不出现在 B 侧。
- B 侧**不**直接读写后端 A 的 SQLite 文件;历史落库由 A 侧负责(A 把 B 上行 `pose`/`telemetry` 缓冲后批量入库,总体架构 §5.2)。B 只产上行事件,不接触数据库。

### 1.5 启动脚本(`run_backend_b.sh`)

```bash
#!/usr/bin/env bash
set -euo pipefail

# 1) ROS 环境(Noetic)
source /opt/ros/noetic/setup.bash
# 2) 若有 catkin workspace(阶段2 或 sim-drone 已 catkin_make),source 它
if [ -f ros_ws/devel/setup.bash ]; then source ros_ws/devel/setup.bash; fi

source .venv-B/bin/activate
# 4) 环境变量(roscore URI、IPC socket 路径等)

export ROS_IP=127.0.0.1          # mirrored 模式下 localhost 直通;非 mirrored 需用 WSL2 IP

```

## 2. B 侧组件注册表与消息总线

### 2.1 B 内内存总线(`bus/`)

后端 B 复用总体架构 §3.7 的**统一消息总线协议**:`{schema_version, from, to, msg_type, call_id, tool, args, payload, ts}`。B 内组件经内存总线通信,B 主进程的 IPC 桥把"A 发来的 call/event"注入 B 总线,把"B 组件产出的 event"上行 A。

- **`bus/protocol.py`**:`SCHEMA_VERSION = 2`(与 A 一致,2026-07-07 升级),消息类型常量(`call`/`result`/`event`/`error`),`MSGPACK_USE_BIN_TYPE`(默认 True,与 A 对齐,见 §4.2 联调验证),`to` 名常量(`small_model`/`monitor`/`ego_planner`/`lidar`/`alpha`/`beta`/`broadcast`)。
- **`bus/registry.py`**:B 侧组件静态注册。启动时注册 `small_model` 和 `monitor`;`ego_planner`/`lidar` 阶段2/4 注册。各声明接受 `tool` 列表(见总体架构 §3.7 表):
  - `small_model` 接受 `generate_goal` / `abort` / `hover`。
  - `monitor` 不接受 call(它只持续产出 `event:alert`)。
- **`bus/router.py`**:`bus.call(to, tool, args)` → 查注册表 → 同步调用组件 → 返回 `result`(同 `call_id`)。B 内是同步函数语义(small_model 调用快,monitor 不被 call)。

### 2.2 B 侧组件接口契约(对齐总体架构 §3.7 "组件接口契约")

每个组件实现统一接口:接受总线消息、产出总线消息/事件。

**small_model 组件(`small_model/component.py`,先导 stub;阶段4 真模型迁机载):**

| 接受 `tool` | `args` | 产出 | 说明 |
|---|---|---|---|
| `generate_goal` | `{action:ActionCommand, pose, env}` | `result`(成功/不可达)+ 内部下发 setpoint | 小模型由动作编码+状态+环境算出目标点;先导 stub 用规则映射,夹紧 `safety_constraints.boundary`;阶段2/4 目标点交 ego-planner |
| `abort` | `{}` | `result` | 中止当前动作序列,切到悬停目标点(当前位姿) |
| `hover` | `{pose?}` | `result` | 安全悬停(用当前位姿或 home) |

small_model 还可在执行中产出上行事件:`status`(当前动作/总动作进度)、`reject`(无法生成可达目标:未知动作编码、夹紧后仍越界、阶段2/4 ego-planner 规划失败)。

> **先导 stub**(`small_model/stub.py`):规则占位实现——按动作编码表硬编码映射(`takeoff`→目标点 `[home.x,home.y,value]`、`goto`→`target`、`climb`→当前 xy + (z+value)、`hover`→当前位姿、`return_home`→`home` 等),产出目标点后夹紧到 `safety_constraints.boundary` 并应用 `speed_max`/`ceiling`/`floor`,直接作 setpoint 下发(先导无 ego-planner)。未知动作编码 → 上行 `reject`(回告 α 重想)。
> **阶段4 真模型**(`small_model/goal_gen.py`):ONNX runtime 推理(见 [2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md)),输入离散环境+状态+动作编码,输出目标点,迁机载电脑。

**ego_planner 组件(阶段2/4,`ego_planner_bridge/`,先导不存在):**
- 接受 `plan` tool(目标点+雷达点云 → 连续轨迹),先导不注册。
- 实现 `Planner` 接口(总体架构 §3.7):输入目标点+点云 → 输出避障连续轨迹。
- 在线避障失败 → 上行 `reject`。

**lidar 组件(阶段2/4,`perception/`,先导不存在):**
- 不接受 call;持续产 `event:pointcloud`(供 ego_planner)。先导不注册。

**monitor 组件(`monitor/component.py`):**

- 不接受 call;持续读 ROS 数据流(BState),每个检测周期产出 0~N 条 `event:alert`(经总线上行 A)。
- 实现 `Detector` 接口(总体架构 §3.7):输入数据流 → 输出 alert 事件。检测器注册表:阈值/趋势先导,后期可加 ML 检测器(开放式架构)。

### 2.3 A↔B 消息契约(对齐总体架构 §3.2,逐字对齐)

| 方向 | `msg_type` | `tool` | 含义 | B 侧处理 |
|---|---|---|---|---|
| A→B | `call` | `action` | **ActionCommand 下发**(替代旧 `trajectory`)| `bus.call(to="small_model", tool="generate_goal", args={action,pose,env})` |
| A→B | `call` | `abort` | 中止当前动作序列 | `bus.call(to="small_model", tool="abort")` |
| A→B | `call` | `hover` | 安全悬停 | `bus.call(to="small_model", tool="hover")` |
| A→B | `event` | `ping` | 心跳(2s) | 回 `pong` |
| B→A | `event` | `pose` | 当前位姿(10Hz) | `rosbridge/subscriber` 周期产出 |
| B→A | `event` | `telemetry` | 速度/加速度/IMU(10Hz) | **实现**(A8):10Hz 与 pose 分帧,富 IMU 入历史库,不推前端 |
| B→A | `event` | `status` | 小模型/无人机状态(低频) | small_model 产出(当前动作/总动作进度) |
| B→A | `event` | `reject` | 小模型/ego-planner 无法生成可达目标 | small_model 产出(先导);ego-planner 产出(阶段2/4)(回告 α 重想) |
| B→A | `event` | `pong` | 心跳回应 | IPC 桥回 |
| B→A | `event` | `alert` | 监控程序检测到异常 | monitor 产出 |

- **`schema_version` 升 1→2**(2026-07-07):`trajectory`+TrajectorySpec 路径废弃。
- **重连**:B 侧**恒定时间重连**(非指数退避,总体架构 §3.2),socket 断开每 1s 重连一次。
- **心跳**:A 每 2s 发 `ping`,B 回 `pong`;A 超 5s 无 pong 视为断连。B 侧不主动判定 A 断连(由 A 判定,A 断连后 B 收不到新意图 → B 自降级悬停,见 §8)。

### 2.4 上行事件分发(B→A → 总线 → A 侧处理,对齐后端 A spec §3.4)

B 上行的 event 在 A 侧被分发:`pose` → AppState + WS + 遥测缓冲;`alert` → monitor-trigger → WS alert + 唤醒 β;`reject` → α 上下文 + WS。B 侧只管"按频率产 event 并推 IPC",不感知 A 侧如何用。

## 3. 进程模型与线程架构

### 3.1 单进程多线程

后端 B 在**单进程**内,因 rospy 是基于线程的回调模型,而小模型/IPC 是同步阻塞,采用**混合线程模型**:

- **主线程**:rospy spin(订阅回调在此线程触发,更新 BState)。
- **目标点线程**:接收 A 下发 `action` call,调用 small_model 生成目标点,以定频(10~50Hz,先导直接目标点作 setpoint;阶段2/4 经 ego-planner 出连续轨迹)产出 setpoint 推到 ROS publisher 队列。
- **上行线程**:以 10Hz 从 BState 读位姿/遥测,封 msgpack 帧发 IPC。
- **监控线程**:以检测周期(默认 10Hz)读 BState,跑检测器,产 alert 推 IPC。
- **IPC 线程**:socket 收发循环 + 心跳响应。

> 用 `threading` + 线程安全队列(`queue.Queue`)做线程间消息传递,BState 用 `threading.Lock` 保护高频字段(位姿)。不引入 asyncio(避免在 ROS Noetic + Python 3.8 下与 rospy 事件循环冲突)。

### 3.2 线程职责表

| 线程 | 触发 | 职责 | 写 BState? |
|---|---|---|---|
| rospy 主线程 | rospy 回调 | 订阅位姿/IMU → 更新 BState | 是(加锁) |
| 目标点线程 | A 的 `action` call + 定频 tick | small_model 生成目标点 / ego-planner(阶段2/4) 连续轨迹 → 下发 setpoint | 读 BState(当前位姿) |
| uplink 线程 | 10Hz 定时 | BState → IPC 帧 | 读 |
| monitor 线程 | 检测周期定时 | 跑检测器 → alert | 读 |
| ipc 线程 | socket 可读 | 解帧 → B 总线;心跳 | 读/写 ipc_connected |

### 3.3 关停顺序

1. 停止接收新 A 消息(IPC 线程停)。
2. small_model 切 `hover`(下发最后意图为安全悬停目标点)。
3. 关闭 ROS publisher(让假无人机/PX4 收到末帧后维持)。
4. rospy signal_shutdown(等回调退出)。
5. 关 IPC socket。
6. 清理资源。

> 任何关停/异常路径,确保**最后下发的 setpoint 是悬停**(总体架构 §3.6:"任何环节失联/失败,无人机进入安全悬停")。

## 4. 端侧小模型与目标点生成(`small_model/`;ego-planner 阶段2/4)

> **2026-07-07 重构**:原"求解器"章节(含 TrajectorySpec 翻译、连续轨迹生成、场地碰撞校验)**整章删除**。新组件:端侧小模型,输入 `ActionCommand`+离散环境+无人机状态,输出**目标点**(ego-planner 所需输入子集)。先导用规则 stub 占位(打通链路),阶段4 真模型迁机载(见 [2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md))。场地碰撞检测**废弃**,交阶段2/4 ego-planner 在线雷达避障。

### 4.1 ActionCommand → 目标点(`goal_gen.py`)

输入(总体架构 §2.2 `ActionCommand` 格式,`schema_version=2`):
```json
{
  "task_id": "20260630143012",
  "schema_version": 2,
  "actions": [ {"code":"goto","target":[3.0,2.0,1.0]}, {"code":"hover","value":2.0}, ... ],
  "safety_constraints": {"speed_max":1.5, "ceiling":2.5, "floor":0.3, "boundary":[[0,0,0],[5,4,3]]}
}
```

`goal_gen` 把当前要执行的动作编码 + 当前位姿/速度 + 离散环境特征 → **目标点 `[x,y,z]`**(场地系)+ 速度上限(`speed_max` 夹紧)。
- 复合任务(`actions` 列表)逐条产出目标点,目标点到达后切下一条;先导每条动作产 1 个目标点,不插值(假无人机自己做运动学逼近)。
- `safety_constraints` 夹紧:目标点夹紧到 `boundary`,高度夹紧 `[floor, ceiling]`,速度夹紧 `speed_max`。
- 动作编码表见 `action_codes.py`(与总体架构 §2.2 一致):`takeoff`/`land`/`goto`/`move`/`climb`/`descend`/`yaw`/`hover`/`return_home`。

### 4.2 先导 stub 实现(`stub.py`,规则占位)

先导小模型**不训练**,用规则映射动作编码→目标点(打通链路优先):

| `code` | stub 目标点 |
|---|---|
| `takeoff` `(value)` | `[home.x, home.y, value]`(夹紧 `[floor,ceiling]`) |
| `goto` `(target)` | `target` 夹紧 boundary |
| `move` `(value)` | 当前位姿 + 当前朝向单位向量 × value,夹紧 boundary |
| `climb` `(value)` | 当前 xy + (当前 z + value),夹紧 `[floor,ceiling]` |
| `descend` `(value)` | 当前 xy + (当前 z − value),夹紧 `[floor,ceiling]` |
| `yaw` `(value)` | 目标点=当前位姿(只改航向,setpoint 带 yaw) |
| `hover` `(value)` | 当前位姿 |
| `return_home` | `home.position` |
| `land` | `[home.x, home.y, floor]` |
| 未知 `code` | 上行 `event:reject`(`reason:"unknown_action_code"`),回告 α 重想 |

- 无 ego-planner:stub 产出的目标点**直接作 setpoint**下发假无人机(经 `rosbridge/publisher`)。
- 物理安全:stub 夹紧 boundary + 假无人机边界自保(§9.1);**无碰撞检测**(先导不验避障,决策 4)。

### 4.3 small_model 组件执行流程(`component.py`)

```
收到 A 的 call(action, ActionCommand)
  ├─ 取首条/下一条未执行 action
  ├─ goal_gen: action + pose + env → 目标点(夹紧 safety_constraints)
  │   ├ 未知 action.code → 产 event:reject(reason, actionIndex) 上行 A,回 result:rejected
  │   └ 成功 → 目标点入"当前目标"缓存,目标点线程下发 setpoint(阶段2/4 经 ego-planner)
  └─ result:ok

目标点线程 tick(20Hz,先导直接 setpoint;阶段2/4 ego-planner 出连续轨迹):
  ├ 若有当前目标点:
  │   按 speed_max 限速,下发 setpoint(目标位置 + yaw)
  │   到达阈值内 → 切下一条 action(若有)→ status(currentAction/totalActions)
  │   全部完成 → status(completed)
  └ 若无目标 / abort / hover:
      下发"当前位姿悬停"setpoint
```

- **abort**:清当前动作序列缓存,切悬停(当前位姿目标点)。
- **hover**:若 args 给了 pose 用之,否则用 BState 当前位姿;无位姿时用场地 home。

### 4.4 ego-planner 桥(阶段2/4,先导空)

> 阶段2/4 接入。先导无 ego-planner、无雷达,本节为占位。

- `ego_planner_bridge/`:把小模型目标点 + 雷达点云(`lidar` 产 `pointcloud` event)喂 ego-planner ROS 节点,产出避障连续轨迹(B-spline/采样),代替 §4.3 的"直接 setpoint"。
- ego-planner 规划失败(目标被障碍包围不可达)→ 上行 `event:reject`(`reason:"ego_planner_unreachable"`)。
- 详见 [PX4 阶段2 spec](PX4-阶段2-design.md)(G4,远期)。
- 不再设"场地配置 collision.py / field.py 障碍物":场地由雷达在线感知,`field.yaml` 先导仅 `home`+`boundary` 软约束(见总体架构 §5.3)。

### 4.5 不在先导依赖第三方轨迹库

> 先导用规则 stub + 假无人机运动学逼近,**不引入** numpy/scipy 轨迹插值(无连续轨迹生成)。阶段2/4 ego-planner 自带轨迹优化,本 spec 不锁定其内部库。这一选择与"先导打通链路、不验避障"的目标一致。

## 5. ROS 桥(`rosbridge/`)

### 5.1 话题契约(对齐总体架构 §3.4)

**阶段1(纯 ROS 假无人机):**

| 话题 | 类型 | 方向 | B 侧 |
|---|---|---|---|
| `/drone/setpoint_position/local` | `geometry_msgs/PoseStamped` | B→无人机 | publisher |
| `/drone/setpoint_velocity/cmd_vel` | `geometry_msgs/Twist` | B→无人机 | publisher |
| `/drone/local_position/pose` | `geometry_msgs/PoseStamped` | 无人机→B | subscriber |
| `/drone/local_position/velocity` | `geometry_msgs/TwistStamped` | 无人机→B | subscriber |
| `/drone/imu/data` | `sensor_msgs/Imu` | 无人机→B | subscriber |

**阶段2(PX4 SITL + mavros):** 话题前缀换 `/mavros/...`,消息类型不变(总体架构 §3.4:"求解器对下游用统一话题接口(配置前缀),换实现只改配置")。

> **PX4 offboard 推荐话题(已核实,见 §12)**:PX4 + mavros 1.20.1 下,offboard 模式下发 setpoint 推荐 `mavros/setpoint_raw/local`(`mavros_msgs/PositionTarget`,带 `type_mask` 控制自由度:位置/速度/加速度/yaw/yaw-rate 位掩码)+ 持续 stream ≥2Hz(PX4 要求,否则 offboard 自动退出);亦支持 `~setpoint_position/local`(`geometry_msgs/PoseStamped`,仅 yaw)与 `~setpoint_velocity/cmd_vel_unstamped`(`geometry_msgs/Twist`)。本 spec 以**话题配置前缀 + 类型可配**实现,阶段2 落地按 §12 核实结果在 `rosbridge/topics.py` 与 `adapter.py` 定具体话题与 `type_mask`,不动 small_model 上层。先导阶段1 用 `/drone/setpoint_position/local` 即可。

# 伪代码:阶段切换只改配置,不动 small_model / ego_planner

class Phase2Adapter(SetpointAdapter):       # PX4 + mavros(阶段2)

```

`topics.py` 读配置(`backend-B/config/rostree.yaml` 或环境变量)选 `phase: 1|2`,实例化对应 adapter。small_model(先导)或 ego-planner(阶段2/4)产出目标点/连续轨迹,经 adapter 下发,不感知阶段。

### 5.3 publisher(`publisher.py`)

- 接收目标点线程(先导)/ego-planner 线程(阶段2/4)的 setpoint(20Hz),转 ROS 消息下发。
- **首帧前填一个当前位置 setpoint**,避免无人机跳变(从 0/0 起跳)。
- 阶段2(PX4 offboard)需先发若干秒 setpoint 流再切 offboard 模式(标准做法),此逻辑放 adapter 内,small_model 不感知。

### 5.4 subscriber(`subscriber.py`)

- 订阅位姿/速度/IMU。回调加锁写 BState。
- 同时维护"最近一次更新时间",监控线程据此判"无人机数据是否超时停产"(产 alert:drone_data_stale)。

### 5.5 10Hz 上行节流

- 上行线程以 10Hz 读 BState,产 `event:pose` 和 `event:telemetry`,封 msgpack 发 IPC。
- 节流实现:上行线程 `while True: read; pack; send; sleep(0.1)`。不依赖 ROS 的话频率,解耦 ROS 抖动与上行频率。

## 6. 监控程序(`monitor/`)

> 对齐总体架构 §1 末:"异常监控程序(monitor/):读 ROS 数据流,阈值+趋势检测,异常经总线上行 alert(B 是飞行数据源头)";研究内容 3:"开发数据监控程序,接入飞行数据流,依据预设阈值与趋势分析逻辑检测异常,并触发大模型调度,生成告警信息与处置建议。"

### 6.1 Detector 接口与注册表(`detector.py`)

```python
class Detector(Protocol):
    name: str
    def update(self, sample: TelemetrySample) -> list[Alert]: ...

DETECTORS: list[Detector] = [ThresholdDetector(), TrendDetector()]
```

- 检测器注册表:阈值/趋势先导,后期可加 ML 检测器(开放式)。
- monitor 组件每个周期跑所有检测器,聚合 alert 上行。

### 6.2 阈值检测(`thresholds.py`)

预设阈值(读 `default_constraints.yaml` 或单独 `monitor_thresholds.yaml`,先放全局默认):

| 检测项 | 阈值来源 | alert code |
|---|---|---|
| 位置超出 boundary(软告警,先导无预编障碍) | 场地配置 `boundary`(先导仅 home/boundary) | `out_of_boundary` |
| 高度超 ceiling / 低于 floor | 约束 | `ceiling_breach`/`floor_breach` |
| 速度超 speed_max | 约束 | `overspeed` |
| 加速度超 accel_max | 约束 | `overaccel` |
| 角速度超 angular_velocity_max | 约束 | `over_angular` |
| 无人机数据停产(>500ms 无更新) | 固定 | `drone_data_stale` |
| small_model 下发的 setpoint 与实际位姿持续偏差大 | 趋势 | `tracking_error` |

> **2026-07-07 重构**:原"求解器超 boundary 直接 reject/abort"硬终止废弃(无求解器)。位置超 boundary 改为**软告警** `out_of_boundary`(monitor 上行 alert,不自动终止飞行;物理夹紧由 small_model stub + 假无人机边界自保兜底)。`keep_clear_distance` 相关碰撞检测废弃。

### 6.3 趋势检测(`trends.py`)

- **突变**:相邻样本二阶差分超阈值(抖动)。
- **持续偏离**:滑窗内 setpoint vs actual 偏差均值超阈值(轨迹跟不住)。
- 先导用 numpy 滑窗统计;后期可换频域(与 A 侧 analytics/FFT 联动,但 B 侧只产原始 alert,FFT 分析在 A 侧)。

### 6.4 alert 事件格式(对齐前端 AlertMessage,见前端详细设计 §3.3)

```python
# B 上行 event:alert 的 payload
{
  "level": "warn",         # info/warn/critical
  "code": "overspeed",
  "detail": "速度 2.1 m/s 超过 1.5 m/s",
  "suggestion": None,        # B 不给建议,β 给(被唤醒后)
  "ts": 1234567890,
  "action_index": 2          # 可选, 若执行中(对齐 ActionCommand 序列)
}
```

- **B 只告警不建议直接执行**(总体架构 §1.3 安全边界):alert 上行 A → A 转发前端(β 对话框系统消息) + 唤醒 β 给处置建议 → 处置需人确认才执行。
- monitor 产 alert 的节流:同一 `code` 在 N 秒内只上告一次(防刷屏,N 可配,默认 2s),critical 级别不节流。

### 6.5 监控程序本身故障(对齐总体架构 §3.6)

- monitor 线程异常 → catch + 记日志 + 告警前端(上行一条 `code=monitor_down` alert)+ 退避重启(1~5s)。
- 不影响飞行(降级:无监控期间靠 small_model stub 的 boundary 夹紧 + 假无人机边界自保兜底,见 §4.2/§9.1)。

## 7. 共享状态 BState(`state.py`)

```python
@dataclass
class BState:
    # 高频(ROS 回调写, 加锁)
    current_pose: PoseData | None         # 来自 /drone/local_position/pose
    current_velocity: VelocityData | None # 来自 /drone/local_position/velocity
    current_imu: ImuData | None           # 来自 /drone/imu/data
    last_data_ts: float                    # 最近一次 ROS 数据时间(判停产)
    # 低频(small_model / monitor 写)
    current_goal: tuple | None             # 当前目标点 [x,y,z](先导;阶段2/4 由 ego-planner 出连续轨迹)
    current_action_index: int              # 执行到 ActionCommand.actions 第几条
    current_action_plan: dict | None       # ActionCommand(动作序列缓存)
    # 连接
    ipc_connected: bool
    # 锁
    pose_lock: threading.Lock
    # small_model 执行状态
    small_model_status: str                # idle/executing/hovering/aborted
    # 配置(启动加载, 只读)
    field: Field                           # 先导仅 boundary + home;obstacles 废弃
    default_constraints: dict
```

- 高频位姿更新用 `pose_lock`,与低频队列/状态操作互锁分开。
- BState 不进数据库,不跨进程,只在本进程内。

## 8. 失败/断连处理(对齐总体架构 §3.6)

| 场景 | B 侧处理 |
|---|---|
| A↔B socket 断开 | `ipc_connected=False`;solver 切 `hover`(用 BState 当前位姿);记录上次位姿维持;等待重连(1s 恒定) |
| A 崩溃(收不到新意图) | 同上:solver 维持悬停,无新意图即悬停 |
| B 自己崩溃 | rospy 节点独立可停;假无人机/PX4 收不到 setpoint → 假无人机继续积分最后速度?需 sim-drone 侧"超时无 setpoint 自动悬停"(见 §9 sim-drone 契约) |
| small_model reject(未知动作/夹紧后越界/阶段2/4 ego-planner 不可达) | 上行 `event:reject`(reason/actionIndex);α 下轮重想或兜底 hover |
| 监控异常 alert | 上行 `event:alert`;A 转发前端 + 唤醒 β(不自动执行) |
| 监控程序本身故障 | 记日志,告警 `monitor_down`;退避 1~5s 重启;不影响飞行 |
| ROS 数据停产(无人机离线) | monitor 产 `drone_data_stale` alert;small_model 维持最后目标点(不下发新位置) |
| msgpack 解帧失败 | 记日志,丢弃该帧,不断连(可能是版本不一致,见 §4.2 联调) |

**核心原则**:任何失败让 B 切到安全悬停,无人机总有合法 setpoint;异常只上行告警,不自动做飞行处置(人确认)。

### 8.1 目标点线程的安全保底逻辑

```
目标点线程 tick 结构图(伪代码):
  if not ipc_connected or no current_goal:
      publish_hover(current_pose)        # 安全悬停
      return
  if current_action_plan finished:
      publish_hover(current_pose)
      emit status: completed
      return
  goal = small_model.next_goal(current_action_plan, pose, env)  #夹紧 boundary/speed
  if goal is None:        # 未知动作编码或夹紧后仍越界
      publish_hover(current_pose)
      emit reject
      return
  # 先导无在线碰撞校验(无雷达/ego-planner);阶段2/4 由 ego-planner 在线避障兜底
  adapter.publish_position(goal.pos, goal.yaw)
```

- 先导**无在线碰撞校验**(无雷达、无预编障碍,决策 4);物理安全靠 boundary 夹紧 + 假无人机边界自保(§9.1)。
- 任何异常分支都落到 `publish_hover`,保证不下发非安全 setpoint。
- 阶段2/4:ego-planner tick 内对当前目标点做在线避障,规划失败 → 上行 reject → 此分支兜底 hover。

## 9. sim-drone(阶段1 假无人机)契约

> 假无人机是独立 catkin 包(`sim-drone/`),不属于 backend-B,但两者契约必须明确。

### 9.1 假无人机节点行为(`sim-drone/scripts/fake_drone_node.py`)

- 订阅 `/drone/setpoint_position/local`(PoseStamped)作为目标位置。
- **运动学积分**(非动力学):当前位姿向目标位置以受限速度逼近(线性插值,速度上限可配,默认 2 m/s)。
- 50Hz 发布 `/drone/local_position/pose`、`/drone/local_position/velocity`、`/drone/imu/data`(IMU 由位置二阶差分合成)。
- **超时悬停**:若 >500ms 未收到 setpoint,自动悬停(停止移动),并维持发布位姿(让 B 侧 monitor 不误报 stale)。
- **边界自保**:若目标位置超 boundary,假无人机停在边界内最近点(不越界),作为最后一道物理安全网。

### 9.2 catkin 包结构(Noetic `catkin_make` 已核实可用,见 §12)

```
sim-drone/
├── CMakeLists.txt      # catkin_package() + catkin_install_python(PROGRAMS scripts/fake_drone_node.py)
├── package.xml         # <depend>rospy</depend> <depend>geometry_msgs</depend> <depend>sensor_msgs</depend>
├── scripts/
│   └── fake_drone_node.py   # 加可执行位 (+x)
└── launch/
    └── fake_drone.launch    # <node name="fake_drone" pkg="sim_drone" type="fake_drone_node.py" output="screen"/>
```

- 在 `ros_ws/src` 软链 `sim-drone`,`cd ros_ws && catkin_make` 后 `source ros_ws/devel/setup.bash`。
- 也可不进 catkin 直接 `rosrun --prefix ./sim-drone/scripts python fake_drone_node.py`,但标准做法是 catkin 包(便于 roslaunch)。
- **Noetic 下 catkin_make 与 catkin_tools 均可用**,先导用 `catkin_make`(更通用)。

### 9.3 阶段2 切 PX4 SITL(对齐总体架构 §3.4 末)

- 换 `rosbridge/topics.py` 配置 `phase=2`,话题前缀 `/mavros/...`。
- `sim-drone` 在阶段2 不启动(被 PX4 SITL + mavros 节点替代)。
- PX4 SITL 在 WSL2 跑法(headless)已核实(见 §12):`make px4_sitl jmavsim` / `HEADLESS=1 make px4_sitl gz_x500` / `make px4_sitl none_iris`。

## 10. 生命周期与启动顺序(`lifecycle.py`)

**启动 N 步:**
1. 加载配置(`field.yaml` + `default_constraints.yaml`,Pydantic 或纯 yaml 校验,失败即退出)。
2. 初始化 BState(dataclass + Lock)。
3. 初始化 ROS 节点(`rospy.init_node("backend_b", anonymous=False)`),创建 publisher/subscriber(adapter 按 phase 选)。
4. 初始化 B 内总线(注册 small_model/monitor 组件;阶段2/4 加 ego_planner/lidar)。
5. 初始化 IPC client(连 A 的 Unix socket,恒定时间重连循环起)。
6. 启动目标点线程(先导 small_model 直发 setpoint,默认 tick 20Hz)。
7. 启动 monitor 线程(检测周期 10Hz)。
8. 启动 uplink 线程(10Hz)。
9. rospy.spin()(主线程,等关停信号)。

**关停**(信号处理 SIGINT/SIGTERM):
1. small_model 切 `hover`(下发最后意图—当前位姿目标点)。
2. 关 publisher(发末帧悬停)。
3. `rospy.signal_shutdown()`。
4. 关 IPC。
5. 清理。

> **关键**:B 启动时**先连 A 再启目标点线程**,确保 A 断连感知已就绪后才接受 setpoint 流;若 A 未连上,small_model 直接维持悬停(无动作可执行)。

## 11. 联调步骤(分阶段打通链路)

> 与架构概要"先做一个最简单的假无人机,跑通了再接 PX4"对齐。每一步**可独立验证**才进下一步。

### S0 — 环境(前置)
- WSL2 Ubuntu 20.04,`roscore` 能起(本机实测:ROS Noetic 已装 `/opt/ros/noetic`,见 §12 核实回填表;WSL2 跑 roscore 无需 X server)。
- venv-B 装好(apt 装 ROS + python3-numpy/scipy/msgpack,venv 装 pyyaml 等)。
- **联调验证 #1**:在 A/B 两端各跑一段 msgpack pack/unpack,确认 `use_bin_type` 一致(帧能互解,见 §4.2)。

### S1 — 假无人机单跑(无 B)
- `roscore` + `rosrun sim_drone fake_drone_node.py`。
- 手动 `rostopic pub /drone/setpoint_position/local` 发目标点,假无人机移动,`rostopic echo /drone/local_position/pose` 看位姿。
- 验收:假无人机收 setpoint 即动、发回位姿、超时悬停。

### S2 — B 单跑(无 A,内部 stub)
- B 启动,用一段 hardcoded ActionCommand(不走 IPC,直接调 `bus.call(to="small_model", tool="generate_goal", args={action:stub_action, pose, env})`)。
- 验收:小模型 stub 产出目标点、夹紧 boundary、setpoint 话题有数据、假无人机按目标点动、上行 `pose` 帧能在 B 内打印(msgpack 自解自验)。

### S3 — A↔B IPC 通
- A、B 互联(同机 Unix socket,路径配 `/tmp/flight_control_AB.sock` 或项目内)。
- A 发 `ping`,B 回 `pong`(心跳通)。
- A 发一段 `action`(可先在 A 侧 hardcoded 不经 α),B 收到 small_model 执行,假无人机动。
- 验收:A 收到 B 上行 `pose`(10Hz)、`status`、动作执行完。

### S4 — 小模型 reject 回路(先导不验避障)
- A 下发一个含**未知动作编码**的 ActionCommand,B small_model stub 无法识别,上行 `reject`(`reason="unknown_action_code"`)。
- 验收:A 收到 reject,WS 推前端 reject,α 上下文收到(下轮重想或兜底 hover)。
- **碰撞校验回路延后到阶段2/4**(先导无雷达、无 ego-planner,决策 4 不验避障)。

### S5 — 经 α 翻译链路打通
- 前端 β Chat 发"飞到 (3,2,1)",β → α 翻译 **ActionCommand**(`{code:"goto",target:[3,2,1]}`)→ B small_model → 目标点 → 假无人机。
- 验收:前 3D 看到无人机移动、轨迹线、α_output 系统消息(动作编码+目标点)。

### S6 — 监控 alert 回路
- 人为让假无人机超速(发超过 speed_max 的 setpoint,或直接改假无人机速度),monitor 产 `overspeed` alert 上行。
- 验收:前端 β 对话框出现 alert 系统消息,β 被唤醒给处置建议。

### S7 — 断连安全
- 杀 A 进程,B 应在 1s 内切 hover,假无人机悬停。
- 杀 B 进程,假无人机应超时悬停(依靠 §9.1 sim-drone 超时逻辑),A 收到 IPC 断连后前端 link_status=down。
- 验收:任何环节断,无人机不乱飞。

### S8(阶段2)— 切 PX4 SITL
- 装 PX4 SITL + mavros 1.20.1(阶段2),改 `topics.py` phase=2。
- PX4 SITL 在 WSL2 跑法已核实(见 §12):`make px4_sitl jmavsim` 启 SITL+iris(GUI,WSLg 可用);`HEADLESS=1 make px4_sitl gz_x500` 跑 headless Gazebo;`make px4_sitl none_iris` 接自定义仿真器。话题切 `/mavros/...`,setpoint 用 `mavros/setpoint_raw/local`(`mavros_msgs/PositionTarget` + `type_mask`,见 §5.2)。

# 试飞控制系统 — 总体架构设计

> 先导项目:打通"大模型理解试飞意图 → α 翻译为动作编码 → 端侧小模型产出目标点 → (阶段2/4 ego-planner 雷达在线避障) → 模拟无人机执行"完整链路。
> 本阶段无真实无人机,用 ROS1 模拟无人机代替(无雷达,先导不验避障)。
>
> **2026-07-07 架构重构**:按甲方要求引入"端侧小模型",求解器结构废弃。
> - α(LLM,先导 DeepSeek-V4-Flash)**不再产 TrajectorySpec**,改产 **`ActionCommand`**(动作编码+数值的 JSON)。
> - **端侧小模型**(数 MB 神经网络)在机载电脑(阶段4)/先导 B 侧 stub 上运行,输入离散环境数据+无人机状态+动作编码,输出 **目标点**(ego-planner 所需输入的子集)。
> - **ego-planner**(阶段2/4)接入,根据机载雷达自动避障;先导无雷达,不验避障。
> - 场地不再预先编码,碰撞检测交由阶段2/4 ego-planner 在线完成;先导仅留 `home`+`boundary` 软约束兜底。
> - 详见 [2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md)、[开放式接口规范.md](开放式接口规范.md)。
>
> 原 v1 `TrajectorySpec`/求解器/航点段约束体系已废弃(`schema_version` 升 1→2);旧文本如残留引用,以本 spec 为准。

### 1.3 安全边界

- 大模型绝不直接碰飞控:α 只产出 `ActionCommand`(动作编码+数值),由**端侧小模型**翻译成目标点,再经 **ego-planner(阶段2/4)/setpoint 直发(先导)** 落实物理;软安全约束(`speed_max`/`ceiling`/`floor`/`boundary`)在 `ActionCommand` 内随附,B 侧 stub 小模型与假无人机夹紧兜底。
- β→α 的两种路径:
  - `propose_to_alpha(intent)`(β 自主想法):**必须人审核**后才注入 α,β 不直连飞行路径。
  - `forward_last_human_message()`(转发人刚对 β 说的指令):直接进 α 队列,无需再审核(人已发话)。
- 异常监控程序**只告警不建议直接执行**:异常经总线 → 前端(β 对话框系统消息)+ 唤醒 β 给处置建议,处置需人确认才执行。
- A↔B 消息总线桥接点是天然安全检查点。
- **ego-planner(阶段2/4)是物理安全的最终执行者**:在线雷达避障;先导无 ego-planner,物理安全靠软约束兜底(假无人机边界自保)。

## 2. 数据流与中枢调度

### 2.1 翻译链与中枢调度

> 龙虾模式:中枢大模型 β 调度多组件(α 动作翻译器、端侧小模型、监控程序、数据分析工具、看板驱动器)。人类只与 β 对话,不与 α 对话。

```
人类自然语言(语音/文字) → β(中枢) ──┬─→ 产出飞行计划(动作意图) → 人审核 → 注入 α
                                        ├─→ forward_last_human_message → α
                                        ├─→ 调数据分析工具 / 历史查询
                                        └─→ 调看板驱动器(工具替换展示哪些数据 + 筛选)
α(动作翻译器,不对话) → ActionCommand(动作编码+数值)
                                        → 端侧小模型(离散 环境/状态/动作) → 目标点
                                        → [阶段2/4 ego-planner 雷达避障 → 连续轨迹]
                                        → setpoint → 假无人机/PX4(先导直连,无 ego-planner)
异常监控程序(B) → alert(总线) → 前端(β 对话框系统消息)+ 唤醒 β 给处置建议
```

| 层 | 执行者 | 输入 | 输出 | 工具/依赖 | 频率 |
|---|---|---|---|---|---|
| L1 调度 | Agent β(中枢) | 人类自然语言(语音/文字)+ 异常唤醒 + 飞行数据 | 飞行计划 / 数据分析结果 / 处置建议 / 看板配置 / 转发指令给 α | `propose_to_alpha`/`forward_last_human_message`/`analytics`/历史查询/`dashboard_configure` 工具 | 按需 + 异常唤醒 |
| L1.5 审核 | 人 | β 的 `propose_to_alpha` 提议 | 确认/修改/驳回 | β 对话框 | 按需 |
| L2 动作翻译 | Agent α(**非对话**动作翻译器) | 人确认的 β 提议 / β 转发的人话指令 | **`ActionCommand`**(动作编码+数值 JSON + 软安全约束) | 读当前位姿/环境(只读) | loop,1~5 秒 |
| L3 目标点生成 | **端侧小模型**(B 内组件,先导 stub;阶段4 机载) | `ActionCommand` + 离散环境 + 无人机状态 | **目标点**(ego-planner 所需输入子集)+ 软约束夹紧 | ONNX 推理 / 先导规则占位 | 与 α loop 同频 |
| L3.5 在线规划 | **ego-planner**(B 内组件,**阶段2/4**) | 目标点 + 雷达点云 | 连续轨迹(避障) | ego-planner ROS 节点 | 10~50Hz |
| L4 执行 | 假无人机/PX4 | setpoint(目标点/连续轨迹) | 实际飞行 | ROS 话题 | 毫秒级 |
| L 异常 | 监控程序(B 内组件) | B 的 ROS 数据流 | alert(经总线上行) | 阈值+趋势检测器 | 持续 |

> **先导(阶段1)**:L3 端侧小模型用 stub 占位(规则:动作编码→目标点映射,夹紧到 boundary),L3.5 ego-planner 不存在,L4 直接由小模型产出的目标点作 setpoint。**阶段2/4** 接入真实 radar + ego-planner,小模型迁机载。

### 2.2 ActionCommand schema(α 输出,替代旧 TrajectorySpec)

> **2026-07-07 重构**:原 `TrajectorySpec`(航点序列 + 逐段约束 + 求解器)已废弃。α 现输出 `ActionCommand`——动作编码+数值的 JSON,供端侧小模型离散化输入。`schema_version` 升 1→2。

α 每个 loop 周期产出一个或一串 `ActionCommand`(简单任务单条,复合任务离散序列):

```json
{
  "task_id": "20260630143012",
  "schema_version": 2,
  "actions": [
    {
      "code": "takeoff",
      "value": 1.0,
      "units": "m",
      "comment": "起飞到 1.0m"
    },
    {
      "code": "goto",
      "target": [3.0, 2.0, 1.0],
      "units": "m",
      "comment": "飞往 (3,2,1)"
    },
    {
      "code": "hover",
      "value": 2.0,
      "units": "s",
      "comment": "悬停 2 秒"
    },
    {
      "code": "return_home",
      "comment": "返回起点"
    },
    {
      "code": "land",
      "comment": "降落"
    }
  ],
  "safety_constraints": {
    "speed_max": 1.5,
    "ceiling": 2.5,
    "floor": 0.3,
    "boundary": [[0.0, 0.0, 0.0], [5.0, 4.0, 3.0]]
  }
}
```

**动作编码表**(可扩展,先导子集;编码由 α prompt 约束,新增编码经组件注册表登记):

| `code` | `value` 语义 | `target` | 说明 |
|---|---|---|---|
| `takeoff` | 目标高度(m) | — | 起飞到指定高度 |
| `land` | — | — | 降落至地面 |
| `goto` | — | `[x,y,z]` m | 飞往目标点(绝对场地系) |
| `move` | 距离(m) | — 或方向单位向量 | 沿当前朝向/给定方向移动 |
| `climb` | 高度增量(m) | — | 相对爬升 |
| `descend` | 高度增量(m) | — | 相对下降 |
| `yaw` | 角度(deg) | — | 偏航到/偏航 |
| `hover` | 时长(s) | — | 原地悬停 N 秒 |
| `return_home` | — | — | 返回 `home.position` |

可扩展:加新 `code` = 动作编码表登记 + α prompt 更新 + 小模型输入维度扩展(`schema_version` 不动,新增编码向后兼容;旧小模型遇未知编码 → 兜底 `hover`)。

**`safety_constraints`**(软安全约束,随附每条 ActionCommand 下发):
- `speed_max`/`ceiling`/`floor`:B 侧 stub 小模型与假无人机夹紧兜底(先导物理安全网)。
- `boundary`:场地软边界(先导来自 `field.yaml`;阶段2/4 由雷达/ego-planner 替代)。
- `keep_clear_distance` 等**碰撞相关约束废弃**(交 ego-planner 阶段2/4 在线避障)。

**`mode` 字段取值**(`StatusMessage.mode`,映射当前执行段的操控来源):

| `mode` | 触发条件 |
|---|---|
| `manual` | 当前执行段由人通过 β Chat(`forward_last_human_message` 免审路径)直接驱动的 α 翻译产出 |
| `auto` | 当前执行段来自 β `propose_to_alpha` 提议经人审核批准后的 α 翻译产出(预设动作执行) |

人确认前的 β 提议、α 悬停兜底阶段沿用上一次合法 `mode`;非 LLM 小模型 α 后端切换不改 `mode` 语义。

### 2.3 α 动作翻译器职责与 loop 仲裁

> **α 不与人对话,不输出文字给人。** α 是纯动作翻译器:接收 β 转发/人确认的指令,翻译成 `ActionCommand`(动作编码+数值),经总线发端侧小模型生成目标点。先导用 LLM(DeepSeek-V4-Flash)实现 `ActionTranslator` 接口;远期可换蒸馏小模型 α(见 [2026-07-05-小模型α-design.md](2026-07-05-小模型α-design.md)),接口留好。

α 在 loop 内的输入来源:

- 来源 i:β 的 `forward_last_human_message` 转发的人话指令(直接进 α 队列,无需审核,人已发话)。
- 来源 ii:人审核通过的 β 提议(`propose_to_alpha`,经人确认后进 α 队列)。
- 来源 iii:无人机数据(B 上行 pose/telemetry,loop 每周期读,作为翻译上下文);环境数据(`get_current_environment`)。

**α 的输入不再含"人直接打字给 α"——前端已取消 α 对话框,人只与 β 对话。**

**loop 仲裁逻辑:**

```
α loop 周期开始
  ├ 读取:当前位姿(从 B 上行)、当前执行中的预设动作序列、α 输入队列(β 转发 + 人确认的提议)
  ├ 仲裁规则:
  │   if α 输入队列非空:
  │       本周期处理输入(翻译成 ActionCommand,打断预设)
  │   elif 执行中预设还有未完成动作:
  │       本周期下发下一个动作的目标点(由小模型算出)
  │   else:
  │       本周期下发"保持悬停"(安全默认,code=hover)
  ├ 翻译:指令 → ActionCommand(调用 LLM,1~5 秒,非阻塞;远期换蒸馏小模型)
  └ 输出:ActionCommand(经消息总线发 small_model 组件 → 产目标点 → setpoint)
周期结束,新输入累入下个周期
```

**打断规则:**
- loop 单步内:新输入排队到下一周期(不抢占当前周期)。
- 预设动作序列整体:可被新输入打断(β 转发新人话 / 人确认新提议优先)。被打断的预设剩余部分丢弃或暂停。
- 安全默认:无任务、无新输入时 α 下发 `hover`(`code=hover`),飞控总有合法期望值。

**α 实现统一 `ActionTranslator` 接口**(组件注册表的一项),先导 LLM 实现,后期换蒸馏小模型实现此接口,总线调度不变。`TrajectoryTranslator`(旧)已废弃。

### 2.4 β 工具集(中枢调度的组件)

β 是中枢大模型,通过工具调度各组件。工具分三类:

**历史/状态查询:**

| 工具 | 输入 | 输出 | 数据源 |
|---|---|---|---|
| `get_field_map()` | 无 | 场地边界+障碍列表 | 场地配置文件 |
| `get_current_pose()` | 无 | 当前位姿/速度 | B 上行状态(A 缓存) |

### 2.5 中枢调度协作(龙虾模式)

```
┌────────────────┐         人类(语音/文字)
│   前端          │ ◄──────────────────────────┐
│ (仅 β 对话框    │                            │

     ▼                                        │
┌────────────────────────────────────────────┴───┐
│  Agent β(中枢大模型)                          │
│  工具:历史查询 / analytics /                  │
│       propose_to_alpha / forward_last_human / │
│       dashboard_configure / dashboard_filter  │
└──┬───────────────┬────────────────┬───────────┘
   │ propose        │ forward        │ analytics / dashboard call
   │ (人审核)       │ (直接)         │ (总线)
   ▼                ▼                ▼
┌──────┐       ┌────────────┐   ┌────────────┐
│ 人   │──确认──►│ Agent α     │   │ analytics/ │
│ 审核 │       │ (动作翻译器,│   │ dashboard/ │
└──────┘       │  不对话)    │   └─────┬──────┘
               └──────┬─────┘         │
                      │ ActionCommand │ 结果回 β
                      ▼                │
               ┌────────────┐          │
               │ 端侧小模型  │          │
               │ (B, 产目标点)│          │
               └──────┬─────┘          │
                      │ 目标点          │
                      ▼                │
               ┌────────────┐          │
               │ ego-planner │◄─────────┘  ← 先导无,直接 setpoint
               │ (阶段2/4)   │
               └──────┬─────┘
                      │ 连续轨迹(阶段2/4)
                      ▼
               ┌────────────┐
               │ 假无人机/   │
               │ PX4         │
               └──────┬─────┘
                      │ ROS 数据流
                      ▼
               ┌────────────┐
               │ 监控程序(B) │──alert(总线)──► 前端(β 系统消息)+ 唤醒 β
               └────────────┘
```

- 人类只与 β 对话(语音/文字),不与 α 对话。
- α 是动作翻译器,不与人对话,接收 β 转发/人确认的指令,翻译成 `ActionCommand`。
- 端侧小模型把 `ActionCommand`(+离散环境/状态)翻译成目标点;阶段2/4 接 ego-planner 雷达避障。
- β 经消息总线调度 α、analytics、历史查询、看板驱动等组件(龙虾模式)。
- 监控程序(B 侧组件)持续看飞行数据,异常经总线上行 alert,前端作为**系统消息插入 β 对话流**,同时唤醒 β 给处置建议。
- 各组件经统一消息总线通信(§3.7),α 后期可换蒸馏小模型(接口不变);小模型先导 stub→阶段4 真模型;ego-planner 阶段2/4 接入。
- 先导模型:α 用 DeepSeek-V4-Flash(OpenAI 格式,经 Pydantic AI openai provider)。

## 3. 通信 fabric 与协议契约

> 龙虾模式开放式架构:组件间经**统一消息总线**通信(§3.7)。前端↔A 仍用 Web 协议(SSE/WS/REST);A 内组件经内存总线;A↔B 经 Unix socket 桥接总线;B↔无人机经 ROS 话题。所有组件消息格式统一。

### 3.1 链路总表

| # | 链路 | 协议 | 方向 | 频率 | 消息格式 |
|---|---|---|---|---|---|
| L1 | 前端 ↔ A | WebSocket | 双向 | 10Hz~视频 | JSON(含 α_output 系统/alert/看板配置 等,详见 §3.3) |
| L2 | 前端 ↔ A | SSE | A→前端 | 秒级(β 流式) | Pydantic AI UIAdapter 事件 |
| L3 | 前端 ↔ A | REST | 双向 | 按需 | JSON |
| L4 | 前端 ↔ A | StaticFiles | A→前端 | 一次 | HTML/JS/CSS |
| L5 | A ↔ B | Unix domain socket(总线桥接) | 双向 | 上行10Hz/下行秒级 | 统一总线消息,msgpack |
| L6 | B ↔ 假无人机/PX4 | ROS 话题 | 双向 | 毫秒~10Hz | ROS msg |
| L7 | β/A ↔ 历史库 | SQLAlchemy | 双向 | 按需 | SQL |
| L8 | β/A/小模型 ↔ 场地配置 | 文件读 | 单向 | 按需 | YAML(先导仅 home/boundary) |
| L9 | A ↔ DeepSeek | HTTPS | 双向 | 秒级 | OpenAI 格式(经 Pydantic AI,α 后端) |
| L10 | A ↔ 讯飞 STT/TTS | HTTPS | 双向 | 按需 | 讯飞 API(细节待补) |
| L11 | A 内组件 ↔ 组件 | 内存消息总线 | 双向 | 按需 | 统一总线消息(§3.7) |
| L12 | B 内组件 ↔ 组件 | 内存消息总线 | 双向 | 持续 | 统一总线消息(§3.7) |
| L13 | A ↔ 前端看板 | WS(`dashboard_config` A→前端) | A→前端 | 按需 | JSON(看板配置,β 工具触发) |

### 3.2 L5:A↔B 消息契约(统一总线,跨进程桥接)

消息帧: `[4字节长度][msgpack 载荷]`(length-prefixed,防粘包)。载荷为统一总线消息(§3.7 格式),A↔B 双向均用 msgpack。

| 方向(A↔B) | `msg_type` | `tool` | 含义 |
|---|---|---|---|
| A→B | `call` | `action` | **ActionCommand 下发(替代旧 `trajectory`)**:α 产出的动作编码,交 B 侧小模型生成目标点 |
| A→B | `call` | `abort` | 中止当前动作序列(打断预设) |
| A→B | `call` | `hover` | 安全悬停(无任务) |
| A→B | `event` | `ping` | 心跳/存活检测(2s) |
| B→A | `event` | `pose` | 当前位姿(高频,10Hz) |
| B→A | `event` | `telemetry` | 速度/加速度/IMU(高频) |
| B→A | `event` | `status` | 小模型/无人机状态(低频) |
| B→A | `event` | `reject` | **小模型/ego-planner 无法产出可达目标**,回告 α 重想(先导:小模型遇到未知动作编码/夹紧后仍越界) |
| B→A | `event` | `pong` | 心跳回应 |
| B→A | `event` | `alert` | 监控程序检测到异常(异常上告,见 §研究内容3) |

- **`schema_version` 升 1→2**:`trajectory`+TrajectorySpec 路径废弃,改 `action`+ActionCommand;`reject` 语义从"求解器拒某段"改为"小模型/ego-planner 无法生成可达目标"。
- B 侧恒定时间重连(非指数退避);A 侧 2s ping,5s 无 pong 视为断连。
- 异常 `alert`:B 侧监控程序检测到→经总线上行→A 转发前端(β 对话框系统消息)+ 唤醒 β 给处置建议(不自动执行)。
- 详见 [A-B 接口冻结](2026-07-05-A-B-接口冻结.md)。

### 3.7 开放式架构:统一消息总线与组件注册(研究内容 5)

**统一消息格式**(所有组件间通信,不论 A 内内存还是 A↔B IPC):

```json
{
  "schema_version": 2,
  "from": "beta",
  "to": "fft_analyzer",
  "msg_type": "call",
  "call_id": "uuid",
  "tool": "fft",
  "args": {"data": [...], "options": {...}},
  "payload": {},
  "ts": 1234567890
}
```

| 字段 | 说明 |
|---|---|
| `schema_version` | 协议版本(跨进程一致,当前 `2`) |
| `from` / `to` | 发送/接收组件名(或 `broadcast` / `human` / `frontend`) |
| `msg_type` | `call`(请求)/ `result`(响应)/ `event`(事件,如 pose/alert/status)/ `error` |
| `call_id` | 请求-响应配对(call 与 result 同 id) |
| `tool` | 调用的工具/动作名(如 `action`/`fft`/`propose_to_alpha`/`dashboard_configure`) |
| `args` / `payload` | 调用参数 / 事件数据 |
| `ts` | 时间戳 |

**组件注册表**(启动时静态注册):

| 组件 | 位置 | 接受的 `tool` |
|---|---|---|
| `alpha` | A | `translate`(β 调 forward/propose 触发),产出 `ActionCommand` |
| `beta` | A | 由前端/异常唤醒 |
| `small_model` | B | `generate_goal`(输入 `ActionCommand`+离散环境/状态 → 输出目标点;先导 stub 占位,阶段4 机载真模型) |
| `ego_planner` | B(阶段2/4) | `plan`(输入目标点+雷达点云 → 输出连续轨迹;先导不存在) |
| `lidar` | B(阶段2/4) | 持续产 `pointcloud` event;先导不存在 |
| `monitor` | B | 持续读 ROS 流,产出 `alert` 事件 |
| `fft_analyzer` / `stats` / `filter` | A(analytics/) | `fft`/`stats`/`filter`(β 调) |
| `history_query` | A | 历史查询(β 调) |
| `dashboard_driver` | A | `dashboard_configure`/`dashboard_set_filter`/`dashboard_list_panels`(β 调,转发前端) |

**传输分层:** 格式统一,A 内走内存总线(asyncio 队列,函数语义),A↔B 走 IPC(msgpack 帧),B 内走内存总线;阶段4 小模型/ego-planner 在机载,经机载↔云端链路(远期).组件不感知传输差异(总线抽象)。

**先导:** 静态注册(代码/配置列出组件),不做动态发现/热加载。后期可扩动态注册与发现(开放式架构演进)。

**组件接口契约:** 每个组件实现统一接口(接受总线消息、产出总线消息/事件),具体:
- α 实现 `ActionTranslator` 接口(输入指令→输出 `ActionCommand`),先导 LLM 实现,远期换蒸馏小模型 α 接口不变(见 [2026-07-05-小模型α-design.md](2026-07-05-小模型α-design.md))。
- 端侧小模型实现 `GoalGenerator` 接口(输入 `ActionCommand`+离散环境+状态 → 输出目标点),先导 stub→阶段4 真模型(见 [2026-07-07-端侧小模型-design.md](2026-07-07-端侧小模型-design.md))。
- ego-planner 实现 `Planner` 接口(输入目标点+点云 → 输出连续轨迹),阶段2/4。
- analytics 工具实现 `AnalyticsTool` 接口(输入数据+参数→输出结果)。
- 监控程序实现 `Detector` 接口(输入数据流→输出 alert 事件),检测器注册表(阈值/趋势先导,后期可加 ML 检测器)。
- 看板驱动器实现 `DashboardDriver` 接口(输入配置规格→前端渲染指令)。

**开放式接口规范文档**(研究产出 4,本轮交付):本节为摘要,完整接入标准见 [开放式接口规范.md](开放式接口规范.md)。指导后续新模型/程序快速集成。

### 3.6 失败/断连处理

| 场景 | 处理 |
|---|---|
| A↔B socket 断开 | B 维持最后位姿安全悬停,A 告警前端(WS link_status) |
| A 崩溃 | B 收不到新意图,维持安全悬停 |
| B 崩溃 | A 检测断连告警;无人机 ROS 节点独立可停 |
| 大模型 API 失败 | α/β 当前回合失败,不下发新意图(维持悬停),重试/告警 |
| 小模型推理失败/ego-planner reject | A 收到 `reject`,α 重新翻译(下个 loop 周期)或兜底 `hover` |
| 监控程序检测异常 | 经总线上行 alert → 前端 β 对话框系统消息 + 唤醒 β 给处置建议(不自动执行,人确认) |
| 监控程序本身故障 | B 记日志,告警前端;不影响飞行(降级:无监控) |
| analytics 工具调用失败 | 返回 error 给 β,β 告知人;不影响飞行 |

核心原则:任何环节失联/失败,无人机进入安全悬停;异常只告警+建议,不自动执行飞行处置(人确认)。

## 5. 历史数据 schema + 场地配置格式

### 5.1 数据库表结构(SQLAlchemy ORM)

```sql
-- ① 环境条件(JSON 灵活字段)
CREATE TABLE environments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    data TEXT NOT NULL,   -- JSON: {"weather":"晴","wind":[0.2,0.0,0.1],"humidity":45,...}
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- ② 试飞会话
CREATE TABLE flight_sessions (
    id TEXT PRIMARY KEY,                -- "20260630143012"(YYYYMMDDHHMMSS,与 ActionCommand.task_id 同值)
    created_at TEXT NOT NULL,
    environment_id INTEGER REFERENCES environments(id),
    task_description TEXT,
    beta_plan TEXT,                     -- β 产出的自然语言计划(动作意图概要)
    alpha_actions TEXT,                 -- α 翻译的 ActionCommand 序列 JSON(原 alpha_trajectory 已废弃)
    status TEXT DEFAULT 'idle'          -- idle/hovering/planned/executing/completed/aborted(对齐 StatusMessage.flightStatus)
);

-- ③ 轨迹时序(高频)
CREATE TABLE telemetry (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES flight_sessions(id),
    t REAL NOT NULL,
    position_x REAL, position_y REAL, position_z REAL,
    velocity_x REAL, velocity_y REAL, velocity_z REAL,
    accel_x REAL, accel_y REAL, accel_z REAL,
    angular_velocity_x REAL, angular_velocity_y REAL, angular_velocity_z REAL,
    quat_w REAL, quat_x REAL, quat_y REAL, quat_z REAL,
    UNIQUE(session_id, t)
);
CREATE INDEX idx_telemetry_session ON telemetry(session_id);

-- ④ 对话记录
CREATE TABLE conversations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES flight_sessions(id),
    agent TEXT NOT NULL,                -- 'alpha' / 'beta'
    role TEXT NOT NULL,                 -- 'human' / 'agent' / 'tool_call' / 'tool_result'
    content TEXT NOT NULL,
    metadata TEXT,                      -- JSON: {tool_name, tool_args, ...}
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_conv_session ON conversations(session_id);
```

关键设计:
- `environments.data` 用 SQLAlchemy `JSON` 列类型(SQLite→JSON,Postgres→JSONB),查询用 ORM JSON 操作。
- `telemetry` 用 `(session_id, t)` UNIQUE + 索引。
- `conversations.agent` 区分 α/β,`role` 区分人/agent/工具调用/返回。

### 5.2 数据写入时机

| 事件 | 写入 |
|---|---|
| 试飞开始 | INSERT `flight_sessions`(status=planned) |
| 环境手动输入 | INSERT/UPDATE `environments` |
| α 产出 ActionCommand | UPDATE `flight_sessions.alpha_actions`(JSON 序列) |
| 小模型产出目标点 | 可选 INSERT `conversations`(agent='small_model',role='tool_result') 作训练标的 |
| 飞行中(10Hz) | 批量 INSERT `telemetry`(缓冲,每秒一批) |
| α/β 每轮对话 | INSERT `conversations` |
| 试飞结束 | UPDATE `flight_sessions.status` |

### 5.3 场地配置(`config/field.yaml`,先导收缩版)

> **2026-07-07 重构**:原 `obstacles` 列表与求解器碰撞检测表已废弃(碰撞交阶段2/4 ego-planner 在线雷达避障)。先导 `field.yaml` 仅保留 `home` + `boundary` 软约束;阶段2/4 场地由雷达在线感知,`field.yaml` 可仅留 `home` 作返航参考。

```yaml
boundary:        # 软边界(先导物理安全网,B 侧 stub 小模型与假无人机夹紧兜底;阶段2/4 由 ego-planner 取代)
  x: [0.0, 5.0]
  y: [0.0, 4.0]
  z: [0.0, 3.0]

home:            # 返航点(ActionCommand code=return_home 目标)
  position: [0.0, 0.0, 0.5]
  yaw: 0.0

# obstacles: 已废弃(阶段2/4 用雷达点云)
```

> **先导物理安全**:无预编障碍物、无 ego-planner。物理安全靠①`ActionCommand.safety_constraints.speed_max/ceiling/floor/boundary`(α 随附)②B 侧 stub 小模型产出目标点时夹紧到 boundary ③假无人机边界自保(B spec §9.1)。
> **阶段2/4**:雷达点云 + ego-planner 在线避障;`boundary` 降为告警参考,不再用于夹紧。

### 5.4 默认安全约束(`config/default_constraints.yaml`,收缩版)

```yaml
global:
  speed_max: 1.5
  accel_max: 2.0
  angular_velocity_max: 0.5
  ceiling: 2.5
  floor: 0.3
  # keep_clear_distance 废弃(交 ego-planner 阶段2/4)

presets:
  careful:
    speed_max: 0.8
    accel_max: 1.0
  normal:
    speed_max: 1.5
    accel_max: 2.0
  fast:
    speed_max: 2.5
    accel_max: 3.0
```

约束值来源:全局默认 + α 按需覆盖进 `ActionCommand.safety_constraints`;碰撞相关约束废弃。

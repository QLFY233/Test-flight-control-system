# 试飞控制系统 — 总体架构设计

> 先导项目:打通"大模型理解试飞意图 → 翻译为无人机目标点 → 求解器生成安全轨迹 → 模拟无人机执行"完整链路。
> 本阶段无真实无人机,用 ROS1 模拟无人机代替。

### 1.2 组件边界

| 组件 | 对外接口(别人怎么用它) | 可独立改的内部 |
|---|---|---|
| 前端 | 只认 A 一个地址(Web 协议) | UI 布局、3D/2D 库 |
| 后端 A | 给前端:Web 协议;经消息总线调度组件 | β prompt、α 翻译器实现、analytics 算法 |
| 后端 B | 经消息总线上行 alert/pose;给无人机:ROS 话题 | 求解器算法、监控程序逻辑、ROS 节点结构 |
| 模拟无人机 | 给 B:ROS 位姿话题;从 B:ROS setpoint 话题 | 运动学模型 / PX4 SITL |
| 历史数据库 | 经 A 的仓储访问 | SQLite→Postgres 迁移 |
| 场地配置 | 文件,被 B/β 读 | 文件格式 |
| **可注册组件**:α / β / 求解器 / 监控程序 / 数据分析工具集 | 经消息总线注册,统一 `from/to/msg_type/call/tool/args/payload` 协议 | 各自实现,符合组件接口即可(α 后期换小模型) |

### 1.3 安全边界

- 大模型绝不直接碰飞控:α 只产出 TrajectorySpec,求解器做物理校验(场地内?会撞?速度合理?)才下发。
- β→α 的两种路径:
  - `propose_to_alpha(intent)`(β 自主想法):**必须人审核**后才注入 α,β 不直连求解器路径。
  - `forward_last_human_message()`(转发人刚对 β 说的指令):直接进 α 队列,无需再审核(人已发话)。
- 异常监控程序**只告警不建议直接执行**:异常经总线 → 前端(β 对话框系统消息)+ 唤醒 β 给处置建议,处置需人确认才执行。
- A↔B 消息总线桥接点是天然安全检查点。

### 1.4 功能范围

| 维度 | 决策 |
|---|---|
| 场地 | 室内先导(室外远期) |
| 场地建模来源 | 人工定义配置文件(YAML,场地建模=场地边界+障碍物配置,非 AI "建模") |
| 单/多机 | 单机 |
| 环境条件(天气/风速/湿度) | 前端手动输入,提供预设模板和历史记录复用 |

| 部署 | 单机,内网 HTTP,WSL2 mirrored 模式 |

## 2. 数据流与中枢调度

### 2.1 翻译链与中枢调度

> 龙虾模式:中枢大模型 β 调度多组件(α 翻译器、监控程序、数据分析工具)。人类只与 β 对话,不与 α 对话。

```
人类自然语言(语音/文字) → β(中枢) ──┬─→ 产出飞行计划 → 人审核 → 注入 α
                                        ├─→ forward_last_human_message → α
                                        └─→ 调数据分析工具 / 历史查询
α(翻译器,不对话) → TrajectorySpec → 求解器 → 连续轨迹 → 假无人机/PX4
异常监控程序(B) → alert(总线) → 前端(β 对话框系统消息)+ 唤醒 β 给处置建议
```

| 层 | 执行者 | 输入 | 输出 | 工具/依赖 | 频率 |
|---|---|---|---|---|---|
| L1 调度 | Agent β(中枢) | 人类自然语言(语音/文字)+ 异常唤醒 + 飞行数据 | 飞行计划 / 数据分析结果 / 处置建议 / 转发指令给 α | `propose_to_alpha`/`forward_last_human_message`/`analytics`/历史查询工具 | 按需 + 异常唤醒 |
| L1.5 审核 | 人 | β 的 `propose_to_alpha` 提议 | 确认/修改/驳回 | β 对话框 | 按需 |
| L2 翻译 | Agent α(**非对话**翻译器) | 人确认的 β 提议 / β 转发的人话指令 | TrajectorySpec(带逐段约束) | 读当前位姿/场地(只读) | loop,1~5 秒 |
| L3 求解 | 求解器(B 内组件) | TrajectorySpec | 连续轨迹(位置/速度/角速度/姿态时变曲线) | 场地配置(防碰撞) | 10~50Hz |
| L4 执行 | 假无人机/PX4 | 连续轨迹期望(setpoint) | 实际飞行 | ROS 话题 | 毫秒级 |
| L 异常 | 监控程序(B 内组件) | B 的 ROS 数据流 | alert(经总线上行) | 阈值+趋势检测器 | 持续 |

### 2.2 全链路约束体系

三层约束,后者覆盖前者:全局 → 段 → 点。约束值来源:全局默认配置 + α 按需覆盖。

```json
{
  "task_id": "20260630143012",
  "schema_version": 1,
  "global_constraints": {
    "keep_clear_distance": 0.5,
    "ceiling": 2.5,
    "floor": 0.3
  },
  "segments": [
    {
      "waypoints": [
        {
          "position": [3.0, 2.0, 1.0],
          "yaw": 0.0,
          "constraints": {
            "speed_max": 1.2,
            "accel_max": 2.0,
            "angular_velocity_max": 0.5
          }
        },
        {
          "position": [3.5, 2.5, 1.0],
          "constraints": {"speed_max": 1.0}
        }
      ],
      "segment_constraints": {
        "speed_max": 1.2,
        "accel_max": 2.0,
        "angular_velocity_max": 0.5
      }
    },
    {
      "waypoints": [
        {
          "position": [0.0, 0.0, 0.5],
          "return_home": true
        }
      ],
      "segment_constraints": {"speed_max": 0.8, "accel_max": 1.0}
    }
  ]
}
```

约束层级:点约束 > 段约束 > 全局约束 > 求解器默认。
可扩展:加新约束类型就是加新 key(如 `jerk_max`/`attitude_limit`/`time_window`)。求解器用约束注册表:认得的执行,不认得的忽略并告警。

**`mode` 字段取值**(`StatusMessage.mode`,映射当前执行段的操控来源):

| `mode` | 触发条件 |
|---|---|
| `manual` | 当前执行段由人通过 β Chat(`forward_last_human_message` 免审路径)直接驱动的 α 翻译产出 |
| `auto` | 当前执行段来自 β `propose_to_alpha` 提议经人审核批准后的 α 翻译产出(预设轨迹执行) |

人确认前的 β 提议、α 悬停兜底阶段沿用上一次合法 `mode`;非 LLM 小模型 α 后端切换不改 `mode` 语义。

### 2.3 α 翻译器职责与 loop 仲裁

> **α 不与人对话,不输出文字给人。** α 是纯翻译器:接收 β 转发/人确认的指令,翻译成 TrajectorySpec,经总线发求解器。未来用小参数非 LLM 小模型替代(研究内容 2),先导仍用 LLM 实现,接口留好。

α 在 loop 内的输入来源:

- 来源 i:β 的 `forward_last_human_message` 转发的人话指令(直接进 α 队列,无需审核,人已发话)。
- 来源 ii:人审核通过的 β 提议(`propose_to_alpha`,经人确认后进 α 队列)。
- 来源 iii:无人机数据(B 上行 pose/telemetry,loop 每周期读,作为翻译上下文)。

**α 的输入不再含"人直接打字给 α"——前端已取消 α 对话框,人只与 β 对话。**

**loop 仲裁逻辑:**

```
α loop 周期开始
  ├ 读取:当前位姿(从 B 上行)、当前执行中的预设段、α 输入队列(β 转发 + 人确认的提议)
  ├ 仲裁规则:
  │   if α 输入队列非空:
  │       本周期处理输入(翻译成 TrajectorySpec,打断预设)
  │   elif 执行中预设还有未完成段:
  │       本周期下发下一段目标点
  │   else:
  │       本周期下发"保持悬停"(安全默认)
  ├ 翻译:指令 → TrajectorySpec(调用 LLM,1~5 秒,非阻塞;未来换小模型)
  └ 输出:TrajectorySpec(经消息总线发求解器组件)
周期结束,新输入累入下个周期
```

**打断规则:**
- loop 单步内:新输入排队到下一周期(不抢占当前周期)。
- 预设轨迹整体:可被新输入打断(β 转发新人话 / 人确认新提议优先)。被打断的预设剩余部分丢弃或暂停。
- 安全默认:无任务、无新输入时 α 下发"悬停",飞控总有合法期望值。

**α 实现统一 `TrajectoryTranslator` 接口**(组件注册表的一项),先导 LLM 实现,后期换小模型实现此接口,总线调度不变。

### 2.4 β 工具集(中枢调度的组件)

β 是中枢大模型,通过工具调度各组件。工具分三类:

**历史/状态查询:**

| 工具 | 输入 | 输出 | 数据源 |
|---|---|---|---|
| `get_field_map()` | 无 | 场地边界+障碍列表 | 场地配置文件 |
| `get_current_pose()` | 无 | 当前位姿/速度 | B 上行状态(A 缓存) |
| `get_recent_telemetry(window_sec)` | 时间窗口 | 最近 N 秒遥测 | AppState 缓存 |
| `get_current_environment()` | 无 | 当前环境(environments.data JSON) | 本次会话 `environment_id` 对应行(未设返回 null) |
| `query_sessions(filter)` | 时间/任务等过滤 | 试飞会话列表 | 历史库 |
| `query_telemetry(session_id, range)` | 会话ID+时间范围 | 轨迹时序点 | 历史库 |
| `query_environment(env_id)` | 环境ID | 环境条件(JSON) | 历史库 |
| `query_conversations(session_id)` | 会话ID | 对话记录 | 历史库 |

**α 调度(向 α 发指令的两路径):**

| 工具 | 输入 | 输出 | 安全路径 |
|---|---|---|---|
| `propose_to_alpha(intent)` | β 的飞行意图 | 提交人审核(前端展示) | **人审核**后注入 α |
| `forward_last_human_message()` | 无(取人刚对 β 说的话) | 直接进 α 输入队列 | 免审核(人已发话) |

**数据分析(指令驱动数据处理,研究内容 4):**

| 工具 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `analytics_fft(data, options)` | 时序数据+参数 | 频谱结果 | 经 `analytics/` 组件,FFT |
| `analytics_stats(data, metric)` | 时序数据+指标 | 均值/方差/极值/趋势 | 统计分析 |
| `analytics_filter(data, type, params)` | 时序数据+滤波参数 | 滤波后数据 | 低通/高通/移动平均 |
| 未来扩展 | — | — | 工具注册表,加新工具不动框架 |

> β 可以"读飞行数据→分析→产出处置建议/计划"。数据分析工具是可注册组件,经总线调用,先导预定义,后期可扩(含代码执行模式,留接口)。

### 2.5 中枢调度协作(龙虾模式)

```
┌────────────────┐         人类(语音/文字)
│   前端          │ ◄──────────────────────────┐
│ (仅 β 对话框    │                            │
│  + α 视图状态)  │ ──系统消息(alert/进度)──►  │
└────┬───────────┘                            │
     │ β 对话流(SSE/WS)                      │
     ▼                                        │
┌────────────────────────────────────────────┴───┐
│  Agent β(中枢大模型)                          │
│  工具:历史查询 / analytics /                  │
│       propose_to_alpha / forward_last_human   │
└──┬───────────────┬────────────────┬───────────┘
   │ propose        │ forward        │ analytics call
   │ (人审核)       │ (直接)         │ (总线)
   ▼                ▼                ▼
┌──────┐       ┌────────────┐   ┌────────────┐
│ 人   │──确认──►│ Agent α     │   │ analytics/ │
│ 审核 │       │ (翻译器,    │   │ (FFT/统计) │
└──────┘       │  不对话)    │   └─────┬──────┘
               └──────┬─────┘         │
                      │ TrajectorySpec │ 结果回 β
                      ▼                │
               ┌────────────┐          │
               │ 求解器(B)   │◄─────────┘
               └──────┬─────┘
                      │ 连续轨迹
                      ▼
               ┌────────────┐
               │ 假无人机/   │
               │ PX4         │
               └──────┬─────┘
                      │ ROS 数据流
                      ▼
               ┌────────────┐
               │ 监控程序(B) │──alert(总线)──► 前端(β 系统消息)+ 唤醒 β
               └────────────┘
```

- 人类只与 β 对话(语音/文字),不与 α 对话。
- α 是翻译器,不与人对话,接收 β 转发/人确认的指令,翻译成 TrajectorySpec。
- β 经消息总线调度 α、analytics、历史查询等组件(龙虾模式)。
- 监控程序(B 侧组件)持续看飞行数据,异常经总线上行 alert,前端作为**系统消息插入 β 对话流**,同时唤醒 β 给处置建议。
- 各组件经统一消息总线通信(§3.7),α 后期可换小模型(接口不变)。
- 先导模型:DeepSeek-V4-Flash(OpenAI 格式,经 Pydantic AI openai provider)。

## 3. 通信 fabric 与协议契约

> 龙虾模式开放式架构:组件间经**统一消息总线**通信(§3.7)。前端↔A 仍用 Web 协议(SSE/WS/REST);A 内组件经内存总线;A↔B 经 Unix socket 桥接总线;B↔无人机经 ROS 话题。所有组件消息格式统一。

### 3.1 链路总表

| # | 链路 | 协议 | 方向 | 频率 | 消息格式 |
|---|---|---|---|---|---|
| L1 | 前端 ↔ A | WebSocket | 双向 | 10Hz~视频 | JSON(含 α_output 系统/alert 等,详见 §3.3) |
| L2 | 前端 ↔ A | SSE | A→前端 | 秒级(β 流式) | Pydantic AI UIAdapter 事件 |
| L3 | 前端 ↔ A | REST | 双向 | 按需 | JSON |
| L4 | 前端 ↔ A | StaticFiles | A→前端 | 一次 | HTML/JS/CSS |
| L5 | A ↔ B | Unix domain socket(总线桥接) | 双向 | 上行10Hz/下行秒级 | 统一总线消息,msgpack |
| L6 | B ↔ 假无人机/PX4 | ROS 话题 | 双向 | 毫秒~10Hz | ROS msg |
| L7 | β/A ↔ 历史库 | SQLAlchemy | 双向 | 按需 | SQL |
| L8 | β/A/求解器 ↔ 场地配置 | 文件读 | 单向 | 按需 | YAML |
| L9 | A ↔ DeepSeek | HTTPS | 双向 | 秒级 | OpenAI 格式(经 Pydantic AI) |
| L10 | A ↔ 讯飞 STT/TTS | HTTPS | 双向 | 按需 | 讯飞 API(细节待补) |
| L11 | A 内组件 ↔ 组件 | 内存消息总线 | 双向 | 按需 | 统一总线消息(§3.7) |
| L12 | B 内组件 ↔ 组件 | 内存消息总线 | 双向 | 持续 | 统一总线消息(§3.7) |

### 3.2 L5:A↔B 消息契约(统一总线,跨进程桥接)

消息帧: `[4字节长度][msgpack 载荷]`(length-prefixed,防粘包)。载荷为统一总线消息(§3.7 格式),A↔B 双向均用 msgpack。

| 方向(A↔B) | `msg_type` | `tool` | 含义 |
|---|---|---|---|
| A→B | `call` | `trajectory` | TrajectorySpec 下发求解器 |
| A→B | `call` | `abort` | 中止当前轨迹(打断预设) |
| A→B | `call` | `hover` | 安全悬停(无任务) |
| A→B | `event` | `ping` | 心跳/存活检测(2s) |
| B→A | `event` | `pose` | 当前位姿(高频,10Hz) |
| B→A | `event` | `telemetry` | 速度/加速度/IMU(高频) |
| B→A | `event` | `status` | 求解器/无人机状态(低频) |
| B→A | `event` | `reject` | 求解器拒绝某段,回告 α 重想 |
| B→A | `event` | `pong` | 心跳回应 |
| B→A | `event` | `alert` | 监控程序检测到异常(异常上告,见 §研究内容3) |

- B 侧恒定时间重连(非指数退避);A 侧 2s ping,5s 无 pong 视为断连。
- 异常 `alert`:B 侧监控程序检测到→经总线上行→A 转发前端(β 对话框系统消息)+ 唤醒 β 给处置建议(不自动执行)。

### 3.7 开放式架构:统一消息总线与组件注册(研究内容 5)

**统一消息格式**(所有组件间通信,不论 A 内内存还是 A↔B IPC):

```json
{
  "schema_version": 1,
  "from": "beta",
  "to": "fft_analyzer",
  "msg_type": "call",
  "call_id": "uuid",
  "tool": "fft",
  "args": {"data": [...], "options": {...}},
  "payload": {},
  "ts": 1234567890
}
```

| 字段 | 说明 |
|---|---|
| `schema_version` | 协议版本(跨进程一致) |
| `from` / `to` | 发送/接收组件名(或 `broadcast` / `human` / `frontend`) |
| `msg_type` | `call`(请求)/ `result`(响应)/ `event`(事件,如 pose/alert/status)/ `error` |
| `call_id` | 请求-响应配对(call 与 result 同 id) |
| `tool` | 调用的工具/动作名(如 `trajectory`/`fft`/`propose_to_alpha`) |
| `args` / `payload` | 调用参数 / 事件数据 |
| `ts` | 时间戳 |

**组件注册表**(启动时静态注册):

| 组件 | 位置 | 接受的 `tool` |
|---|---|---|
| `alpha` | A | `translate`(β 调 forward/propose 触发) |
| `beta` | A | 由前端/异常唤醒 |
| `solver` | B | `trajectory`/`abort`/`hover` |
| `monitor` | B | 持续读 ROS 流,产出 `alert` 事件 |
| `fft_analyzer` / `stats` / `filter` | A(analytics/) | `fft`/`stats`/`filter`(β 调) |
| `history_query` | A | 历史查询(β 调) |

**传输分层:** 格式统一,A 内走内存总线(asyncio 队列,函数语义),A↔B 走 IPC(msgpack 帧),B 内走内存总线。组件不感知传输差异(总线抽象)。

**先导:** 静态注册(代码/配置列出组件),不做动态发现/热加载。后期可扩动态注册与发现(开放式架构演进)。

**组件接口契约:** 每个组件实现统一接口(接受总线消息、产出总线消息/事件),具体:
- α 实现 `TrajectoryTranslator` 接口(输入指令→输出 TrajectorySpec),后期换小模型接口不变。
- analytics 工具实现 `AnalyticsTool` 接口(输入数据+参数→输出结果)。
- 监控程序实现 `Detector` 接口(输入数据流→输出 alert 事件),检测器注册表(阈值/趋势先导,后期可加 ML 检测器)。

**开放式接口规范文档**(研究产出 4):本节及 §3.2 共同构成组件接入标准,指导后续新模型/程序快速集成。

### 3.6 失败/断连处理

| 场景 | 处理 |
|---|---|
| A↔B socket 断开 | B 维持最后位姿安全悬停,A 告警前端(WS link_status) |
| A 崩溃 | B 收不到新意图,维持安全悬停 |
| B 崩溃 | A 检测断连告警;无人机 ROS 节点独立可停 |
| 大模型 API 失败 | α/β 当前回合失败,不下发新意图(维持悬停),重试/告警 |
| 求解器 reject | A 收到,α 重新翻译(下个 loop 周期) |
| 监控程序检测异常 | 经总线上行 alert → 前端 β 对话框系统消息 + 唤醒 β 给处置建议(不自动执行,人确认) |
| 监控程序本身故障 | B 记日志,告警前端;不影响飞行(降级:无监控) |
| analytics 工具调用失败 | 返回 error 给 β,β 告知人;不影响飞行 |

核心原则:任何环节失联/失败,无人机进入安全悬停;异常只告警+建议,不自动执行飞行处置(人确认)。

## 5. 历史数据 schema + 场地配置格式

### 5.1 数据库表结构(SQLAlchemy ORM)

```sql
-- ① 环境条件(JSON 灵活字段)
CREATE TABLE environments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    data TEXT NOT NULL,   -- JSON: {"weather":"晴","wind":[0.2,0.0,0.1],"humidity":45,...}
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- ② 试飞会话
CREATE TABLE flight_sessions (
    id TEXT PRIMARY KEY,                -- "20260630143012"(YYYYMMDDHHMMSS,与 TrajectorySpec.task_id 同值)
    created_at TEXT NOT NULL,
    environment_id INTEGER REFERENCES environments(id),
    task_description TEXT,
    beta_plan TEXT,                     -- β 产出的自然语言计划
    alpha_trajectory TEXT,              -- α 翻译的点序列 JSON
    status TEXT DEFAULT 'idle'          -- idle/hovering/planned/executing/completed/aborted(对齐 StatusMessage.flightStatus)
);

-- ③ 轨迹时序(高频)
CREATE TABLE telemetry (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES flight_sessions(id),
    t REAL NOT NULL,
    position_x REAL, position_y REAL, position_z REAL,
    velocity_x REAL, velocity_y REAL, velocity_z REAL,
    accel_x REAL, accel_y REAL, accel_z REAL,
    angular_velocity_x REAL, angular_velocity_y REAL, angular_velocity_z REAL,
    quat_w REAL, quat_x REAL, quat_y REAL, quat_z REAL,
    UNIQUE(session_id, t)
);
CREATE INDEX idx_telemetry_session ON telemetry(session_id);

-- ④ 对话记录
CREATE TABLE conversations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES flight_sessions(id),
    agent TEXT NOT NULL,                -- 'alpha' / 'beta'
    role TEXT NOT NULL,                 -- 'human' / 'agent' / 'tool_call' / 'tool_result'
    content TEXT NOT NULL,
    metadata TEXT,                      -- JSON: {tool_name, tool_args, ...}
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_conv_session ON conversations(session_id);
```

关键设计:
- `environments.data` 用 SQLAlchemy `JSON` 列类型(SQLite→JSON,Postgres→JSONB),查询用 ORM JSON 操作。
- `telemetry` 用 `(session_id, t)` UNIQUE + 索引。
- `conversations.agent` 区分 α/β,`role` 区分人/agent/工具调用/返回。

### 5.2 数据写入时机

| 事件 | 写入 |
|---|---|
| 试飞开始 | INSERT `flight_sessions`(status=planned) |
| 环境手动输入 | INSERT/UPDATE `environments` |
| α 产出点序列 | UPDATE `flight_sessions.alpha_trajectory` |
| 飞行中(10Hz) | 批量 INSERT `telemetry`(缓冲,每秒一批) |
| α/β 每轮对话 | INSERT `conversations` |
| 试飞结束 | UPDATE `flight_sessions.status` |

| 形状 | 方法 |
|---|---|
| `box` | 采样点到 AABB 距离 ≤ keep_clear_distance |
| `cylinder` | 采样点到圆柱轴心 XY 距离 ≤ radius + keep_clear_distance(且 z ∈ [0,height]) |
| `sphere` | 采样点到球心距离 ≤ radius + keep_clear_distance |
| `plane` | 采样点是否在法向禁止侧(+ keep_clear_distance 余量) |
| `capsule` | 采样点到胶囊体轴线距离 ≤ radius + keep_clear_distance(先导未实现,见下) |

可扩展:加新形状 = 加 type + 对应参数 + 求解器加碰撞函数。先导实现 box/cylinder/sphere/plane;未实现 type → 配置加载阶段拒绝(启动报错),不静默忽略。capsule 后补。

> **对齐说明**:本表与后端 B spec §4.4 逐字一致(先导采样点法);原 v1 表的"段求交"表述已废弃。

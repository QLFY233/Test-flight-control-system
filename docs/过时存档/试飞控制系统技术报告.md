# 试飞控制系统技术报告

---

## 摘要

本报告围绕"龙虾模式智能辅助系统构建及无人机验证技术"展开，系统阐述了一套面向民航试飞机载应用的智能辅助最小系统的规划设计方案。系统采用"中枢大模型动态调度多个专用组件"的混合架构，以四旋翼无人机为验证平台，旨在实现自然语言驱动的飞行任务规划与执行、实时数据异常监控、指令驱动的数据分析处理等功能。报告从项目背景、系统架构、实现流程、模块设计、关键技术、项目价值等方面进行了全面论证，并在规划阶段完成了全套架构规格与开放式接口规范的设计。项目响应了国家"人工智能+"行动和民航高质量发展的政策号召，为下一代智能航电系统探索了可行的技术路径。

**关键词**：龙虾模式；智能体架构；无人机试飞；中枢大模型；边缘执行模型；人机协同；开放式架构

# 一、项目背景与立项依据

## 1.1 民航试飞与驾驶舱现状

当前民航驾驶舱系统相对封闭，飞行员在日常运行中需要查阅大量技术手册、执行繁琐的标准操作程序，并实时监控众多仪表与数据，认知负荷始终处于较高水平。这一问题在试飞阶段尤为突出：试飞任务需要验证航空器各项性能指标，任务执行与数据分析高度依赖试飞员和试飞工程师的人工判断，从飞行前任务规划、飞行中数据监控到飞行后数据分析，整个流程耗时漫长，效率与准确性均受限于人为因素。

具体而言，试飞阶段的挑战体现在三个维度。第一，**任务规划复杂**：试飞需要根据验证目标编排特定的机动序列，涉及高度、速度、姿态、构型等多种参数的精确设定，传统方式下试飞员和工程师需要花费大量时间查阅试飞大纲和手册，逐一核验参数的合规性与安全性。第二，**实时监控压力大**：飞行过程中试飞员需要同时关注飞行状态、发动机参数、结构响应等多个维度的数据，任何异常都需要及时发现并做出正确处置，这对人的精力和判断力构成极大考验。第三，**数据分析滞后**：飞行结束后数据的整理、清洗、分析通常需要数小时甚至数天才能完成，这意味着试飞发现问题、调整方案的周期较长，拉低了整个试飞项目的效率。

与此同时，新一代航空器（包括新能源飞机、超音速客机、城市空中交通飞行器等）的研发正在加速推进，这些新构型、新动力系统的试飞复杂度远超传统机型，进一步加剧了人工操作模式的瓶颈。以新能源飞机为例，其动力系统涉及电池热管理、电力推进系统效率、能量回收等多个传统飞机所没有的维度，意味着试飞中需要监控的数据类型和数量大幅增加——单靠试飞员和试飞工程师的人力已难以胜任。如何在试飞场景中引入智能化辅助手段，降低人员负担、提升效率与安全，已成为航空领域亟待解决的问题。

## 1.2 国家方针指引

本项目的研究方向与多个国家层面的政策方针高度契合。

《新一代人工智能发展规划》（国发〔2017〕35号）明确提出，要推动人工智能与各行业融合发展，加快关键技术转化应用，培育智能化高端服务。人工智能技术在航空领域的深度应用，正是这一规划的重要落脚点。

《国务院关于深入实施"人工智能+"行动的意见》进一步强调，要加快人工智能与低空飞行等技术的融合和产品创新，探索智能产品新形态。该文件将人工智能与低空经济的结合上升为国家行动层面，为航空智能化研究提供了明确的政策导向。

民航发〔2025〕24号《关于推动"人工智能+民航"高质量发展的实施意见》是最为直接的行业政策依据。文件在"人工智能+安全"领域明确提出三项与本项目高度相关的任务方向：**推进飞行风险主动感知技术研发**，使系统能够自动检测飞行中的异常态势，这一方向在本项目中由数据监控程序具体承接——通过阈值和趋势检测实现飞行数据异常的实时主动感知；**推进机组智能辅助技术研发**，通过智能系统协助飞行员完成部分认知和操作任务，减少其工作负荷，这一方向在本项目中由中枢大模型具体承接——通过自然语言交互、飞行方案制定和工具调度实现机组智能辅助；**推进飞行数据智能研判技术研发**，利用人工智能对大量飞行数据进行深度分析，从海量时序数据中提取潜在价值信息，这一方向在本项目中由数据分析工具集具体承接——通过傅里叶分析、统计计算、信号滤波等自动化数据处理实现智能研判。此外，文件还提出要构建民航高质量行业数据集，研发民航垂直领域人工智能模型和多协同智能应用体系，并特别鼓励"依托无人驾驶航空器开展机载新技术前期试验验证，降低载人航空器试飞成本与安全风险"。本项目的全链路规划——从先导期的仿真验证到远期外场无人机飞行演示，并最终向民航载人航空器迁移——完全吻合文件所鼓励的"无人机先导验证→降低载人航空器试飞成本与风险"的技术路径。

中国民航局《"十四五"民用航空发展规划》同样强调了智慧民航建设的重要性，提出以数字化、智能化技术赋能民航全产业链，提升安全水平与运行效率。

上述政策文件的密集出台表明，人工智能赋能民航已成为国家战略和行业共识，本项目所聚焦的"智能辅助驾驶舱系统"和"基于无人机的先导验证"两个方向，正是政策落地的典型场景。

## 1.3 法律法规与适航要求

《无人驾驶航空器飞行管理暂行条例》（国务院、中央军委，2024年1月1日起施行）对无人机的空域使用、飞行审批、运营资质等做出了系统规定。本项目在规划阶段已将合规要求纳入考量：试验飞行在仿真环境（先导阶段）和受控外场空域（远期外场演示阶段）中进行，严格遵守空域管理要求。

《民用航空器适航管理条例》及CCAR-21《民用航空产品和零部件合格审定规定》规定了航空器系统取得适航认证的程序和要求。本项目目前处于先导技术验证阶段，暂不涉及适航取证；但系统架构设计中充分考虑了对未来适航要求的适应性，包括安全兜底机制（断连悬停）、人在回路决策（关键操作须人工确认）、可审计性（所有指令和响应全量记录）等设计原则，为远期载人航空器部署奠定合规基础。

## 1.4 项目需求与核心要求

本项目的研究目标可概括为：**构建一个面向民航机载应用的智能辅助最小系统，形成开放式接口规范，并以无人机为验证平台进行全流程演示，验证系统协同能力及向民航迁移的可行性**。其核心目标是证明"中枢大模型调度多组件"的混合架构，能够在飞行环境中可靠实现知识问答、任务自动执行与数据监控处理等民航典型需求。

项目涵盖五项核心研究内容：

1. **智能调度与问答引擎**：基于大型语言模型搭建智能体，实现语音和文字交互式问答、任务的自动解析与调度。这是整个系统的"大脑"，负责理解人的意图并协调各类组件工作。

2. **试飞动作执行模型训练**：构建飞行动作数据集，训练轻量化模型，使其能够根据环境数据和飞行状态，将高层动作指令转化为可执行的控制目标，完成规定的飞行动作。这一模型需要在无人机机载电脑上本地运行，不依赖云端。

3. **数据异常监控程序**：开发实时数据监控程序，接入飞行数据流，依据预设阈值与趋势分析逻辑检测异常，并触发调度决策，生成告警信息与处置建议。

4. **指令驱动的数据处理**：利用大模型的工具调用能力，将操作人员的自然语言指令解析为对特定数据处理程序的调用（如傅里叶变换分析、统计分析、滤波处理），并返回处理结果。

5. **开放式架构设计**：定义统一的组件注册、发现与通信协议，使中枢大模型能够灵活调度各类专用组件和程序，支持后续功能的无缝扩展。

在架构设计层面，项目经历了关键的方案深化，确立了以下核心架构要求：（1）在无人机机载侧部署一个数兆级别的轻量神经网络作为"边缘执行模型"，输入离散环境数据、无人机状态参数和动作编码，输出可对接路径规划器的目标点；（2）去掉传统方案中的预编场景碰撞检测求解器，改由路径规划器接收实时雷达点云进行在线避障；（3）离散数据翻译模型输出动作编码与数值的结构化数据，再由系统转化为边缘执行模型的离散输入；（4）明确各模块在统一通信总线上的位置，形成通用的开放式接口规范；（5）前端提供可配置的数据展示看板，具体展示内容和筛选方式由中枢大模型通过工具调用动态决定。

## 1.5 国内外研究现状

在全球范围内，将人工智能引入飞行驾驶舱已成为航空领域研究的热点方向。欧洲航空安全局（EASA）通过HAIKU项目提出了人-AI协同的分级框架，从纯建议系统到能够主动执行任务的智能体，逐级定义了自主权能的边界。该框架强调：智能助手必须设计为与飞行员协作而非取代飞行员，系统需要透明地展示自己的推理过程，并随时接受飞行员的干预。

英国克兰菲尔德大学Saunders等学者2026年发表在《The Aeronautical Journal》上的研究提供了关键的方法论参考。该研究使用"未来系统模拟器"对四名专业飞行员进行了发动机故障场景下的智能助手评估，在飞行员失能条件下测试了自主动作执行任务。研究识别出多项对系统设计具有指导意义的原则，其中与本项目紧密相关的包括：

- **情境适应性权威分配**：当飞行员在座时，系统应保持较低的自主权限，关键切换操作必须经飞行员交叉确认；当飞行员因故离座或失能时，系统可适当提升自主权限以保障飞行安全。
- **信息通道的合理分工**：驾驶舱内音频通道高度拥挤，语音播报应优先用于关键告警和概要信息，使用提示音比长篇播报更有效。
- **充分的交叉检查时间**：自主动作之间应预留足够的延迟，使飞行员有时间评估系统已完成的操作并决定是否介入。
- **返回驾驶舱时的态势恢复**：系统应提供清晰、带有时间标记的动作执行记录和决策理由摘要，帮助飞行员在回到座位后快速重建对系统当前状态的理解。
- **关键动作不可自动执行**：涉及燃油切断、发动机关闭等关键切换的指令，在任何情况下都必须由飞行员确认后方可执行。

上述研究聚焦于大型客机在飞行员失能场景下的智能辅助，使用的智能助手功能相对单一（专注于检查单自动执行）。本项目在此基础上将研究边界拓展到试飞场景和无人驾驶航空器验证，核心差异在于：一是采用"龙虾模式"智能体架构——由一个中枢大模型动态调度多个专用组件（而非单一的智能助手），实现从任务规划、动作翻译、执行控制到数据监控的全链路智能化；二是通过无人机平台进行先导验证，降低技术风险和试飞成本，为民航迁移积累经验。这也是民航发〔2025〕24号文件所鼓励的技术路径。

论文研究的另一重要启示是飞行员对系统透明度的需求。参与实验的飞行员明确表示，他们需要理解系统正在做什么以及为什么这样做——不仅是出于信任，更是因为在紧急情况下飞行员必须快速判断是"信任系统让其继续自动执行"还是"介入接管"。如果系统不透明，飞行员缺乏足够的信息来做这个判断，要么过于信任系统（可能错过真问题），要么过于不信任（频繁打断系统的有效操作）。本项目在设计中将透明度作为核心考量：每次中枢大模型提出方案时，不仅呈现代办动作列表，还附带说明"为什么建议这些动作"的依据（查阅了哪些场地数据、对比了哪些历史记录、考虑了哪些环境约束）；异常告警不仅展示"发生了什么"，还附带中枢大模型对异常原因的分析推断。这种"解释性的智能"是区别本系统与传统的黑箱自动化系统的关键特征。

# 二、项目目标与总体方案

## 2.1 研究目标

本项目的总体目标是：**构建一套基于"龙虾模式"的民航智能辅助最小系统，定义开放式接口规范，并以四旋翼无人机为验证平台进行全流程地面仿真与外场飞行演示**。

"龙虾模式"是本研究提出的核心架构主张，其命名源于龙虾的神经结构特点——龙虾拥有一个中枢神经节，专门负责协调各附肢的传感与运动。类比到智能系统设计中，即由一个中枢大模型（对应于龙虾的中枢神经节）作为核心调度中心，动态组织和调用多个专用的小模型和工具程序（对应于龙虾的附肢），各组件通过统一的通信总线完成信息交换与任务协作。这种架构的优势在于：

其一，**职责分离**：中枢大模型专注于高层语义理解、任务规划和调度决策，而各专用组件专注于其擅长的细分任务——翻译飞机控制动作、生成飞行目标点、执行数据监控与分析、驱动前端展示等。每个组件各司其职，架构清晰、便于独立升级和维护。

其二，**可扩展性**：新的组件只要遵循统一的接口规范，即可注册到消息总线上被中枢大模型调度，无需修改现有组件或核心框架。这为系统功能的持续增强提供了开放的扩展路径。

其三，**安全可控**：不同组件处于不同的安全层级，飞控层的组件（如边缘执行模型、路径规划器）与规划层的组件（如中枢大模型）在物理和逻辑上隔离，确保高层智能决策的错误不会直接传导到飞行控制层面。

项目旨在通过无人机平台验证上述架构在真实飞行环境（含物理约束、通信延迟、环境干扰等）中的可行性与协同能力，检验"自然语言指令→任务规划→动作翻译→目标点生成→路径规划→飞控执行→数据回传与分析"这一完整闭环的效果。

## 2.2 五项研究内容对应

围绕总体目标，项目设定了五项系统性的研究内容，分别对应智能辅助系统的不同功能层面。

**研究内容一：智能调度与问答引擎。** 设计中枢大模型（即龙虾模式中的"中枢"）的核心能力：接收操作人员通过语音或文字发出的自然语言指令和查询，理解其意图，调用适当的工具组件完成操作或返回信息。同时，中枢大模型能够自主分析历史飞行数据和当前环境状态，主动向操作人员提出飞行方案建议，供人审核后执行。中枢大模型的工具调度能力覆盖场地与历史数据查询、数据分析工具调用、飞行方案下发、前端看板配置等全系统功能。

**研究内容二：试飞动作执行模型（边缘执行模型）。** 在无人机机载电脑上部署一个轻量级的动作执行模型，其输入为离散化的环境数据（风速、风向、气温、湿度等）、无人机当前状态参数（位置、速度、姿态、电池电量等）以及来自离散数据翻译模型的动作编码和数值（例如"动作类型：上升，数值：80米"），输出为路径规划器可直接使用的目标点序列。这个模型的体量控制在数兆字节级别，可在机载电脑上完成本地推理，无需依赖云端通信，从而保障响应延迟和控制可靠性。

**研究内容三：数据异常监控程序。** 开发实时数据监控组件，持续接入飞行数据流（位置、速度、加速度、角速度、航姿等），基于预设阈值（如速度上限、高度上下限）和趋势分析逻辑（如速度突变检测、姿态抖动检测、数据停滞告警）自动检测异常。当异常被确认后，监控程序通过通信总线上报告警事件，触发中枢大模型分析异常并生成处置建议——但处置建议须经操作人员确认后方可执行，不进入自动处置流程。

**研究内容四：指令驱动的数据处理。** 赋予中枢大模型调用专业数据处理程序的能力。操作人员可以通过自然语言指令要求系统执行特定的数据分析任务，例如："对这一段飞行轨迹的垂直方向数据进行频谱分析"、"统计本次飞行中姿态偏差超过5度的次数"。中枢大模型自动解析人的意图，将分析任务转化为对傅里叶变换、统计分析、信号滤波等工具程序的调用参数，获取结果后以人类可读的方式返回。

**研究内容五：开放式架构设计。** 定义一套统一的组件接入标准，涵盖组件注册规范（如何声明一个组件及它提供的功能）、消息通信协议（组件之间如何收发消息）和发现调度机制（中枢大模型如何找到并调用一个组件）。该标准是本项目的核心产出之一，将独立成文，指导后续新功能的快速集成。

## 2.3 验证平台选择

本项目选择四旋翼无人机配合ROS（机器人操作系统）仿真环境作为验证平台，平台选择基于以下考量。

首先，四旋翼无人机具有机械结构简单、控制相对成熟、采购和维护成本低等优势，适合作为先导验证的飞行载体。其三维空间运动能力涵盖位置、速度、姿态的完整控制，能够充分验证系统各功能模块的协同效果。

其次，采用"先导仿真→半实物仿真→外场飞行"三步走策略。先导阶段使用纯软件仿真的"假无人机"节点，该节点接收飞行控制指令后，通过运动学积分模拟真实飞行器的位置和姿态变化，并回传模拟数据。这种全仿真环境无需任何硬件，可以在短时间内进行大量迭代测试，验证架构和协议的正确性。第二阶段接入PX4开源飞控的软件在环仿真，使用更逼近真实飞行器物理特性的飞控模型。第三阶段进行外场实际飞行演示，全面评估系统在真实环境中的链路延迟、动作执行准确率、异常检测响应时间及系统扩展能力。

这一平台策略直接呼应了民航发〔2025〕24号文件"鼓励依托无人驾驶航空器开展机载新技术前期试验验证，降低载人航空器试飞成本与安全风险"的指引。通过无人机验证积累的经验和数据，将为后续向民航载人飞机迁移提供关键支撑。

## 2.4 四阶段研究路径

本项目规划为四个递进阶段，每个阶段有明确的目标和产出。

**阶段一：需求与架构设计。** 深入分析民航试飞和驾驶舱操作场景，结合国内外研究现状，确定系统的顶层架构方案。重点完成：龙虾模式各组件的逻辑划分与功能边界定义；中枢大模型、离散数据翻译模型、边缘执行模型、监控程序、数据处理工具等组件之间的交互协议制定；开放式接口标准的初版编写。本阶段的产出物包括：系统总体架构规格、各子系统详细设计说明、开放式接口规范文档、跨进程通信协议定义等。

**阶段二：核心组件开发。** 分别开发和集成本系统的各个核心软件组件。在后端智能中枢一侧：接入云端大模型服务实现中枢调度与问答功能；实现离散数据翻译模型的指令翻译能力（将自然语言意图转化为结构化动作编码序列）。在后端执行桥一侧：开发边缘执行模型的先导版本（基于规则的动作编码到目标点映射，后续阶段替换为神经网络推理）；实现实时数据监控程序；开发与ROS仿真无人机的接口。在前端一侧：完成网页界面开发，包括三维飞行轨迹渲染、状态面板、对话交互、历史记录查看等功能。在集成层面：实现组件间的消息总线通信与跨进程桥接。

**阶段三：集成联调与地面仿真。** 在仿真环境中连接云端大模型和本地各个组件，打通"任务下发→中枢大模型解析→离散数据翻译模型翻译→边缘执行模型生成目标点→仿真无人机执行→数据回传与监控告警→中枢大模型生成处置建议"的完整链路。在此阶段发现并消除各组件之间的接口冲突和时序问题，并通过重复执行预定义的试飞任务序列，验证系统的基本功能完整性和跨组件协同的正确性。

**阶段四：外场无人机演示与评估。** 在真实无人机平台上开展飞行演示。测试不同工况下（不同试飞任务类型、不同环境条件、模拟通信中断等异常场景）的系统表现，收集链路延迟、动作执行精度、异常检测响应时间等关键性能指标。全面评估系统在真实物理环境中的可行性与可靠性，完成项目总结报告。

# 三、系统总体架构

## 3.1 架构总览：龙虾模式

本系统的核心架构理念是"龙虾模式"——由一个中枢大模型担任总调度，通过统一的通信总线（消息总线）连接和协调多个专用组件，共同完成从任务理解到飞行执行的全流程。

在架构全景中，中枢大模型位于逻辑中心，它不直接控制无人机，而是通过发出结构化指令来调动以下"附肢"组件：

- **离散数据翻译模型**：负责将操作人员（直接给出）的或中枢大模型（提案经人审后）的飞行意图翻译为标准化的动作编码序列。每一条动作编码包含动作类型（如起飞、上升、水平移动、悬停、下降、返航等）和对应的数值参数（如上升多少米、移动到哪个坐标点、悬停多少秒等）。该模型不与人直接对话，仅根据输入的意图文本输出结构化的动作指令。
  
- **边缘执行模型**：部署在无人机机载电脑（或仿真环境中的模拟端）的轻量化模型，负责将翻译模型产出的动作编码与当前的离散环境数据（风速、气温、湿度等）和无人机状态（当前位置、速度、姿态、电量等）结合起来，生成路径规划器可直接使用的目标点。该模型在本地完成推理，保障低延迟和高可靠性。

- **路径规划器与雷达感知**：路径规划器接收来自边缘执行模型的目标点以及来自雷达的实时三维点云数据，进行在线避障路径计算，生成连续的安全飞行轨迹。这一组件替代了传统方案中需要人工预先编码场景和障碍物的做法，使系统能够适应未知和动态变化的飞行环境。

- **数据监控程序**：持续读取飞行数据流，依据预设的阈值和趋势分析逻辑检测数据异常。当检测到速度超限、高度越界、数据突变或数据停滞等异常时，立即通过消息总线发出告警事件。

- **数据分析工具集**：包括傅里叶变换（频谱分析）、统计计算、信号滤波等专业数据处理工具。这些工具由中枢大模型通过工具调用功能调度，使操作人员可以用自然语言请求复杂的数据分析操作。

- **历史数据查询**：提供对过往飞行记录的查询能力，包括历史环境条件、飞行轨迹时序数据、操作记录（中枢大模型和离散数据翻译模型的输入输出）、异常事件记录等。历史查询是中枢大模型进行方案制定的知识基础。

- **看板驱动器**：根据中枢大模型的决策，向前端数据展示看板发送配置指令，决定当前展示哪些数据面板以及采用什么筛选条件，实现展示内容的智能动态配置。

- **历史数据库**：存储每次飞行的完整记录，包括环境条件、飞行轨迹（以10赫兹频率采样的完整时间序列）、对话和指令记录、异常事件等。数据库是系统记忆的核心，支撑历史回看、事后分析和模型训练。

所有组件之间的通信均采用统一的消息格式，经由消息总线进行路由分发。组件不需要知道其他组件在哪个进程、哪台机器上运行——总线层根据组件注册表自动处理跨进程或跨机器的消息传输。这一设计使得任何遵循接口规范的新组件都可以随时注册上线，被中枢大模型发现和调度。

这一架构的深层理念在于**通过标准化接口实现系统复杂性的可控分解**。每个组件只需要实现两个能力：一是能够接收消息总线传来的标准格式请求，二是能够将处理结果以标准格式回复到消息总线上。组件内部的实现技术、编程语言、运行环境完全由组件开发者自行选择——只要消息格式一致，消息总线就能连接它们。举例来说，中枢大模型运行在Python 3.10环境中，使用云端大模型API；边缘执行模型运行在Python 3.8环境中（因为需要和ROS交互）；路径规划器远期可能在C++环境中运行以获得更高性能；前端看板运行在浏览器的JavaScript环境中。这些技术栈完全不同的组件之所以能够协同工作，正是因为在它们之上覆盖了一层统一的消息总线——总线屏蔽了语言差异、环境差异和物理位置差异，只暴露一种标准化的通信方式。这种"接口统一、实现各异"的设计，是龙虾模式架构工程可实施性的核心保障。

## 3.2 五大组件定位

为了更直观地理解系统构成，可以将系统归纳为五个逻辑组件，每个组件承担不同的职责，协同运作。

**前端交互界面。** 这是操作人员与系统交互的窗口，同时支持电脑网页和手机两种使用方式。前端界面包含以下核心功能区域：右侧主力视图区——默认显示三维飞行轨迹图（支持旋转、缩放、平移操作），可根据需要切换为视频回传画面（远期）或二维数据图表；左侧控制面板——显示环境状态（风速、湿度等）、任务进度、控制选项等信息；中部（或底部）对话区——操作人员在此与中枢大模型进行文字或语音对话，查看系统生成的飞行方案并完成审核确认；悬浮快捷控制球——始终显示在界面右下角，点击后展开扇形快捷操作菜单，支持自定义快捷键。手机版本采用上下分栏布局，上栏可滑动切换三维视图和数据面板，下栏为对话与输入区域。

前端的设计哲学是"场景适配"而非"一刀切"。在电脑端，屏幕宽大且操作人员通常坐在桌面操作台前使用键盘鼠标——界面采用左右分栏布局，左栏管理"状态与控制"，右栏呈现"空间视图"，确保三维飞行场景有充足的显示面积。在手机端，屏幕窄小且操作人员可能手持设备或将其固定在操控台上——界面改为上下分栏，上部默认显示三维轨迹图（通过左右滑动切换数据视图），下部为对话区和控制区，方便拇指操作。两种终端共享同一套功能逻辑，通过自适应布局适配不同设备形态，无需开发两套独立应用。

**后端A——智能中枢。** 这是整个系统的"大脑"，运行在Linux服务器上，包含两大核心智能体和一系列支持模块。中枢大模型是系统的总调度中心和规划师，负责理解操作人员的自然语言意图、查阅场地和历史数据并制定飞行方案，以及将方案提交给操作人员审核。中枢大模型不直接操控飞机——它只提出"做什么"（例如在场地指定范围内执行一组试飞机动），具体"怎么做"由下游的执行模型和路径规划器负责。离散数据翻译模型是一个专门化的动作翻译器，它不与人直接交流，只负责将操作人员直接给出的指令或经人审核通过的中枢大模型方案，翻译为标准化的动作编码序列。后端A还集成了语音识别与合成功能（对接讯飞云服务），使操作人员可以通过语音与系统交互。

中枢大模型与离散数据翻译模型的分工是理解本系统架构的关键。用一个类比来说明：中枢大模型好比一位经验丰富的"飞行指挥长"，他听取飞行员的汇报、查阅天气和场地资料、根据任务目标制定总体飞行方案，并将方案提交给飞行员审核。但他不亲自操作飞机仪表——这份工作交给了离散数据翻译模型，后者好比一台"导航电子设备"，它将指挥长和飞行员确认的行动计划转化为精确的飞行参数（坐标、速度、姿态等）。指挥长会犯错——提出的方案可能考虑不周，但导航电子设备的参数化转换本身就是一道安全网：模糊的计划必须经过量化翻译才能进入飞控。指挥长不需要会用导航设备的具体按钮，导航设备也不需要理解为什么飞行员选择了这个坐标——它们通过一套标准化的指令格式交换信息。更进一步，导航设备在远期还可以换成一个本地运行的轻量计算单元（蒸馏轻量化翻译模型），指挥长甚至不知道设备被换了——只要它继续输出相同的指令格式。这个类比精确反映了龙虾模式的精髓：不同能力层次的组件通过标准化接口协作，职责分明、安全隔离、可独立升级。

**后端B——实时执行桥。** 这是连接智能中枢与无人机飞控系统的桥梁，运行在后端A同一台服务器上但使用独立的Python虚拟环境。其核心职责包括：接收来自翻译模型的动作编码，调用边缘执行模型（先导阶段使用基于规则的计算逻辑，后续阶段切换为神经网络推理）将动作编码转化为具体的飞行目标点（三维坐标、偏航角、速度上限等）；将目标点通过ROS话题发送给模拟无人机或真机飞控；订阅无人机实时回传的位姿、速度、IMU（惯性测量单元）数据，以高频率回传给后端A用于前端展示和历史记录；持续运行监控程序，对飞行数据流执行异常检测。在远期阶段，后端B还将集成路径规划器和雷达感知模块，实现在线避障能力。

可以这样理解后端B的定位：如果后端A是"大脑"（负责理解和决策），后端B就是"脊柱和反射弧"——它忠实地将大脑的意图信号转换为肌肉（无人机）可以执行的运动指令，同时将身体各处的感觉信号（飞行数据）回传给大脑，并在检测到异常危险时通过先天反射（安全悬停兜底）保护整体安全。这种"大脑-脊柱-身体"的生物学类比精确反映了系统三层架构的设计意图。

**模拟无人机。** 在开发过程中没有真实无人机参与，使用ROS软件模拟一个"假无人机"节点。该节点接收位置和速度控制指令后，通过运动学模型模拟真实飞行器的响应，并按照标准格式发布位置、速度、姿态的模拟数据。先导阶段的假无人机仅做最基本的指令响应，不做复杂的物理仿真；后续阶段将接入PX4开源飞控的软件在环仿真，提供更逼真的飞行动力学模拟。

**历史数据库。** 使用轻量级数据库（SQLite，后续可升级为PostgreSQL）存储系统的全部历史记录，包括四张核心数据表：环境条件表（记录每次飞行时的风速、气温、湿度等环境参数）、飞行会话表（记录每次试飞任务的起止时间、执行状态、中枢大模型生成的方案和翻译模型产出的动作序列）、遥测时序表（以10赫兹频率记录的完整飞行轨迹数据，含位置、速度、加速度、角速度、姿态四元数等）、对话记录表（记录操作人员与中枢大模型的所有对话、中枢大模型的工具调用操作及返回结果）。数据库支撑中枢大模型查询历史进行方案制定，也支撑前端的历史回看功能。

选择SQLite作为先导阶段的数据库是经过权衡的：它是单文件嵌入式数据库，不需要独立的数据库服务进程和运维配置，非常适合开发阶段快速迭代和单机部署。在数据结构设计上，飞行轨迹的时序存采用了缓冲批量写入机制——以10赫兹频率产生的位姿数据不逐条写入数据库（频繁写入会降低性能），而是每秒汇总一批数据后统一写入，在保障数据完整性的前提下兼顾了写入性能。当项目进入远期阶段，如果飞行数据存储和查询的并发需求超出了SQLite的能力边界，可以通过修改数据库连接配置迁移到功能更强大的PostgreSQL，而数据库表结构和业务逻辑层代码无需任何改动——这得益于标准化的数据访问层设计。

## 3.3 核心数据流

数据在系统中的流转遵循一条清晰的主链路。下面以一个完整的试飞场景为例，串联各环节的数据往来。

**第一步：任务发起。** 操作人员在前端对话区以打字或语音方式输入一个自然语言任务描述，例如："在2号场地执行一组试飞任务：先起飞到30米悬停5秒，然后向坐标(3, 2, 1)移动，到达后执行180度偏航转弯，最后返航。"这段文字被发送给中枢大模型。

**第二步：方案制定。** 中枢大模型收到任务描述后，调用历史数据查询工具检查2号场地的边界范围、过往无人机在该场地的飞行记录，以及当前的环境条件（如风速较大时可能影响某些机动的安全性）。基于这些信息，中枢大模型制定一份飞行方案，列出拟执行的动作序列和注意事项——例如"检查发现2号场地边界为5×4×3米，目标点(3,2,1)在边界内。当前风速12公里/时，属正常范围。建议动作序列：起飞至30米→悬停5秒→水平移动至(3,2,1)→执行偏航转弯至180度→返航至原点。"该方案作为待审核提案暂存，等待操作人员确认。

**第三步：人工审核。** 飞行方案以卡片形式在前端展示，操作人员查看方案内容，如认可则点击"批准"，如需要修改则与中枢大模型对话调整方案后再次提交。方案被批准后，其意图描述被送入离散数据翻译模型处理。

**第四步：动作翻译。** 离散数据翻译模型接收批准的飞行意图（自然语言），将其翻译为结构化的动作编码序列。例如，上述方案被翻译为：[动作1: 起飞, 目标高度30米; 动作2: 悬停, 持续5秒; 动作3: 水平移动, 目标位置(3,2,1); 动作4: 偏航, 目标角度180度; 动作5: 返航, 目标原点]。每一条动作编码都附带安全约束参数（最大速度、高度上下限等）。

**第五步：目标点生成。** 动作编码序列通过跨进程通信发送给后端的边缘执行模型。边缘执行模型读入当前的环境数据（风速、风向等）和无人机实时状态（当前位置和速度），结合动作编码计算出每一步的具体飞行目标点。例如对于"水平移动至(3,2,1)"，边缘执行模型计算出一条从当前位置到目标点的参考轨迹上的目标点序列，并根据风速情况适当调整目标点的偏移量以补偿风的干扰。

**第六步：轨迹执行。** 在先导阶段（无路径规划器和雷达），边缘执行模型产出的目标点直接作为飞行指令，通过ROS消息发送给模拟无人机。无人机根据目标点调整自身位置和姿态，并将实时位姿数据以10赫兹频率回传给后端A，再转发到前端进行三维轨迹更新。

**第七步：监控与告警。** 数据监控程序持续分析飞行数据流。假如在飞行过程中无人机速度突然超过预设的安全上限，监控程序会立即向中枢大模型发送告警事件。中枢大模型收到告警后，调取最近的飞行数据进行分析，生成处置建议——例如"检测到无人机速度超限，建议立即切换至悬停模式并排查原因"——并通过前端展示给操作人员。处置建议须经操作人员确认后，才会被转化为悬停指令并发送给无人机。

**第八步：飞行结束与分析。** 无人机返航并着陆后，本次飞行会话状态被标记为"已完成"，全部轨迹数据和对话记录存入历史数据库。操作人员可以要求系统对本次飞行数据执行分析——例如"分析起飞后5秒内的垂直速度频谱"——中枢大模型自动调用频谱分析工具完成计算并展示结果。

上述链路完整地展示了龙虾模式的核心工作逻辑：中枢大模型负责"理解、规划、建议"等高层认知任务，离散数据翻译模型和边缘执行模型负责"翻译、计算、执行"等中层转换任务，飞控系统负责"控制、稳定"等底层物理任务。每一步都有明确的责任主体和安全兜底，信息流在三个层次之间双向循环——既有无障碍的"上行"（传感器数据传到操作人员和中枢），也有严密把关的"下行"（人的决策经过翻译和校验后下达给飞控）。

## 3.4 双Python环境与通信链路

由于本系统各组件依赖的技术栈版本存在冲突——人工智能相关的库（如智能体框架、大模型调用接口等）需要Python 3.10或更高版本，而ROS机器人操作系统（Noetic版本）则绑定在Python 3.8且不允许升级——系统采用双虚拟环境隔离策略。

**环境A（Python 3.10+）**：运行后端智能中枢的所有组件——中枢大模型、离散数据翻译模型、Web服务框架、数据库访问、语音处理、数据分析工具集等。该环境依赖最新的AI相关库。

**环境B（Python 3.8）**：运行后端执行桥的所有组件——边缘执行模型（先导阶段）、数据监控程序、ROS节点接口、与模拟无人机的通信等。该环境依赖ROS Noetic系统库。

两个环境不共享任何Python库，相互完全隔离。它们之间的通信通过Unix域套接字（Unix Domain Socket）在同一台Linux机器上进行。Unix域套接字是操作系统内核提供的本地进程间通信机制，不走网络协议栈，延迟极低、吞吐量高，适合后端A与后端B之间的高频数据传输（位姿数据以10赫兹持续上行）。通信内容采用简洁的二进制序列化格式（MessagePack）进行编解码，并辅以长度前缀帧防止"粘包"问题。

这种双环境隔离的设计虽然看似增加了部署的复杂度，但它解决了一个在航空相关软件系统开发中反复出现的难题：飞行控制相关的成熟库（ROS、Mavlink等）与新潮的人工智能库（智能体框架、大模型客户端等）之间的版本兼容性问题。如果试图让两者共处同一环境，往往意味着要么降级AI库从而损失功能和性能，要么冒险升级ROS从而导致飞控组件的行为不确定。双环境方案"宁愿多维护一个环境，也绝不在飞控软件栈上妥协"，体现了航空软件开发中以确定性和安全性优先的设计理念。

前端与后端A之间的通信采用三条不同的通道，各司其职：**HTTP REST接口**用于请求-响应式的数据查询（如查询历史记录、获取场地配置）；**WebSocket长连接**用于实时双向数据推送（如无人机位姿的持续更新、告警事件的即时通知、语音识别的流式结果）；**SSE（服务器推送事件）**用于中枢大模型回复的流式文字输出——中枢大模型逐词生成回复内容，通过SSE推送到前端，使操作人员可以边生成边阅读，无需等待完整回复后才看到结果。

## 3.5 安全设计原则

飞行安全是航空系统设计的首要前提。本项目在架构层面确立了以下安全设计原则，确保在任何异常情况下都不会发生失控飞行。

**人在回路把关关键决策。** 中枢大模型可以提出飞行方案、分析异常数据、给出处置建议，但所有涉及飞行动作执行的关键决策必须经过操作人员审核确认。具体而言：中枢大模型制定的飞行方案以"提案"形式提交，未经操作人员批准不会进入动作翻译环节；数据监控程序检测到异常后生成的处置建议以"提示"形式展示，未经操作人员确认不会转化为飞行指令。这一设计明确将最终决策权保留在人类手中，防止大模型因"幻觉"（生成看起来合理但实际错误的内容）导致危险后果。

**高层智能与飞控层解耦。** 离散数据翻译模型只产出"动作编码"——告知要做什么（如"水平移动到某坐标"），但不决定具体怎么飞（如每个控制周期输出什么电机指令）。具体的飞行控制（位置→速度→姿态→电机输出的四级控制器）由飞控系统（PX4或其模拟替代）负责。这一层次化的职责划分确保了即使上层的智能决策出现偏差，飞控层的物理安全约束仍然有效。

**断连自动安全悬停。** 系统中的每一个通信环节都设计了断连检测和兜底策略。如果后端A与后端B之间的套接字连接断开，后端B在收不到新指令的情况下维持无人机在当前位姿悬停，不执行任何新的飞行动作。如果后端A整体崩溃，后端B同样维持悬停。如果后端B崩溃，无人机的ROS节点可独立保持在最后指令状态。在任何组件失联或故障的场景下，无人机的默认行为都是安全悬停——不会出现"没人控制就乱飞"的情况。

**异常告警不自动执行。** 数据监控程序检测到异常后，仅通过消息总线上报告警事件，触发中枢大模型生成处置建议并展示给操作人员。系统不会在未经人工确认的情况下自动执行任何飞行处置操作。这一原则与《The Aeronautical Journal》论文中提出的"关键动作不可自动执行"原则高度一致——论文中飞行员明确表示燃油切断、发动机关闭等关键开关操作决不接受自动执行，本项目将同一安全哲学延伸到了飞行处置层面。

上述安全原则共同构成了系统的纵深防御体系：操作人员的审核是最高防线，高层与飞控的解耦是结构防线，断连悬停是通信防线，异常不自动执行是操作防线。四道防线相互独立，任意一道失效都不导致危险后果，前一道失效仍有后一道兜底。

# 四、实现流程（设计层面）

## 4.1 一次完整试飞任务的全链路时序

下面以一项具体的试飞任务为示例——"在2号试验场地执行：起飞至30米高度、悬停5秒、水平移动到坐标(3,2,1)、到达后执行180度偏航转弯、最后返航至原点"——详细展示指令在系统中一步步传递和执行的过程。选择这个任务是因为它涵盖了上升、悬停、水平移动、偏航、返航五种基础动作类型，能比较完整地呈现系统的多动作序列执行能力。

**时刻零：操作人员发起任务。** 操作人员在前端对话区输入上述任务描述。描述以纯文字形式通过HTTP接口发送给后端的中枢大模型。同时，操作人员在本阶段可用语音替代打字——按住麦克风按钮说话，语音流通过WebSocket发送到后端，后端调用云服务将语音转为文字后，效果等同于文字输入。

**时刻一：中枢大模型制定飞行方案。** 中枢大模型接收到任务描述后，自动执行一系列信息收集操作：它先调用"场地查询"工具，读取2号场地的配置文件，获得场地边界（5米×4米×3米的全域范围）和预设返航点（原点位置）。它再调用"历史查询"工具，检索该场地过往的飞行记录，了解是否存在已知的风险区域。它还会读取当前的环境条件（风速、气温、湿度等——这些数据由操作人员在前端输入或预设载入）。基于收集到的全部信息，中枢大模型输出一份飞行方案，内容包括：拟执行的动作序列及每步参数、与场地边界和约束的符合性说明、基于环境条件的安全提示。方案以结构化数据形式暂存，同时通过SSE流式推送至前端，以可读的卡片形式展示。

**时刻二：操作人员审核并批准方案。** 操作人员查看前端展示的飞行方案卡片，逐项核对方案内容。假如对某个参数有异议（例如认为悬停时间应改为10秒），操作人员可以直接在对话中回复修改要求，中枢大模型接收反馈后调整方案并重新提交卡片。方案最终被认可后，操作人员点击"批准"按钮，触发一个批准请求发送到后端。后端收到批准后，将方案的意图描述（自然语言摘要）注入离散数据翻译模型的输入队列。

**时刻三：离散数据翻译模型解析意图。** 离散数据翻译模型在一个固定的工作周期内（可配置为1至5秒）从输入队列取出来自操作人员的意图（可能是上述批准的方案意图，也可能是操作人员直接通过键盘快捷键或语音发出的简单指令）。翻译模型将自然语言意图转换为结构化的动作编码序列。以本示例任务为例，输出的动作编码序列为：

- 第一条：动作类型=起飞，目标高度=30米，速度上限=1.5米/秒
- 第二条：动作类型=悬停，持续时间=5秒
- 第三条：动作类型=水平移动，目标位置=(3,2,1)，速度上限=1.5米/秒
- 第四条：动作类型=偏航，目标角度=180度，角速度上限=0.5弧度/秒
- 第五条：动作类型=返航，目标点=原点

每条动作编码还附带一组安全约束参数（高度上限2.5米、高度下限0.3米、最大速度、最大加速度等），这些约束由翻译模型从默认配置文件中读取并与动作编码打包。翻译模型不产生面向人的对话文字——它只是安静地将意图翻译为动作编码，工作状态和产出通过WebSocket消息推送到前端供展示。

**时刻四：边缘执行模型生成目标点。** 动作编码序列通过跨进程的套接字连接发送到后端B的边缘执行模型。边缘执行模型取出序列中的第一条动作编码"起飞至30米"，同时读入当前环境数据（例如当前风速在x轴方向有0.3米/秒的轻微扰动）和无人机当前状态（当前高度0米，垂直速度0），计算出一个目标点：(x=0, y=0, z=30, 偏航角=0)，附带速度上限1.5米/秒。在先导阶段（没有路径规划器和雷达），这个目标点直接作为飞行控制指令，通过ROS消息发布给模拟无人机。在远期阶段，目标点还将经过路径规划器的在线避障处理后才发布给飞控。

**时刻五：无人机执行并回传数据。** 模拟无人机接收到目标点指令后，通过其内部运动学模拟逐步向目标位置移动。无人机以10赫兹的频率持续发布其当前位姿数据（位置、姿态四元数、速度、加速度、角速度），这些数据由后端B订阅后，立即通过套接字回传给后端A。后端A收到位姿数据后，一方面实时更新前端的三维轨迹图（操作人员可以直观看到无人机正在向30米高度上升），另一方面将数据缓冲到遥测缓冲区，准备批量写入历史数据库。

**时刻六：监控程序持续守护。** 在飞行全程中，数据监控程序以独立方式运行在后端B。它持续接收无人机的飞行数据流，逐帧检查各项数值是否在安全范围内：飞行高度是否超过2.5米上限或低于0.3米下限、速度是否超过1.5米/秒上限、数据更新是否连续（若超过一定时间无新数据则报告"数据停滞"）。假如此刻监控程序检测到无人机速度突然飙升至2.0米/秒——超出了翻译模型设定的1.5米/秒安全上限——它立即通过消息总线发送一个告警事件，内容包含告警级别（"警告"）、异常代码（"速度超限"）、具体数值（当前速度2.0米/秒 vs 上限1.5米/秒）。

**时刻七：告警唤醒中枢大模型。** 告警事件通过消息总线到达后端A的告警触发器模块。触发器做两件事：第一，将告警消息转发到前端，在对话区以醒目的系统消息形式展示，提示操作人员注意异常；第二，将告警的详细信息注入中枢大模型的分析上下文中，唤醒中枢大模型对异常进行分析。中枢大模型被唤醒后，调用数据分析工具查询最近数秒的飞行数据（用于还原异常前后发生了什么），分析可能的异常原因，并生成处置建议——例如："检测到无人机垂直速度异常上升至2.0米/秒（安全上限1.5米/秒），可能原因：环境风速突变导致上升阻力减小。建议：立即切换至悬停模式，稳定后再继续任务。"处置建议同样展示在前端的对话区，但**不自动执行**。

**时刻八：操作人员决定处置。** 操作人员在对话区看到告警和处置建议后，根据自身判断决定：接受建议并批准执行悬停指令；或者忽略建议（如果判定为虚警，如监控程序阈值过于保守）；或者做出不同于建议的处置动作。如果操作人员批准了悬停建议，悬停指令将通过离散数据翻译模型生成对应的动作编码，再经边缘执行模型下发到无人机。假如监控程序的告警被确认是虚警，操作人员可以恢复到中断前的任务断点继续执行。

**时刻九：任务完成与数据归档。** 所有五条动作编码依次执行完毕，无人机返航降落在原点。本次飞行会话的状态被更新为"已完成"。飞行期间全部的轨迹时序数据（数千条采样点）从缓冲区批量写入历史数据库的遥测表中。本次飞行的完整对话记录——包括操作人员的任务描述、中枢大模型的方案和建议、操作人员的审核确认、告警事件和处理结果——全部写入对话记录表。操作人员可以在前端历史页面回看本次飞行的全过程，包括轨迹的三维回放、对话的完整记录、告警的时间标记。

## 4.2 关键环节指令流转与数据结构

为了让系统内各组件能够准确理解彼此的意图，本项目设计了一套标准化的指令表达方式。这一表达方式的核心是"动作编码"——一种结构化的飞行指令格式，用固定的编码和数值组合来描述飞行器应当执行的动作，替代模糊的自然语言描述。

动作编码的基本结构包含三个要素：**动作类型**（用一个标识符表示动作的种类——起飞、悬停、水平移动、垂直移动、偏航转弯、返航、紧急悬停等）、**数值参数**（给出动作的执行目标和限制——目标高度多少米、持续多少秒、移动到哪个坐标、转多少度等）、**安全约束**（附带本次动作的各项安全上限——最大速度、最大加速度、高度上下限等）。

以"将无人机水平移动到场地内的(3, 2, 1)坐标点"为例，这条指令的动作编码形式为：动作类型="水平移动"，目标位置=(3, 2, 1)，速度上限=1.5米/秒，加速度上限=2.0米/秒²，高度上限=2.5米，高度下限=0.3米。离散数据翻译模型将操作人员"飞到(3,2,1)看看"的口语化表达统一翻译成这种结构化的动作编码；边缘执行模型收到后，将其中的数值（目标坐标、速度上限等）与环境数据和无人机状态合并计算，生成具体的飞行目标点。

这种设计的精妙之处在于**高层意图与底层执行的解耦**：翻译模型只关心"用什么动作、到哪里去"，不需要知道无人机当前在什么位置、风有多大；边缘执行模型只关心"从当前状态出发怎样达到目标点"，不需要理解为什么要去那个点。二者通过动作编码这个统一格式实现对接，各自独立工作、独立升级。

消息总线上的通信采用统一的消息格式：每条消息包含发送方名称、接收方名称、消息类型（请求/响应/事件/错误）、一个关联本次对话的唯一编号、调用的工具或事件名称、参数或数据内容、以及时间戳。无论消息在同一个进程内部传递（内存直传）还是跨进程传递（套接字+二进制序列化），消息的格式结构保持不变——各组件只需与消息总线交互，无需感知传输细节。

为更直观地理解动作编码和消息总线的搭配方式，以下展示一条完整的指令传递实例。假使离散数据翻译模型产出了这样一条动作编码："动作类型=水平移动，目标位置=(3, 2, 1)，速度上限=1.5米/秒"。翻译模型不直接将这条编码发给边缘执行模型，而是将编码包装为一条标准消息：消息发送方填"离散数据翻译模型"，接收方填"边缘执行模型"，消息类型填"请求调用"，工具名称填"生成目标点"，参数栏填动作编码的具体内容。消息总线收到后查注册表，发现"边缘执行模型"在B进程中，于是通过套接字将消息序列化为二进制数据块发送过去。B进程的消息总线解包后，根据接收方名称路由到边缘执行模型的代码入口，传入参数并触发计算。边缘执行模型返回的结果（目标点的三维坐标和速度值）同样包装为消息，沿反方向经由消息总线和套接字传回翻译模型所在的A进程。整个过程，翻译模型写代码时不需要写任何网络通信逻辑，边缘执行模型也不需要——它们只需要"接收消息，处理，回复消息"即可，通信细节由消息总线完全代理。

## 4.3 实时控制与人机协同

在实际飞行中，操作人员需要随时介入控制，不能完全依靠预定的动作序列。本系统为此设计了三个层面的实时控制能力。

**键盘快捷键接管。** 操作人员可以通过键盘按键直接控制无人机的即时动作：按下对应按键使无人机向指定方向移动或执行上升下降等动作。这些键盘输入被封装为动作指令，直接送入离散数据翻译模型的输入队列，在下一个工作周期被处理。此外，前端界面右下角的悬浮球提供可自定义的快捷键面板，操作人员可以为常用动作绑定自定义快捷键。

**语音指令接管。** 操作人员在飞行中可以通过语音发出控制指令，无需腾出手来操作键盘。语音经过云端识别转为文字后，与键盘指令同等处理——送入翻译模型的输入队列，在下一周期产出对应的动作编码。需要强调的是，语音和键盘输入都不会打断正在执行的那一步动作（例如无人机正在上升过程中收到一条下降指令，系统会等上升动作执行完毕再处理下降指令），但操作人员可以通过专门的"紧急中止"命令，立即终止当前动作序列并将无人机切换至安全悬停模式。

**紧急中止。** 前端界面底部始终显示一个醒目的"紧急中止"按钮。该按钮被点击后，后端不通过翻译模型（绕开LLM环节以保证响应速度），直接发出中止指令清空当前待执行的动作序列，并命令无人机在当前位姿安全悬停。该路径的处理延迟预期在100毫秒以内（同机进程间通信），保障紧急情况下的快速响应。

离散数据翻译模型的工作周期（循环周期）是本系统处理实时性的关键参数。该周期可以在1至5秒之间配置：周期越短，系统的响应越及时（键盘和语音指令被更快地翻译为动作编码）；周期越长，翻译模型可以利用更多的时间进行更"深思熟虑"的动作规划。在先导阶段，周期默认为2秒，后续可根据实际测试效果调节——最终目标是在边缘执行模型（以50至200毫秒的延迟生成目标点）和蒸馏轻量化翻译模型（将翻译延迟从1至5秒降至0.5秒以内）等技术就位后，将系统整体响应延迟降低到亚秒级别。

值得一提的是，离散数据翻译模型的循环周期设计还考虑了"仲裁逻辑"——如果在同一周期内既有操作人员直接发出的键盘/语音指令，又有中枢大模型经过审核的方案在执行中途产生了下一个动作，系统需要在"人的直接意图"和"既定计划"之间做出选择。本系统采用的仲裁规则是：操作人员的直接指令（包括键盘快捷键和语音指令）具有最高优先级，因为它代表操作人员的即时判断；其次是紧急中止指令（绕过翻译模型立即生效）；再次是中断后的既定计划恢复（从中断点继续执行剩余动作序列）。这种优先级层级确保了人在回路不是空洞的口号而是有技术机制保障的实质性原则。

## 4.4 失败与断连处理

系统在设计中考虑了多种失败场景的应对策略，核心原则是：**任何环节出现通信中断或处理失败，无人机都应安全悬停，绝不进入失控状态。**

在智能中枢与执行桥之间的连接中断时：执行桥在指定时间内未收到中枢的任何消息，即判定与中枢失去联系。此时执行桥不再等待新的动作指令，维护无人机在当前位姿的安全悬停。与此同时，中枢检测到心跳超时后，通过前端界面显示链路告警。当连接恢复后，操作人员可以重新发起控制指令。

在智能中枢整体崩溃（进程异常退出）时：执行桥因接收不到新的意图指令而维持无人机悬停。无人机的ROS节点作为独立进程不受中枢影响。中枢重启后自动重新建立连接，读取最近的位姿数据恢复状态。

在执行桥崩溃时：中枢检测到套接字连接断开，通过前端显示告警。无人机ROS节点独立运行，保持最后的位姿。执行桥重启后重新连接中枢并同步状态。

在大模型服务（云端）不可用时：当前轮次的翻译或生成操作失败，中枢不下发任何新的飞行指令——无人机维持执行上一动作或安全悬停。系统在重试失败后通过前端告警提示操作人员当前大模型服务不可用状态。

在边缘执行模型无法生成可达目标点时（例如动作编码命中了未知动作类型或计算出的目标点超出场地边界和高度约束）：执行桥通过消息总线回告中枢"无法生成目标点"，并附带原因说明。中枢收到后通知离散数据翻译模型重新考虑该动作的翻译方式（例如调整速度参数或改用替代动作），或者兜底生成一条"悬停"指令以保安全。

在监控程序本身故障时：执行桥记录日志并通告中枢。飞行功能不受影响——但系统进入"降级"模式（无实时异常监控），操作人员被提醒当前监控功能不可用，需要更加关注飞行状态。

上述所有失败场景的应对策略均经过架构层面的充分推演，确保不存在单点失败导致危险后果的情况。这一纵深防御设计也是本项目区别于传统自动化系统的重要特征——传统方案中如果自动控制模块失效，人可能来不及接管；而本系统的安全悬停机制在亚秒级别自动介入，为人争取到充分的响应时间。

## 5.1 前端交互界面

前端是操作人员与系统交互的唯一窗口，设计目标是在一个界面上同时满足"全局态势感知""对话沟通""精准操控"三个层次的操作需求。界面同时支持电脑网页和手机浏览器两种使用形态，电脑版采用左右分栏布局，手机版采用上下分栏并通过滑动切换视图。

**右侧主力视图区**是操作人员的"眼睛"。这里默认显示三维飞行轨迹图——基于计算机三维图形技术实时渲染一个三维场景，场景中呈现试验场地的边界框、地面参考网格、无人机的三维模型（以四旋翼飞行器的几何形态表示），以及无人机的飞行轨迹（已飞过的路径用亮色实线表示，未来的规划路径用虚线表示）。操作人员可以用鼠标拖拽旋转视角、滚动滚轮缩放画面、右键平移场景，从任意角度观察无人机与场地的空间关系。在远期阶段，当无人机搭载相机后，视图区可以切换为实时视频回传画面。此外还可以显示高度变化曲线等二维数据图表。

**左侧控制面板**展示状态信息和数据。环境状态区显示当前的风速（包含三个轴向的分量）、气温、湿度等环境参数——这些数据在开发阶段由操作人员手动输入，远期将接入真实环境传感器。任务进度区显示当前正在执行的动作类型（如"水平移动"）、动作序号（如"第3/5步"）、预计剩余时间和全局飞行状态（空闲/悬停/执行中/已完成/已中止）。

**对话区**是操作人员与中枢大模型沟通的主要渠道。操作人员在此以打字或语音（点击麦克风按钮）的方式与中枢大模型交互，包括：下达新任务、询问场地和历史信息、请求数据分析、调整飞行方案、响应告警和处置建议。中枢大模型的回复以流式文字方式呈现——每个字生成后立即推送到界面上，操作人员可以边看边理解，无需等待长回复完全生成。对话区还展示系统消息（如告警通知、动作执行状态更新），并以特殊样式与普通对话区分。

**悬浮球与快捷键**提供高效的精准操控通道。界面右下角常驻一个悬浮操作球，点击后展开扇形快捷菜单，提供上升/下降、向各方向移动、悬停、紧急中止等常用操作。操作人员可以为每个操作自定义键盘快捷键，在飞行中快速触发。

**数据展示看板**是中枢大模型智能驱动前端的一种能力。看板上可以同时展示多个数据面板，每个面板显示一类飞行数据（如速度-时间曲线、姿态变化图、高度剖面图等）。与传统固定布局不同，本项目设计为中枢大模型可以根据当前任务场景，动态决定展示哪些面板和采用什么筛选条件。例如在执行偏航转弯测试时，中枢大模型可能自动配置看板重点展示航向角变化和角速度；在执行高度变化测试时则重点展示高度-时间曲线和垂直速度。这一设计使前端不再是一个被动的数据显示器，而是一个由中枢智能驱动的动态信息呈现系统。

## 5.2 后端A——智能中枢

智能中枢是整个系统的"大脑"，运行在Linux服务器上的Python 3.10+虚拟环境中，承载中枢大模型和离散数据翻译模型两个核心智能体，以及Web服务、语音处理、数据库管理等支持功能。

**中枢大模型**是整个龙虾模式中的"中枢"，扮演着规划师和总调度人的角色。它基于云端的大型语言模型构建，具备强大的自然语言理解和生成能力。它的核心职责有三项：第一，**任务理解与规划**——接收操作人员的任务描述（可能是模糊的、口语化的），通过查询场地信息、历史数据和当前环境条件，将模糊的任务意图转化为清晰可执行的飞行方案。第二，**信息查询与数据分析**——能够响应操作人员的各种查询请求，从"列出最近五次飞行中发生过速度超限的记录"到"对这段轨迹数据做频谱分析"这样的专业数据处理请求。第三，**异常响应与处置建议**——在收到监控程序的告警事件后，主动分析异常数据并生成处置建议，但不自动执行。

中枢大模型通过"工具调用"机制与系统内的其他组件交互——当它需要查询场地配置、读取历史数据、执行数据分析、或提交飞行方案时，它并不是把这些操作硬编码在自身的回复逻辑中，而是生成一个标准化的"工具调用请求"，经由消息总线发送到相应的功能组件，组件完成操作后将结果回传。这种松耦合的设计使中枢大模型的能力扩展变得简单——添加一个新工具只需要在组件注册表中登记，中枢大模型即可发现并调用它，无需修改大模型本身。

**离散数据翻译模型**是一个专门的"动作翻译器"。与中枢大模型的"对话+规划+调度"多重角色不同，翻译模型的职责单一而明确：接收自然语言描述的飞行意图（来自操作人员直接的指令或经审核通过的中枢大模型方案），输出标准化的动作编码序列。它不与人对话——不产生任何面向人的文字回复，操作人员看不到它的"思考过程"，只能看到它产出的动作编码摘要通过系统消息推送展示。

两个模型的分工设计非常关键。中枢大模型负责面向人的高层次工作——理解和规划；翻译模型负责面向机器的翻译工作——把意图变成编码。这样的分工有两层深意：其一，**安全边界**——中枢大模型可能会产生"幻觉"（生成不准确的规划），但它的产出必须经过翻译模型的结构化转换才会进入飞行执行链路，而翻译模型的输出又被边缘执行模型和飞控层进一步约束和校验，形成多层安全网。其二，**可替代性**——中枢大模型和翻译模型可以各自独立升级或替换而不影响对方。在远期规划中，翻译模型将由基于大模型的实现替换为经过专门训练的轻量化模型（行为克隆），实现在机载电脑上的本地运行，消除对云端的依赖——这一替换不影响中枢大模型的任何功能。

## 5.3 边缘执行模型

边缘执行模型是连接"高层意图"与"飞行控制"的关键桥梁。它的工作方式可以通俗地理解为：**"翻译好的动作指令（做什么），结合当前的环境条件和飞机状态（现在是什么情况），计算出飞行控制系统能直接理解的目标点（怎么到达）。"**

具体来说，边缘执行模型的输入是三个方面信息的组合：其一，来自离散数据翻译模型的动作编码和数值——例如"动作类型：水平移动，目标位置：(3,2,1)，速度上限：1.5米/秒"；其二，来自环境传感器（或在此开发阶段由操作人员手动输入）的离散环境数据——风速的三轴分量、气温、湿度等；其三，来自无人机飞控系统实时回传的飞行状态数据——当前三维位置、速度向量、姿态角、电池剩余电量等。模型的输出是一个或多个飞行目标点，包含目标位置的三维坐标(x,y,z)、目标偏航角、以及建议的飞行速度上限。这些目标点可以直接作为飞行控制系统（如PX4飞控）位置控制器的输入。

为什么需要在翻译模型和飞控之间插入这样一个模型？翻译模型已经能产出"水平移动到(3,2,1)"这样的指令，为什么不能直接把目标坐标发给飞控？原因是：**同样的动作意图，在不同环境和状态下需要有不同的执行方式**。

举例来说，"水平移动至正前方3米"这一动作，在静风条件下和逆风条件下应该产生不同的目标点序列——逆风时需要更大的推进力补偿，而这在目标点层面体现为沿前进方向适度延伸目标位置。又如从不同起始位置执行"上升至30米"——当无人机已经位于25米高度时所需的上升量（5米）与从地面起飞时所需（30米）完全不同。再如电池电量较低时，无人机动力系统输出能力下降，同样的"以1.5米/秒速度上升"可能需要边缘执行模型适当下调目标速度以确保不超出当前可用推力。这些"因地制宜"的计算不适合由翻译模型来做——翻译模型的专长是理解人的意图并生成动作编码，而不是进行飞行控制层的状态感知和数值计算。如果硬要让翻译模型承担这些计算，要么需要给它输入大量飞控参数（超出其设计职责），要么它在信息不足的情况下做出不准确的计算（带来安全风险）。边缘执行模型正是为这个计算环节而专门设计的：它将动作编码、环境和状态三者综合起来，计算出对当前情况"最优"的目标点。这种逐层分工确保了每个模型都在其信息充分和计算能力匹配的抽象层级上工作。

在开发策略上，边缘执行模型采用"先简后繁"的渐进式路径。**先导阶段**使用基于规则的简单计算方式：动作编码到目标点的映射通过预先定义的规则完成——例如"水平移动"的动作编码对应规则"目标位置=动作编码中的目标坐标（如(3,2,1)），目标点位于从当前位置到目标坐标的直线上"。同时，规则还包含安全夹紧逻辑——如果计算出的目标点超出了场地边界范围或高度限制，则将其修正（夹紧）到合法范围内。这种规则方式简单可靠，足以验证系统链路的连通性和基本功能。**远期阶段**将替换为一个经过专门训练的轻量神经网络（数兆字节大小），输入离散环境数据、飞行状态参数和动作编码，直接推理出优化的目标点。这个神经网络将使用试飞过程中积累的真实数据进行训练——每次飞行中翻译模型产出的动作编码、当时的真实环境和状态、以及实际执行成功的目标点，共同构成神经网络的训练样本。经过充分训练后，神经网络能够比规则更准确地捕捉环境与状态对飞行效果的微妙影响，做出更精细的目标点修正。

边缘执行模型的部署位置随着阶段推进而迁移。先导阶段它运行在服务器上（与智能中枢同机但不同Python环境），通过网络或本地套接字接收动作编码。远期阶段它将迁移到无人机的机载电脑上，以独立进程运行。机载部署的意义在于：即使无人机与地面控制站之间的通信链路中断，机载的边缘执行模型也可以根据最后一次收到的动作编码和本地实时的环境/状态数据，继续生成目标点并维持安全飞行——这是实现"断网可飞"的核心技术保障。

## 5.4 路径规划器与雷达感知（远期阶段）

在远期阶段（第二阶段及以后），系统将集成路径规划器和雷达感知模块，实现在线自主避障能力。这一设计替代了传统方案中需要人工预先将场地障碍物信息编码进系统的做法。

传统方案的工作方式假设所有障碍物都在飞行前已知：操作人员在飞行前将场地的障碍物位置、形状、尺寸等信息逐一录入配置文件，飞行控制系统根据这些静态信息计算避障路径。这种方式的缺陷显而易见——如果场地布局发生变化（例如临时放置了新的设备或停放了其他无人机），预先录入的信息就失效了，无人机会径直撞向"未知"的障碍物。而且即使场地固定不变，人工测量和录入每个障碍物的精确位置本身就是一项繁琐且容易出错的工作。

雷达感知模块从根本上解决了这个问题。无人机搭载三维激光雷达或深度相机等感知设备，在飞行过程中持续扫描周围环境，实时采集三维点云数据——即大量空间点的三维坐标集合，这些点云勾勒出周围物体（墙壁、柱子、其他设备等）的表面轮廓。点云数据以不经过人工预处理的方式直接送入路径规划器。

路径规划器接收到两路输入：一路来自边缘执行模型的目标点（"要去哪里"），一路来自雷达的实时点云（"周围有什么障碍物"）。路径规划器是解决"从A点到B点怎么走才不会撞到东西"这一经典问题的计算模块。它在三维空间中建立环境地图（来自雷达点云），找到一条从当前位置到目标点的连续、无碰撞的路径。与地面导航软件的二维路径规划不同，无人机的路径规划在三维空间中进行——不仅要避开前方和侧面的障碍物，还要考虑上方和下方——而且必须满足飞行器的机动能力限制（无人机不能瞬间急转弯、加速度有上限）。路径规划器实时在线运行：不是在飞行前一次性算好就不再改变，而是在飞行全程中根据最新点云数据不断重新计算和调整路径。因为点云实时更新，即使飞行中出现了未预料到的障碍物（例如另一台无人机临时进入同一空域），路径规划器也能够即时检测并重新规划绕行路径。这种在线避障能力是本系统"不预先编码场景"设计理念的核心支撑——适应未知和动态环境的能力，是系统从受控仿真环境走向真实外场飞行的关键一步。

在先导阶段（第一阶段），路径规划器和雷达感知模块尚未部署，系统的物理安全保障依靠三层防线：边缘执行模型在生成目标点时的安全夹紧（确保目标点在场地边界和高度限制内）、模拟无人机的边界自保护逻辑（超出边界时自动停止）、以及操作人员的实时监控与介入能力。从第二阶段开始，雷达感知和路径规划器陆续加入，逐步建立起完整的在线避障能力。

## 5.5 数据异常监控程序

数据异常监控程序是飞行安全的"哨兵"角色。它运行在飞行控制侧（后端B），以独立方式持续从无人机飞控系统读取实时数据流，不依赖于中枢大模型或操作人员的指令。一旦检测到数据中的异常特征，它立即发出告警，触发中枢大模型和操作人员的关注。

监控程序内置了两种检测逻辑，分别针对不同类型的异常。

**阈值检测**是最直接的异常判定方式：为各项飞行数据预设"安全范围"，当实际读到的数值超出安全范围时触发告警。预设的监控项目包括：飞行速度上限（例如1.5米/秒）和下限（速度异常归零可能表示飞控失效）、飞行高度上限（场地天花板，例如3米）和下限（最低安全高度，例如0.3米）、加速度上限（限制急加速机动）、角速度上限（限制急转）。阈值检测的特点是简单可靠、响应迅速（一旦超限立刻告警，无需积累历史数据）——但也因过于"死板"而容易产生虚警，因此阈值参数被设计为可配置的，操作人员可以根据试验需求调整安全范围。

**趋势检测**用于发现阈值的"瞬间检查"无法捕捉的异常模式。趋势检测不是看单一时刻的数据点是否超限，而是分析数据在一段时间内的变化趋势。例如：速度突变检测——如果在短时间内（例如0.5秒）速度的变化量超过了正常变化范围，这可能表示飞控系统出现了控制过冲或传感器噪声；姿态抖动检测——如果姿态角在短时间内出现了频繁的高频震荡，这可能表示飞控的PID参数调节不当或机械结构出现松脱；持续性偏离检测——如果某个参数虽然没有超出阈值，但在一段持续时间内表现出单调递增或递减的趋势（例如高度在缓慢但持续地下沉），这种"温水煮青蛙"式的漂移可能比瞬间超限更需要关注；数据停滞检测——如果超过一定时间（例如2秒）未接收到新的飞行数据，这可能表示传感器或通信链路故障。

两种检测逻辑互补：阈值检测处理"已经发生"的异常，保障基本安全——其优势是判定直接、响应迅速；劣势是只能发现"越界"问题，对于靠近边界但尚未越界的风险以及"趋势性恶化"无能为力。趋势检测处理"正在酝酿"的异常，提供预警窗口——其优势是能够在风险升级为实际越界之前发出警告，为操作人员争取到更充裕的响应时间；劣势是可能存在一定的虚警率（将正常的短期波动误判为异常趋势）。两者结合，形成一个从"预警"到"告警"的完整安全监控梯队。

当任一检测逻辑触发告警时，监控程序通过消息总线上报告警事件。告警事件包含告警级别（警告/严重/紧急）、异常代码（标识异常类型）、具体数值（触发告警的数据值和相关阈值）等信息。告警事件到达后端A后，经由告警触发器模块处理：一是推送到操作人员的前端界面——以醒目的系统消息形式展示在对话区；二是注入中枢大模型的分析上下文，由中枢大模型运行数据分析工具还原异常前后的数据脉络，生成处置建议。

**告警绝不自动触发飞行处置。** 监控程序只负责"发现问题并喊话"，接下来的"判断问题并决定动作"由中枢大模型和操作人员共同完成。中枢大模型给出分析和建议，最终由操作人员决定是否采纳。这一设计不仅保障了人在安全决策中的最高权威，也避免了自动响应逻辑可能因为"误报"而导致的非必要飞行中断。

## 5.6 指令驱动的数据处理

试飞过程中会产生大量的时间序列数据——无人机在飞行中每个采样的位置、速度、加速度、姿态、角速度等信息。这些原始数据本身是数字的堆砌，必须经过专业的处理方法（如频谱分析、统计分析、信号滤波等）才能从中提取有价值的信息。然而，操作这些专业数据处理工具通常需要一定的技术背景——需要了解每种工具的调用方式、参数格式和输出含义。

本项目通过中枢大模型的工具调用能力，将专业数据处理的复杂性封装在幕后，让操作人员可以用自然语言来驱动数据处理。操作人员不需要知道"傅里叶变换有哪些参数""统计显著性的计算公式是什么"，只需要用日常语言描述"我想分析什么"——例如："帮我看一下这段飞行数据中，垂直方向有没有异常的高频振动？"

当操作人员发出这样的请求后，中枢大模型做以下操作：理解请求的含义（"垂直方向的高频振动"本质是垂直速度数据的频谱分析），确定应该调用傅里叶变换（FFT）工具，从请求中推测出工具所需的参数（选择垂直速度通道作为输入数据、从历史查询中获取对应时间段的数据），通过消息总线发起工具调用。FFT工具组件接收请求后执行计算，将结果（各频率分量的幅度值）通过消息总线返回给中枢大模型。中枢大模型再将数值结果翻译为人可以理解的表述——"在本次飞行的前10秒，垂直速度在约2.5赫兹处有一个明显的峰值，幅度约为0.15米/秒，这表示可能存在一个周期约0.4秒的轻微垂直振荡，建议检查飞控的垂直通道增益参数。"

系统预置了三种基础的数据分析工具，它们都作为消息总线上的可注册组件实现，未来可以根据需求添加新的分析工具。**傅里叶变换（FFT）工具**将时间域的信号（一段随时间变化的数值序列）转换为频率域（各个频率分量的强弱），用于发现数据中的周期性特征——振荡、谐振、抖动等。**统计分析工具**计算一组数据的各项统计指标——均值、标准差、最大值、最小值、超过阈值的次数或比例——用于对飞行表现做定量评价。**滤波工具**对数据进行平滑降噪处理，剔除传感器的高频噪声以突出真实信号趋势——在分析飞行数据时，原始数据通常会叠加测量噪声，需要先滤波再做进一步分析。

这种"自然语言驱动数据处理"的方式，不仅降低了操作数据分析工具的技术门槛，更重要的是将"提问"从"试飞后"提前到了"飞行中"。传统流程中，操作人员在飞行结束后将数据导出到专门的分析软件中处理，往往在飞行完成好几个小时后才能看到分析结果。而在本系统中，操作人员可以在飞行过程中随时对已经飞出的数据段发出分析请求，在飞行还在进行中就获得分析结果——这就好比病人不是等到出院后才看到检查报告，而是在检查的同时医生就给出了诊断。这种实时的数据洞察能力对于试飞效率的提升是质的飞跃。

## 5.7 历史数据库

历史数据库是系统的记忆和知识库。先导阶段使用轻量级的SQLite数据库（一个独立的文件、无需专门运维、适合单机开发场景），远期可根据数据增长评估是否需要迁移到更强大的PostgreSQL数据库。数据库的设计不会因为日后更换数据库系统而需要大量修改——使用标准的数据访问层隔离了具体的数据库产品差异。

数据库包含四张核心数据表，分别记录不同类型的信息。

**环境条件表**存储每次飞行时的环境参数。由于开发阶段没有真实传感器，环境数据由操作人员通过前端界面手动输入或从预设模板中选择（例如"晴、微风"模板自动填充风速0.2米/秒、湿度45%等值）。每天的环境条件以一条记录的形式存储，多条试飞任务可以共享同一条环境记录。环境数据采用灵活的格式存储，便于远期添加新的环境参数类型（如气溶胶浓度、电磁干扰强度等）而不需要修改表结构。

**飞行会话表**是每次飞行任务的"索引"。每条记录对应一次完整的飞行任务，包含任务唯一编号（以发起时间的年月日时分秒作为标识）、开始和结束时间、关联的环境条件、操作人员的任务描述文字、中枢大模型生成的飞行方案（文本形式保存，供事后回溯分析）、以及离散数据翻译模型产出的完整动作编码序列。会话记录还跟踪任务状态——从"空闲"到"方案已制定""执行中""已完成""已中止"的完整生命周期流转。

**遥测时序表**存储飞行轨迹的"每一帧"。无人机在飞行全程中以10赫兹频率（每秒10次）持续回传位姿数据，每次采样包括：时间点、三维位置(x,y,z)、三维速度分量、三维加速度分量、三维角速度分量、以及姿态四元数（一种用于三维旋转数学表达的四个数值）。以一次5分钟的飞行计算，遥测表将存储约3000条采样记录。这些高频数据不仅是前端实时三维轨迹渲染和二维图表展示的数据来源，也是飞行后数据分析的主要素材——上述的FFT频谱分析、统计分析等，都是基于遥测表中的海量时序数据执行的。

**对话记录表**存储操作人员与系统的所有交互。每条记录记下发言人（操作人员/中枢大模型/工具调用）、发言内容、以及附带的元信息（如果是工具调用则记录工具名称和参数，如果是翻译模型则记录产出的动作编码类型）。对话记录区分两类主体：中枢大模型的对话和离散数据翻译模型的操作记录。对话记录表支撑两项重要功能——在中枢大模型制定飞行方案时，它可以查阅对应历史对话作为上下文参考（例如"上次在这个场地飞行时操作人员特别强调不要超过2米高度"）；操作人员在前端历史页面回看飞行过程时，可以根据对话记录还原当时的决策逻辑。

## 5.8 语音交互

语音交互是提升操作效率的重要手段——在飞行操控过程中，操作人员的手可能需要操作键盘快捷键，或者纯粹出于便利不愿打字，语音成为一种自然高效的交互方式。本系统集成语音识别（将语音转为文字）和语音合成（将文字转为语音）两项能力。

**语音识别**的工作流程为：操作人员在前端按住麦克风按钮开始说话，前端浏览器采集麦克风的原始音频信号，按40毫秒为一帧的切分频率，通过WebSocket持续发送音频数据到后端。后端收到首帧后，建立与讯飞云服务的连接（使用签名鉴权），并将后续收到的音频帧逐帧转发至讯飞。讯飞以流式方式返回识别结果——随着语音的推进，识别结果可能经过多次修正（系统会根据后文语境自动修正前面的识别文字）。中间结果以"非最终"形式回填前端的输入框（动态消隐），最终确认的识别文字以"最终"形式确定下来。识别的文字自动填入前端输入框，操作人员检查无误后点击发送，文字被送入中枢大模型处理。

**语音合成**的工作流程为：中枢大模型生成回复文字后（或操作人员手动选择某段文字要求朗读后），后端将目标文字分段发送给讯飞云服务（长文本分段以保持流式播放的节奏），讯飞返回合成音频，后端逐段下发到前端。前端收到音频数据后自动解码并播放。操作人员可以在设置中关闭自动朗读，仅保留手动选择朗读功能。

驾驶舱（及其模拟环境）的音频通道高度拥挤——论文研究已经明确指出，不相干的语音播报会干扰操作人员听取关键的无线电通信和告警提示音。因此本系统的语音合成设计遵循了论文的建议原则：默认不自动朗读中枢大模型的所有回复，仅在告警等关键场景使用音频提示（例如以简短提示音宣告系统关注要点），将音频用作辅助通道而非主通道。

语音交互的引入解决了飞行操控中的一个实际痛点：操作人员的双手可能被键盘快捷键、飞行摇杆等其他操控设备占用，眼睛需要关注三维视图和状态面板——此时语音成为一种"不抢手、不占眼"的第三交互通道。同时，语音通道也增加了系统设计的复杂性：语音识别在嘈杂环境中准确率可能下降，语音合成在已有告警音和无线电通信的环境中可能造成信息干扰。系统的设计策略是保持语音通道为"可选增强"而非"强制依赖"——所有语音操作都有键盘或触屏等替代输入方式，确保语音通道故障不影响基本操控能力。

## 5.9 开放式接口规范

开放式接口规范是本项目的核心研究产出之一——一份指导后续新组件快速接入系统的标准文档。它定义了任何新组件接入系统所需遵循的统一格式和协议。

**统一消息总线**是整个开放式架构的通信骨架。系统内的所有组件——不管它们运行在同一个进程内、同一台机器的不同进程间、还是不同机器上——都通过发送和接收标准格式的消息与彼此通信。消息格式的关键字段包括：发送方和接收方的组件名称（消息总线路由表据此分发）、消息类型（是请求调用、响应结果、还是事件通知）、一个唯一编号（用于请求和响应配对）、本次调用的工具或事件名称、附带的参数或数据，以及时间戳。

组件不感知通信的传输层差异——调用者只需发起"发给组件X，调用工具Y，参数为Z"，消息总线自动判断组件X在哪里（本地内存 vs 跨进程套接字 vs 远期可能的机载-云端链路），选择合适的传输方式并完成序列化和路由。这意味着组件开发者完全不需要关心通信基础设施的细节。

**组件注册表**是系统的"黄页"——列出当前系统中所有可用组件，以及每个组件接受哪些工具的调用。组件在系统启动时向注册表登记，注册的内容包括：组件名称（唯一标识）、接受的工具列表（如"FFT分析""统计分析"等）、组件所在位置（在哪个进程中）。中枢大模型在需要调度组件时，查询注册表找到目标组件的位置，然后通过消息总线发起调用。

这种注册机制还为系统治理提供了方便。管理员可以清晰地从注册表中看到当前系统有哪些组件在运行、哪些工具可用——这相当于系统的"运行清单"。如果某个组件在启动时注册失败或意外崩溃后注册表记录了该组件"不可达"的状态，中枢大模型在调度时会收到"目标组件不可用"的明确反馈，而不是面临一个无法解释的"调用失败"——提高了系统的可诊断性和可维护性。

这种设计使得系统具备天然的可扩展性。一个新增的数据处理工具——例如"小波变换分析"——只需要编写该工具的计算逻辑，实现标准的消息接收和回复接口，然后在注册表中注册"组件名：wavelet_analyzer，接受的工具：wavelet_transform"。完成这三步之后，中枢大模型就可以在操作人员发出"用小波变换帮我分析这段数据"的请求时，自动发现并调度这个新工具。整个过程中不需要修改中枢大模型的代码、不需要修改消息总线的代码、不需要修改任何已有组件的代码。

开放式接口规范的价值还可以从另一个角度理解。在传统软件系统开发中，添加一个新功能通常意味着"找到相关代码→修改代码→测试修改→部署"这一循环。如果新功能与已有功能有耦合（例如新的数据分析工具需要修改中枢大模型的调用逻辑），这个循环会变得冗长且容易引入回归性错误——改了A导致B出问题。而在本系统的开放式架构中，添加新功能是一个"开发独立的组件→注册到总线"的过程，新组件与已有组件之间不存在代码层面的耦合（它们只通过标准消息交互），因此不会引入回归风险。这种封装与解耦是大型系统长期可维护性的核心保障。

# 六、关键技术介绍

## 6.1 龙虾模式智能体架构

"龙虾模式"是本系统最核心的技术创新所在。这一命名源于龙虾的神经生物学特点——龙虾拥有一条腹神经索，其中包含若干对神经节，最大的前端神经节（"脑"）作为中枢，负责整合来自全身感官的信息，并发出指令协调各附肢（步足、螯足、游泳肢）执行捕食、逃逸、求偶等复杂行为。每对附肢都有一定程度的本地反射能力，但中枢神经节的调度确保所有附肢的行动协调一致。

将这一生物启发迁移到智能系统架构中，形成了本系统所主张的"中枢调度多组件"模式。在这个架构中，中枢大模型扮演"脑"的角色，负责全局规划和统一调度；各专用功能组件（离散数据翻译模型、边缘执行模型、监控程序、数据分析工具等）扮演"附肢"的角色，各自具备在特定领域的处理能力。中枢大模型不亲自执行任何底层操作——它发出调度指令，由消息总线精确地递送到目标组件，组件执行完成后将结果通过消息总线返回。

龙虾模式与传统人工智能系统架构有本质区别。传统架构通常是"一个模型解决所有问题"——例如一个大型语言模型不仅要理解语言，还要直接尝试控制飞行器（输出控制指令）。这种做法存在两个根本问题：一是一般性语言模型擅长语义理解但不擅长精确的数值控制，直接让它输出飞行参数存在精度风险；二是一般性大模型运行在云端，网络延迟和连接依赖使其不适合处理需要快速响应的飞行控制任务。龙虾模式通过"中枢智能 + 专用执行"的分层设计解决了这两个问题——让中枢大模型做它擅长的高层规划和语义理解，让专用组件做它们擅长的底层执行和数值计算，二者通过标准化接口协同而非互相替代。这种架构可以通俗地类比为一个"交响乐团"：中枢大模型是指挥，它了解整部交响乐的全局（任务目标与约束），但它不亲自演奏任何乐器；各专用组件是乐手，每个精通自己的乐器（动作翻译、目标点生成、数据监控、频谱分析等），听从指挥的调度并在恰当的时机演奏；消息总线是乐谱，确保所有乐手在统一的时间框架和格式标准下协同。指挥不需要会拉小提琴，小提琴手也不需要理解整部作品的配器——它们各司其职，通过标准化的乐谱语言完成协作。

这种架构还带来了一个重要的工程优势——功能扩展的"即插即用"。在传统一体化系统中，增加一个新功能可能意味着修改核心模型，涉及重新训练或大量代码调整。在龙虾模式中，增加一个新功能只需要实现一个新的组件并注册到消息总线上，中枢大模型随即可以通过工具调用机制发现和调度它。这为系统的持续成长和功能迭代提供了开放的基础框架。

## 6.2 双智能体分离与人在回路

本系统采用双智能体（中枢大模型和离散数据翻译模型）的架构划分，而非由一个模型同时处理人与机器的交互。这一设计的核心考量是**安全边界**——通过在不同推理步骤之间插入人工审核和结构化校验环节，防止大模型的"幻觉"问题传导到飞行控制层面。

大模型的"幻觉"是指模型生成的内容看起来合理流畅，但实际上包含事实性错误。在一般的问答场景中，幻觉可能只是给出一个错误的答案（可以纠正），但在飞行控制场景中，一个错误的飞行动作可能造成不可挽回的后果。双智能体分离从架构层面为幻觉建立了两道屏障。

第一道屏障在**中枢大模型与执行链路之间**。中枢大模型可以产生各种想法和建议——但它不直接操作飞行器。它生成的飞行方案以"提案"形式提交，由操作人员审核，操作人员可以批准、修改或驳回。假设中枢大模型出现了幻觉，在飞行方案中错误地认为"场地高度限为5米"（实际限高为3米），操作人员在审核方案卡片时会发现这一错误并纠正。中枢大模型不能跳过审核环节直接将方案送入执行链路——系统在设计上保证了"必须经人确认"的控制流。

第二道屏障在**翻译模型与飞控之间**。即使操作人员审核通过了含有潜在风险的飞行方案（操作人员也可能疏忽），方案在进入离散数据翻译模型后，被翻译为标准化的动作编码，编码本身受限于预定义的动作类型集合和参数范围约束（例如"上升高度"参数不能为空、"目标位置"必须在场地边界内）。翻译模型产出的动作编码还要经过边缘执行模型的安全夹紧处理（目标点不能超出边界和高度约束），以及飞控层自身的物理限制。层层校验确保即使前面环节出现偏差，后续环节仍能兜底。

这一设计与Saunders等（2026）提出的"关键动作不可自动执行"原则一脉相承，并在此基础上做了前推（在规划阶段就引入人工审核，而不是等到动作执行时再拦截）和后延（翻译层和飞控层的多层约束），形成了一个纵深安全体系。

## 6.3 动作编码与边缘执行模型

动作编码是本系统的"通用语"——在不同智能水平和不同硬件平台之间传递飞行意图的统一格式。它的设计精髓在于**将"做什么"与"怎么做"彻底解耦**。

在动作编码的设计中，一条动作指令被表达为三个部分的组合。第一部分是**动作类型**，用一个标识符描述飞行器要执行的基本动作——当前设计的动作类型包括：起飞（从地面升至指定高度）、悬停（在当前位置稳定保持指定时长）、上升/下降（向垂直方向移动指定距离）、水平移动（向指定二维坐标或方向移动指定距离）、偏航转弯（绕垂直轴旋转指定角度或朝向指定方位）、和返航（飞回预设的起降点或安全点）。第二部分是**数值参数**，给出每种动作类型所需的量化目标——对于起飞是"目标高度多少米"，对于水平移动是"目标位置的三维坐标"，对于偏航转弯是"目标角度的度数"，对于悬停是"持续时长多少秒"。第三部分是**安全约束参数**，附带本次动作执行的安全限制——最大速度、最大加速度、最大角速度、高度上限和下限等。这些约束值从全局默认配置文件中读取，也可根据任务需要由翻译模型按情况改写（例如在"谨慎模式"下自动收紧约束参数）。

与许多无人机系统直接将高层指令映射到低级控制量（电机转速、舵机角度）不同，本系统的动作编码停留在"目标点"这个抽象层次——它告诉飞行器"在空间中要去哪里"，而不关心"各台电机应该转多快"。这层抽象起到了至关重要的作用：它将飞行器的物理运动控制（由PX4飞控或等效系统负责，涉及复杂的运动学和动力学计算）与飞行任务规划（由中枢大模型和翻译模型负责，涉及语义理解和任务编排）彻底分开。飞控系统的研发团队可以专注于将"目标点"转换为安全稳定的飞行控制指令，无需理会目标点是来自人工操作还是智能系统生成；而智能系统的研发团队可以专注于如何从自然语言生成"准确、安全的目标点"，无需深入飞控的底层动力学。这种职责分离大幅降低了开发的耦合度和风险。

动作编码的"解耦"价值在边缘执行模型身上体现得最为充分。边缘执行模型不需要理解"操作人员为什么想要水平移动到(3,2,1)"，它只需要收到"动作类型=水平移动，目标位置=(3,2,1)"这条编码，然后结合当前风速和无人机位置算出具体的目标点。反过来，离散数据翻译模型不需要知道"当前风速多少、无人机在哪里"——它只需要按操作人员的意图生成正确的位置坐标。两个模型各自专注于自己擅长的事，通过动作编码这个固定接口对接。

这种解耦还意味着两个模型可以各自独立进化和替换。远期规划中将离散数据翻译模型从基于大模型的实现替换为更小更快的专用模型（见6.6节），这个替换不会影响边缘执行模型的任何代码。同样，将边缘执行模型从规则计算替换为神经网络推理（如5.3节所述），不需要翻译模型做任何适配——只要动作编码格式保持不变，下游实现可以任意替换。这种基于标准化接口的模块独立性，是系统架构长期可维护和可扩展的基石。

## 6.4 统一消息总线与组件注册

统一消息总线是本系统实现开放式架构的核心基础设施。如果把系统的各个组件比喻为一座城市的各个建筑，消息总线就是连接所有建筑的道路网络——任何一栋建筑都可以通过道路网络向任何另一栋建筑发送和接收包裹（消息），不需要知道对方建筑在哪条街、用什么方式运输。

从技术实现角度，消息总线提供了两层核心能力。第一层是**路由**——当一个组件A想要调用组件B的工具（例如中枢大模型调用傅里叶分析工具进行频谱分析），组件A不需要知道组件B在哪里。它只需告诉消息总线"请找组件B，让它执行工具Y，参数是这些"。消息总线查询组件注册表，得知组件B在当前进程的内存中、或在另一个进程通过套接字可达、或（远期）在机载电脑上通过网络可达，然后自动选择正确的传输方式将消息送达并取回结果。第二层是**格式统一**——不管消息走过的是内存指针、本地套接字还是网络链路，消息的结构格式是统一的。发送方和接收方永远按照相同的消息格式收发，无需为不同的通信场景编写不同的消息处理逻辑。

这种双层抽象带来的工程价值非常显著。在系统部署时，各个组件的物理位置可以根据实际需求灵活调整而无需修改代码：边缘执行模型可以先部署在服务器上与中枢同机（方便开发联调），验证基本逻辑正确后迁移到机载电脑（实现断网可飞），迁移过程只需要更新组件注册表的位置记录，不影响任何组件的业务逻辑。同样，当未来需要支持同时控制多架无人机时，只需启动多个边缘执行模型实例（每架无人机一个），在注册表中登记为"边缘执行模型-1号机""边缘执行模型-2号机"等不同组件名，中枢大模型和离散数据翻译模型不需要任何代码修改即可支持多机协同。

组件注册表是消息总线的"导航系统"。它是一张表，记载着系统中所有已注册的组件、每个组件的名称、它所提供的工具列表、以及它的物理位置。组件在系统启动时完成一次性注册。当一个新组件（例如一个新的数据分析工具）希望被纳入系统时，它需要做的事只有：完成自己的功能逻辑实现，按照规范的消息格式接收和回复消息，并在启动时向注册表登记自己的存在。中枢大模型随后即可通过工具调用机制发现并调度这个新组件——因为中枢大模型不直接与任何组件"绑定"，它只是告诉消息总线"某工具我可以用吗？"消息总线查询注册表得知"FFT分析工具由fft_analyzer组件提供，它在A进程内存中"即回应"可用"。中枢大模型无需做任何配置变更。

这种"注册即可用"的扩展机制，是开放式接口规范的核心价值所在——它赋予系统不断生长和适应新需求的能力，而不需要系统设计者在项目之初就预见所有未来功能。

## 6.5 双虚拟环境隔离与跨进程通信

本系统面临一个典型的版本冲突困境：智能中枢一侧需要使用较新版本的Python（3.10+）来承载最新的AI相关库（智能体框架、大模型调用接口等），而飞行控制一侧因为依赖ROS机器人操作系统（Noetic版本）而必须使用系统自带的Python 3.8，两者无法在同一环境中共存。强行将ROS升级或降级AI库版本都会导致兼容性问题。

解决方案是**双虚拟环境完全隔离**。在操作系统的支持下，创建两个完全独立的Python虚拟环境，各自安装各自需要的库，互不干扰。两个环境之间不共享任何Python库文件——环境A安装的库对环境B完全不可见，反之亦然。这种隔离方式比在同一环境内处理版本冲突要干净和安全得多。

两个环境之间的通信通过**Unix域套接字**实现。Unix域套接字是Unix/Linux操作系统原生支持的进程间通信机制，与网络套接字（TCP/IP）不同，它不通过网络协议栈，而是通过操作系统内核直接在内存在两个进程间传递数据。这使得Unix域套接字具有极低的通信延迟（微秒级别）和极高的吞吐能力，非常适合本系统中飞行数据高频上行（10赫兹）和指令下行对延迟的敏感性。

在安全性方面，Unix域套接字还有一个重要优势：它只允许同一台机器上的进程间通信，不接受来自网络的连接。这意味着智能中枢与执行桥之间的通信天然受到操作系统边界的保护——外部设备无法直接向套接字注入伪造的控制指令。这一安全特性与系统整体的纵深防御设计保持一致。

通信内容的序列化采用**MessagePack**格式——这是一种二进制序列化协议，类似JSON但更紧凑、解析更快。发送方将结构化的消息数据（格式如6.4节所述）打包为二进制数据块，在前面附加4个字节的长度前缀（标识整个数据块的总字节数）后通过套接字发送。接收方根据长度前缀读取完整的数据块，解包为标准格式的消息对象后递交给本地的消息总线分发。长度前缀的设计解决了"粘包"问题——即因为网络或管道缓冲导致的多条消息粘连在一起无法分离的经典通信陷阱。

双环境方案的关键约束是两块代码必须保持通信协议的绝对一致——发送方和接收方必须对消息格式的每一个字段有完全相同的理解，否则会出现解析失败或更隐蔽的数据错乱。为此，通信协议被作为独立文档"冻结"，任何对协议的修改都必须经版本号升级并同步更新两侧代码——不允许任何一方单方面修改。

## 6.6 蒸馏轻量化翻译模型（远期规划）

在系统的远期规划中，目前由云端大模型承担的"自然语言意图→动作编码翻译"功能，将由一个经过专门训练的轻量化模型替代。这一规划的背后有多重驱动力。

**降低对大模型的依赖。** 当前离散数据翻译模型依赖云端的DeepSeek大模型服务——每次翻译都需要将意图文本发送到云端、等待大模型推理、接收结果。这带来两个问题：一是网络延迟——翻译耗时在1至5秒，限制了系统整体响应速度；二是网络依赖性——在通信质量差或断网的场景下，翻译功能完全不可用。虽然系统设计了断网时的安全悬停兜底，但无法执行新的飞行任务始终是一个功能缺陷。轻量化模型在机载或地面电脑上本地运行，无需网络，推理延迟可降至50至200毫秒——比大模型快10倍以上。

**提升专业领域精确度。** 通用大模型虽然语言能力强大，但对民航试飞的专有术语和动作编排逻辑并不精通。通过用试飞过程中积累的真实数据进行"行为克隆"训练——即以人工审核通过的"意图文本→动作编码"配对为训练样本，让轻量化模型学习试飞场景中的翻译规律——训练出的专用模型在专业动作翻译上的准确性和稳定性将有更大保障。此外，轻量化模型权重固定，同样输入永远给出同样输出，具备完全的可复现性和可审计性，这是通用大模型因采样机制带来的不确定性所无法比拟的。

**可部署到机载电脑。** 轻量化模型的体量控制在数十至两百兆字节（对比通用大模型的数十至数百千兆字节），使用T5-small级别的编码器-解码器架构。模型以ONNX（开放神经网络交换格式）导出，搭配轻量级推理引擎（ONNX Runtime），不需要完整的深度学习框架即可运行。这使得模型可以部署到无人机机载电脑上，实现真正的"断网可飞"——无人机即使与地面站失去联系，机载的离散数据翻译模型和边缘执行模型仍能协同工作，根据最后的操作指令和本地环境/状态数据自主完成飞行任务。

轻量化翻译模型的引入不会改变任何系统组件与它的交互方式——因为它实现的是离散数据翻译模型的标准接口，中枢大模型和边缘执行模型无法感知翻译后端从大模型切换为了轻量化模型。这种"接口不变、实现替换"的设计原则正是开放式架构的核心思想。

还需要强调的是，蒸馏轻量化翻译模型与边缘执行模型是两个不同的概念，不应混淆。蒸馏轻量化翻译模型（远期）替换的是离散数据翻译模型——它负责"意图文本→动作编码"的翻译，解决的是"理解人要做什么"的问题。边缘执行模型则是始终位于翻译模型下游的独立组件——它负责"动作编码+环境+状态→目标点"的计算，解决的是"在当前条件下怎样达到目标"的问题。两者分别处于不同的抽象层级：翻译模型面向语义理解（自然语言→结构化编码），边缘执行模型面向运动控制（编码→物理目标）。它们各自可以独立训练、独立部署——翻译模型可部署在机载或地面，边缘执行模型始终部署在机载——而它们之间的接口（动作编码格式）是连接两者的标准化契约。

# 七、项目价值

## 7.1 降低试飞员认知负荷，提升效率与安全

试飞员在执行试飞任务时需要同时监控大量仪表、执行严格操作程序、与地面工程师保持沟通、并随时准备响应异常——其认知负荷之高，在航空领域仅极致条件下的紧急情况可与之相比。本项目通过"中枢大模型规划 + 操作人员审核"的模式，将繁琐的任务规划、数据查询、动作编排等工作交由智能系统处理，试飞员可以将精力集中在更高价值的决策判断和异常响应上。

在传统的试飞流程中，试飞员和试飞工程师需要在飞行前做大量的案头准备工作：查阅试飞大纲确认本次飞行的验证项目和参数范围、核查气象条件是否满足试飞要求、根据机场和空域条件编排飞行剖面、对照飞机性能手册核验各项数据是否在安全范围内。这些工作繁杂且易出错——任何一个参数的遗漏或误判都可能导致试飞结果无效，甚至在飞行中引发不安全事件。引入智能辅助系统后，大量的案头查阅和参数核验工作可以由系统自动完成：中枢大模型调取场地配置和历史飞行数据做参数比对，将比对结果清晰呈现在飞行方案中；边缘执行模型在动作编码层面自动夹紧安全边界；监控程序实时校验飞行数据。试飞员从"查阅和核验者"转变为"审核和决策者"，大幅降低了重复性脑力劳动的负荷，同时减少了人为疏忽的风险敞口。以一次典型试飞任务为例，从查阅场地信息、核实天气条件、编排机动动作序列、到核对安全约束参数——这原本可能需要试飞员和试飞工程师花费数十分钟准备的工作，在系统辅助下可压缩至数分钟甚至更短，且不会因人工疏忽而遗漏检查项。效率的提升意味着同样时间内可以完成更多的试飞任务，将试飞项目的整体周期缩短。安全性的提升体现在双重维度——系统永不疲劳、永不遗漏，而人始终保持最终决策权。

## 7.2 验证混合架构在飞行环境的可行性

现有的大模型智能体系统主要在信息检索、内容创作、代码辅助等数字领域得到了广泛应用，但在包含物理世界约束（运动学、动力学、通信延迟、传感器噪声）的真实飞行环境中，这种架构是否依然可靠、可用，尚未经过系统验证。数字领域的错误代价是"一个不准确的搜索结果"或"一段不通顺的文段"——这些错误可以被立即发现和纠正。而在飞行环境中，一个错误的控制指令可能导致飞行器姿态失稳甚至坠毁——错误的代价不可逆。因此，将大模型智能体引入航空领域不能仅从数字领域的成功经验中类推，必须经过实际的飞行验证。本项目以四旋翼无人机为载体，在仿真和实际飞行条件下全面检验"中枢大模型 → 动作编码 → 边缘执行模型 → 目标点 → 飞控执行"这一完整链路的效果。验证成果将为后续在更复杂飞行平台（包括载人飞机）上部署同类型系统提供"可行性证据"和"经验教训"两类关键信息——如果某环节在无人机上表现良好，它为民航迁移提供了正面证据；如果某环节在无人机上出现问题，它为民航设计指出了需要改进的方向。这种"低成本先导验证、高风险场景迭代"的路径，体现了工程实践中的审慎和高效。

## 7.3 开放式接口规范为新组件集成提供标准

在本项目规划完成时，一套完整的开放式接口规范已经成形。这套规范定义了任何新组件接入系统所需的全部标准和契约——统一的消息格式、组件注册与发现机制、工具调用与响应流程、跨进程通信协议、安全约束传递规则等。这套规范的价值不仅体现在本项目内部——它为系统的后续扩展和功能迭代提供了"即插即用"的标准化机制——更体现在对整个智能辅助系统领域的外溢效应。其他团队在研发类似的"中枢调度多组件"系统时，可以借鉴本项目的接口设计思路和标准化经验，减少从零设计协议的成本和风险。规范文档本身就是本项目具有长期价值的独立产出。

## 7.4 边缘执行模型为机载离线运行提供算法基础

本项目规划在无人机机载电脑上部署一个数兆字节规模的轻量神经网络作为边缘执行模型，输入离散环境数据、飞行状态参数和动作编码，输出路径规划器接纳的目标点。这一设计的意义不仅是为本项目当前验证服务——它探索了一条"将云端大模型的部分能力蒸馏到机载端侧"的可行路径。在民航应用场景中，飞机不能依赖持续的网络连接——跨洋飞行、远程航线、电磁干扰等情况都可能导致网络中断。届时，完全依赖云端大模型做飞行决策的系统将陷入困境，而拥有机载端侧推理能力的系统则能继续保持有限但关键的自主飞行能力。本项目积累的边缘执行模型训练方法、数据集构建经验、量化压缩技术路径、以及实测的推理延迟和精度数据，将为民航机载端侧智能部署提供第一手的技术参考。

## 7.5 为下一代智能航电探索技术路径

民航发〔2025〕24号文件提出的"人工智能+民航"愿景中，"机组智能辅助""飞行风险主动感知""飞行数据智能研判"都是下一代智能航电系统的关键能力模块。本项目所搭建的龙虾模式架构，恰好将这些能力模块有机地整合在一个统一的框架下——中枢大模型提供机组智能辅助（规划、分析、建议），数据监控程序提供飞行风险主动感知（实时异常检测），数据分析工具集提供飞行数据智能研判（自动化数据后处理）。本项目的系统架构在无人机平台上的验证效果，将为这些技术在载人航空器航空电子系统中的移植提供宝贵的先行经验。

从更宏观的技术发展角度看，航空电子系统正在经历从"自动化"到"智能化"的范式转变。传统航电系统的自动化是指按照预先编程的规则执行任务（例如自动驾驶仪按照预设航路飞行），而智能化是指系统能够理解情境、动态决策、与人类协作。龙虾模式为这一转变提供了具体的架构方案——它解决的核心问题是"如何在保留人类最高决策权的前提下，最大限度地发挥人工智能的辅助能力"。这恰恰是当前航空领域人工智能应用面临的最根本挑战。此外，论文提出的五条人-AI协同原则与本项目安全设计高度吻合，表明本项目在设计哲学上与国际前沿研究保持一致，具备向国际标准看齐的潜力。

## 7.6 经济与社会效益

在经济层面，试飞是飞机研制中成本最高的环节之一——每次试飞的直接成本包括燃油、机场使用费、人力工时、数据分析费用等，间接成本包括因试飞周期拉长而推迟的适航取证和交付时间。提升试飞效率、缩短试飞周期、减少因数据质量问题导致的重复试飞架次，将带来显著的成本节约。以当前国内外典型试飞项目的规模估算，一架新型号飞机的试飞周期可能跨越数年，累计试飞架次可达数百乃至上千次。如果智能辅助系统能将单次试飞的准备时间缩短百分之三十、将试飞后数据分析时间缩短一半，累积节约的人力和时间成本将是一个可观的数字。更深远的影响在于，更高效的试飞意味着新型号飞机更早取得适航证、更早投入航线运营——这不仅关系到制造商的经济回报，也影响到整个航空运输业的服务能力和旅客的出行体验。在社会层面，对民航智能辅助技术的前沿探索服务于两个重大社会议题：一是飞行安全——更智能的辅助系统意味着更低的运行风险和更好的异常处置能力，直接造福广大旅客；二是低空经济——随着无人机物流、城市空中交通等低空经济形态的快速兴起，对飞行器的智能化和自主化提出了新的要求，本项目的技术积累可以辐射到这些新兴领域。综上所述，项目在技术先进性、政策符合性、安全必要性、经济效益和社会效益等多个维度均具有显著价值。

# 八、规划进展与实施计划

## 8.1 已完成规划

截至本报告撰写时，项目已经完成了充分的规划设计工作，形成了一系列系统性成果。

**总体架构规格**确立了龙虾模式的顶层设计，明确定义了中枢大模型、离散数据翻译模型、边缘执行模型、监控程序、数据分析工具集、路径规划器、雷达感知、看板驱动器、历史查询等全部组件的功能边界、职责划分和交互关系。该规格同时涵盖了安全设计原则、双Python环境与通信链路、历史数据存储方案等全局性设计决策，为各子系统的开发提供了统一的依据。

**各子系统详细设计说明**分别为后端智能中枢、后端实时执行桥、前端交互界面、边缘执行模型、轻量化翻译模型（远期）等子系统编写了详尽的设计规格，内容包括模块目录结构、核心类与函数设计、数据流路径、接口契约、错误处理策略等。这些设计文档指导后续的编码实现，确保开发工作在明确的规格约束下进行，减少返工风险。

**跨进程通信协议定义**完整规范了智能中枢与执行桥之间的通信方式——传输层采用Unix域套接字加长度前缀帧，序列化层采用MessagePack二进制格式，消息格式采用统一的消息总线规范，并在约定版本号的管控下进行变更管理。

**开放式接口规范（独立文档）**定义了一套完整的组件接入标准，包括统一消息格式、组件注册与发现机制、工具调用与响应流程、传输分层抽象等核心规范，供后续新功能的快速集成和新组件开发者的参考。

**架构方案深化**完成了从初版方案到当前方案的架构重构——核心变化包括：引入边缘执行模型替代传统求解器、将碰撞检测从预编码场景改为在线雷达避障、将离散数据翻译模型的输出从连续航点改为结构化动作编码、新增前端数据看板的智能驱动机制。这次重构使系统架构更贴合机载端侧部署的需求，也为民航发〔2025〕24号文件鼓励的"依托无人驾驶航空器开展机载新技术前期试验验证"提供了完整的架构方案。

**测试与质量路线图**制定了分层测试策略（单元测试 / 集成测试 / 端到端测试 / 安全断连测试）、验收门槛、持续集成规划、项目进展跟踪约定等，建立了从开发到交付的质量保障框架。

## 8.2 实施计划

项目的开发与验证分为四个递进阶段，各阶段之间通过明确的验收标准衔接。

**阶段一（已完成）：需求与架构设计。** 输出全套架构规格和接口规范文档，完成关键技术的可行性论证。本阶段已经完成。

**阶段二（核心组件开发）：** 分别开发智能中枢一侧和飞行控制桥一侧的各个软件组件，实现组件间的基本通信和协同。智能中枢侧接入云端大模型实现中枢调度和翻译功能；飞行控制桥侧实现边缘执行模型的先导规则版本和数据监控程序；前端完成界面框架和核心视图的开发。在各组件单元测试通过后，进入集成联调阶段。阶段二的交付里程碑是：前端界面可打开并连接后端、智能中枢可接收自然语言输入并生成飞行方案、翻译模型可产生合法的动作编码序列、边缘执行模型的先导规则可生成不超出边界的目标点、监控程序可检测到模拟的异常数据并发出告警。这些核心功能各自独立验证通过后，进入阶段三系统级联调。

**阶段三（集成联调与地面仿真）：** 将所有组件连接起来，在纯软件仿真环境中打通"任务下发 → 规划审核 → 动作翻译 → 目标点生成 → 仿真执行 → 数据回传与监控 → 告警与处置建议"的完整闭环。本阶段的工作重心不是开发新功能，而是在已有组件之间消除接口不匹配（双方对同一字段理解有偏差）、修复时序问题（A组件期望B组件在收到消息后立即返回，但B有内部处理延迟）、优化性能瓶颈（如高频数据传输导致CPU占用过高等）。通过重复执行预定义的试飞任务序列（涵盖起飞、悬停、水平移动、偏航、返航等基础动作类型，以及复合动作序列），验证系统的基本完整性和跨组件协同的正确性。在本阶段发现并消除接口冲突、时序错误、性能瓶颈等集成问题。阶段三的验收标准是"S0至S7"八个联调步骤全部通过——从最基本的二进制兼容性测试到完整的端到端场景测试（包括模拟断连和异常告警场景）。

**阶段四（外场无人机演示与评估）：** 在实际无人机平台上进行飞行演示。测试不同工况下（多样化的任务类型、环境条件变化、模拟通信中断和组件故障等异常场景）的系统表现，收集关键性能指标——包括链路延迟（从操作人员批准方案到无人机开始执行动作的时间）、动作执行精度（边缘执行模型生成目标点的位置误差）、异常检测响应时间（从数据异常发生到监控程序发出告警的时间）、断连安全悬停时间等。全面评估系统在真实物理环境中的可行性与可靠性，并基于测试结果形成项目总结报告与改进建议。

## 8.3 风险与应对

在项目的实施过程中，以下风险因素被识别并制定了相应的应对策略。

**大模型服务的稳定性风险。** 系统核心功能依赖云端大模型服务的可用性和响应延迟。如果大模型服务不可用（服务端故障或网络中断），中枢大模型的规划和翻译功能将受到影响。应对措施包括：系统设计上保证断网时无人机自动安全悬停（不依赖大模型维持基本飞行安全）；在远期部署轻量化翻译模型实现本地推理，逐步消除对云端的依赖；为关键操作（如紧急中止）设计不经过大模型的快速处理路径。

**边缘执行模型的精度风险。** 在先导阶段，边缘执行模型使用规则计算方式，可能无法充分补偿复杂环境条件（如变化的风场）对飞行路径的影响。应对措施包括：先导阶段的规则设计预留充足的安全余量（保守的速度和高度限制）；远期切换为神经网络推理后，通过充分的训练数据覆盖多样化环境和状态条件来提升精度；设置边界夹紧和安全悬停双重兜底。

**组件间的接口兼容性风险。** 多组件并行开发时，各组件的开发者对接口契约的理解可能存在偏差，导致联调阶段出现大量不匹配问题。应对措施包括：冻结跨进程通信协议（以独立文档形式约定，任何更改须经版本号升级和双方确认）；在开发早期即建立联调测试框架（"S0至S7"验收步骤），尽早发现和修复接口不匹配。特别地，"S0"验证步骤要求在开发环境搭建完成后的第一时间，就验证智能中枢侧和执行桥侧的二进制通信链路是否可以互解——这是所有后续验证的前提，"S0"不通过就不进入任何功能联调。

**仿真与实机的差异风险。** 在纯软件仿真环境中验证通过的系统，在外场真实飞行中可能面临仿真未覆盖的物理效应（如地面效应、电池电压下降导致的动力衰减、GPS信号噪声等）。应对措施包括：在阶段二后期接入PX4软件在环仿真（逼真度高于纯运动学模拟），缩小仿真与实机的差距；在阶段三设计边界条件测试（极限风速、低电量等场景），提前暴露潜在问题。此外，在阶段四外场飞行时，采取保守渐进的安全策略——先在低高度、低速度下验证基本功能，确认无问题后再逐步扩展飞行包线。

**适航合规风险。** 虽然项目目前处于技术验证阶段，不需要取得适航认证，但若未来向载人航空器迁移，需确保系统的设计符合适航法规要求。应对措施包括：在架构设计中预留合规扩展能力（例如，将关键安全决策路径设计为"可被审定机构审查的独立逻辑"）；在安全设计原则中充分借鉴民航领域的成熟做法（人在回路权威、可审计性、失效安全）——这些原则也正是适航审定关注的核心理念。特别地，系统所有指令和响应的全量记录（对话记录表中的完整审计轨迹）为未来的适航审定的追溯性要求提供了数据基础。

| **龙虾模式** | 本系统提出的核心架构理念：由一个中枢大模型作为调度中心，通过统一消息总线动态协调多个专用组件，完成从任务理解到飞行执行的全流程。名称灵感来源于龙虾中枢神经节协调各附肢的生物学机制。 |

| **边缘执行模型** | 部署在飞行控制侧的轻量化模型，负责将离散数据翻译模型产出的动作编码与当前环境数据和无人机状态结合起来，计算出飞行控制系统可直接使用的目标点。先导阶段使用基于规则的计算方式，远期阶段替换为经过训练的轻量神经网络，并迁移到无人机机载电脑上本地运行。 |
| **动作编码** | 一种结构化的飞行指令格式，由三个部分组成：动作类型（如起飞、悬停、水平移动、偏航、返航等）、数值参数（目标高度、目标位置、持续时长、目标角度等）、安全约束参数（最大速度、高度上限等）。动作编码是高层意图与飞控层之间的"通用语"。 |
| **路径规划器** | 远期阶段的组件，接收边缘执行模型产出的目标点和雷达实时点云数据，进行在线避障路径计算，生成从当前位置到目标点的连续安全飞行轨迹。 |
| **统一消息总线** | 系统各组件间通信的核心基础设施，提供消息路由和格式统一两层核心能力。组件通过消息总线发送和接收标准格式的消息，无需感知通信的传输层细节（内存、套接字、网络）。 |
| **组件注册表** | 记录系统中所有可用组件及其提供工具列表的信息表。中枢大模型通过查询注册表发现可调用的组件。新组件在启动时登记，即可被中枢大模型发现和调度。 |
| **人在回路** | 一种核心安全原则：系统可以提出建议和分析，但涉及飞行安全的关键决策（方案批准、处置执行等）必须经过操作人员的审核和确认。 |
| **安全悬停** | 一种安全保护机制：当系统检测到通信中断或组件故障时，无人机自动在当前位姿维持稳定悬停，不执行任何新的飞行动作，直到接收到新的有效指令或操作人员介入。 |

# 试飞控制系统 — 架构概要

试飞控制系统是一个先导项目,目标是用大模型操控四旋翼无人机完成试飞任务。整个系统跑在一台 Linux 服务器上(内网),没有真实无人机,用 ROS 模拟。

## 五个组件

**前端** — 电脑和手机都能用的网页。支持三维轨迹图(可旋转)、二维图表、视频回传、大模型对话(打字或语音)。手机排版竖着堆,电脑左右分栏。

**后端 A(大脑)** — 跑大模型,负责理解人的意图。里面有两个 AI agent,各管各的:

- **Agent α** 是"遥控器"。键盘操控或人类批准的计划,它翻译成无人机能执行的目标点。
- **Agent β** 是"规划师"。翻看历史飞行记录,分析数据,写出飞行方案给人审。人不点头,方案不执行。

α 和 β 不直接说话,唯一的通道是"β 写方案 → 人看了同意 → 交给 α 执行"。这是故意设计的:大模型会幻觉,所以人必须在关键决策上把关。

**后端 B(执行桥)** — 和 ROS、飞行控制打交道。里面有个求解器,它拿 α 的目标点算出一条安全、不撞墙的飞行轨迹,再下发给无人机。

**模拟无人机** — 先做一个最简单的假无人机(收到指令就动,发回位置),跑通了再接真正的 PX4 飞控模拟。

**历史数据库** — 存每次飞行的轨迹、对话、环境数据。β 可以查历史来分析;前端可以翻看。

## 数据怎么流

```
人的任务(自然语言)
    → β 查场地、查历史 → 写出飞行方案
    → 人审核通过
    → α 翻译成"目标点+速度上限"
    → 求解器生成安全轨迹
    → 无人机执行
```

飞行中,人随时可以按键盘接管(向上/向下/悬停)。键盘指令放进下一个循环,不会中途打断当前这一步;但如果正在跑一段自动计划,键盘能直接打断它。

## 怎么连接的

- **A 和 B**:在同一台机器上,走 Unix socket(不走网络,快)。A 发给 B 的是 JSON(低频),B 发给 A 的是二进制(高频位姿)。
- **前端和 A**:混合协议——大模型的文字用 SSE 流式推送,位姿和键盘走 WebSocket,查历史用普通 HTTP。
- **B 和无人机**:用 ROS 的话题,标准做法,将来换成 PX4 只改话题前缀不改代码。

任何环节断开或失败,无人机自动切到安全悬停——不会出现"没人控制就乱飞"。

## Python 环境

系统里有两套 Python 虚拟环境:

- **A 侧用 Python 3.10+**(新),因为 AI 库要求新版本。
- **B 侧用 Python 3.8**(系统自带),因为 ROS Noetic 绑定了这个版本。

两套环境完全隔离,只通过上面说的 Unix socket 传数据,版本冲突不存在。

## 场地和安全

场地用配置文件定义——房间有多大、哪里有柱子、哪里有墙。支持方形、圆柱、球、平面、胶囊体等形状的障碍物。

环境条件(天气、风速、湿度)目前是人工在前端填的,提供了常用预设模板可以一键填充。

## 关键决策总结

- 大模型用 DeepSeek-V4-Flash(云端),agent 框架用 Pydantic AI
- 语音用讯飞(具体 API 后面再对接)
- 前端先原生开发(不引框架),手机用一键打包工具
- 数据库先用 SQLite(单文件、零运维),将来要升级再换 PostgreSQL
- 所有组件跑在一台机器上,WSL2 开发环境

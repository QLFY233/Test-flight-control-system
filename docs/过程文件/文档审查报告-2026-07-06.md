## 1. 硬冲突(spec 之间互相矛盾,实现会出错)

### C1. Session ID 格式不一致
- 总体架构 §5.1: `flight_sessions.id TEXT` → `"20260630-001"`(日期+序号),TrajectorySpec 示例 `task_id:"20260630-001"`。
- 后端 A §4.1: `POST /api/sessions` "新建试飞 session(返回 `YYYYMMDDHHMM`)" → 缺序号、含时分。
- 前端 §5.1/§5.2: 注释 "返回 id(YYYYMMDDHHMM)"+示例 "202606301430",但 `FlightSessionSummary.id` 注释又写 "20260630-001",BottomBar 也显示 "20260630-001"。
- 影响:`flight_sessions.id` ↔ `TrajectorySpec.task_id` ↔ `StatusMessage.taskId` 三处必须同值,目前格式漂移。
- 修复:统一为单一格式(建议 `YYYYMMDD-NNN`,如 `20260630-001`),同步 A §4.1、前端 §5.1/§5.2 注释、BottomBar 示例。

### C2. AlphaPage 仍带 ChatPanel 与"α 无对话"自相矛盾
- 多处声明 α 无对话:总体架构 §1.1/§2.3、A spec §2.3、前端开发计划 P3、前端详细设计第 70 行注释、§13.1 时序图。
- 但前端详细设计 §9.2 `AlphaPageCoordinator` 仍含:
  ```
  chatPanel: ChatPanel;             // Chat 对话+操控输入(打字/语音)
  private handleChatSend(text) // 发送 Chat 消息给 α (REST/SSE)
  ```
- 这是"删除键盘/遥控器、操控统一为 β Chat"重构的遗留。若实现按 §9.2,会再长出 α 对话框,违背龙虾模式。
- 修复:从 AlphaPageCoordinator 删除 `chatPanel`/`handleChatSend`;操控只在 BetaPage.ChatPanel 进入 β。

### C3. `approvePlan` 与 `approveProposal` 两套审核 API 并存,无主路径
- 总体架构权威路径:β 调 `propose_to_alpha(intent)` → `pending_proposal` → `/api/proposals/{id}/approve` → 注入 α。
- 但 REST 表另有 `POST /api/plan/approve`(A §4.1 第 310 行),前端 §5.2 注释自承"先导实现可只启用其一;若两者并存…"。
- FlightPlanCard 的 [批准] 走 `approvePlan`(传 FlightPlan 结构),而 propose 路径传 intent 字符串。两者进 α 的入参类型、是否落 conversations、是否触发 `metadata.approved` 都不明确,可能出现"人审了 FlightPlan 但没走 propose_to_alpha 工具",绕过 β 中枢调度。
- 修复:收敛为单一路径,或在两个 endpoint 上明确:FlightPlan↔proposal 的双向绑定、各自如何写 `conversations.metadata`,并明确 β 工具调用与 REST endpoint 的对应(谁触发 propose_to_alpha 工具,谁直接进 α 队列)。

### C4. 碰撞检测方法在总体架构 §5.3 与 B spec §4.4 不一致
| 形状 | 总体架构 §5.3 | B spec §4.4 |
|---|---|---|
| box | "轨迹段与 AABB/OBB 求交" | "采样点到 AABB 距离 ≤ keep_clear_distance" |
| cylinder | "段到圆柱轴心距离 ≤ radius" | "采样点到轴心 XY 距离 ≤ radius + keep_clear_distance(且 z∈[0,height])" |
| sphere | "段到球心距离 ≤ radius" | "采样点到球心距离 ≤ radius + keep_clear_distance" |
| plane | "轨迹点是否在法向禁止侧" | "采样点…(+ keep_clear_distance 余量)" |
- B spec 改为采样点法并加了 `keep_clear_distance` 余量;总体架构未同步,且 B spec §13 声称"逐字一致"实际不一致。
- 影响:实现者按总体架构 §5.3 写或按 B §4.4 写会得到不同碰撞结果。
- 修复:B spec 较新且可执行,以 B 为准,回写总体架构 §5.3 全表,并显式声明"先导用采样点法,段法为远期"。

## 2. 模糊 / 不一致(尚可实现,但要决策)

### A1. flight status 枚举域分裂
- DB `flight_sessions.status`= planned/executing/completed/aborted(4 值,总体架构 §5.1)。
- WS `StatusMessage.flightStatus` 与前端 store.flight.status = idle/hovering/planned/executing/completed/aborted(6 值)。
- 同名 `status`,语义不同:`idle`/`hovering` 是否落库?映射到哪一行?session 进入 executing 后再回到 hovering 怎么记账?
- 修复:明确"会话生命周期 status(落库,4 值)"与"实时飞行 flightStatus(WS,6 值)"的映射表,例如 `(idle→未开始/planned)、(hovering→executing)、(executing→executing)…`。

### A2. FlightPlan 在 SSE 流上的事件格式未在 A spec 定义
- 前端 §4.2 SSE 定义 `PlanOutput { type:'plan'; segments; summary }`,据此渲染 FlightPlanCard。
- A spec §4.2 只说"Pydantic AI UIAdapter 返回 StreamingResponse",未列 `plan` 事件字段。β `output_type=FlightPlan` 与 SSE 中 `PlanOutput` 的序列化规则缺失。
- 修复:A spec §4.2 补一节 SSE 事件枚举(`text`/`tool_call_start`/`tool_call_result`/`plan`),与前端 §4.2 逐字对齐。

### A3. `mode:'manual'|'auto'` 语义未定
- 人类经 β forward/propose 操控属 manual 还是 auto?α loop 执行预设段属 auto?无人定义。
- 修复:在总体架构 §2.2 约束/调度表后补 `mode` 取值与触发条件映射。

### A4. video_frame 视频源在先导期无来源
- 总体架构 §3.3 与 A spec §4.3 都有 `video_frame` 下行;冻结 §6 注"由 A 侧语音与同步模块产生"(此处把 video_frame 列入"非 B 产生"似乎是笔误,video 与语音无关系)。B 不产,前端 VideoPanel 需要源。先导用 ROS 假无人机无摄像头。
- 修复:明确"先导期 video_frame 暂不实现,VideoPanel 占位无信号;阶段2 PX4 + 相机接入再补",并修正冻结 §6 把 video_frame 与 voice_* 区分注释。

### A5. field.yaml 默认含 capsule 障碍物但 solver 先导不实现 capsule
- `config/field.yaml` 示例含 id=5 capsule(操作员站位);总体架构 §5.3 末"先导实现 box/cylinder/sphere/plane,capsule 后补",B spec §4.4 同。
- 加载含 capsule 的配置时求解器如何处理?当前未定义"未知/未实现 obstacle type"的兜底(忽略+上行 reject?启动失败?降级到全 keep_clear 球?)。
- 修复:B spec §4.4 补一句"未实现的 type→该障碍视为 `sphere(center, radius= keep_clear_distance + 最大边长/2)` 兜底碰撞或启动 reject",或把默认 field.yaml 的 capsule 改成 sphere。

### A6. 坐标系 y/z 交换约定只在一处注释
- 前端 §7.2 注释"Three.js Y-up,场地 (x,y,z) → Three.js (x,z,y)"。但 DroneModel.setTargetPosition(pos:[x,y,z])、TrajectoryLine、WaypointMarker 接口都按 [x,y,z] 传,谁负责交换没说。
- 修复:在 Scene3D 入口处统一约定"所有 store.drone.position 用场地系 (x,y,z),Scene3D 内部转 Three.js (x,z,y)",并写到 §7.1 Scene3D 接口。

### A7. `conversations.metadata.approved` 反作用未回填 A spec §5.5
- 小模型 α spec §12.1 明确要求 A spec §5.5 补 `metadata.approved`/`path` 标志,作为训练标的筛选依据。
- 但 A spec §5.5 当前文本未含此项。这是"反作用未落地"——小模型 spec 给 A spec 提了修改单,A 未执行。
- 修复:A spec §5.5 末补一句(小模型 spec 已给出建议文案)。

### A8. B→A `telemetry` event 默认不发与 B spec §2.2 表并列并存
- 冻结 §3 注"telemetry 可选,默认不发"。B spec §2.3 仍把 `pose` 与 `telemetry` 并列为"两个上行事件"。实现者会困惑要不要写 telemetry 分帧。
- 修复:B spec §2.3/§3.2 把 telemetry 标为"可选,先导不实现",与冻结对齐。

### A9. SSE / REST sendMessage 语义
- 前端 §4.1 SseManager 既有 `connect(endpoint)` 又有 `sendMessage(endpoint, text)`。SSE 单向,POST 才是"发+接收流式回复"。
- 修复:写明 `/api/chat/beta` 是一次 POST,Response 为 `text/event-stream`(server-sent events 流式);不存在"先建立 SSE 长连接再发消息"路径。

### A10. 紧急中止(abort)的下行链路未明
- 前端 BottomBar `abort-click` event、`abortSession(id)` REST(PATCH /sessions/{id}/abort)。
- 但总体架构 §3.2 A→B 只 `trajectory`/`abort`/`hover`/`ping`;A 收到 REST abort 后是发 `abort` 还是 `hover` 给 B?是否需要专用 WS 上行?(REST 与 α loop 时序?)
- 紧急悬停要求高实时,经 REST 触发再经 A 总线到 B 是否满足?未注明延迟预算。
- 修复:A spec §4.1 给 `POST /api/sessions/{id}/abort` 的内部处理(发 `abort` 或 `hover` 到 solver、是否清 α 队列、是否同步 set status=aborted)。

### A11. 环境(environment)如何进入 β/α 上下文
- 设置页注释说"经 α/β Chat 上下文传给大模型"。但 A spec β 工具只有 `query_environment(env_id)`(需 env_id,不能取当前)。β 自动注入到 prompt 还是工具调用未定。
- 修复:明确 environment_id 写入 flight_sessions,β 启动时自动把 environment.data 注入 system prompt 或作为 `get_current_environment()` 工具。

### A12. 移动端历史/β 入口缺失
- P8 移动端四 Tab[环境/任务/Chat/数据] 无"历史检索/筛选/发送到 β"入口,P8 备注里只是"折叠入口"未设计。原始 ui.md 要求移动端含历史检索。
- 修复:P8 增加移动端历史入口(如 Chat Tab 顶部"历史"按钮 → 半屏 HistoryPage),或 SettingsTab 内嵌。

### A13. AlphaOutputMessage.trajectory 是摘要还是完整 TrajectorySpec
- A spec §4.3/A 总体架构 §3.3:`alpha_output` 注"仅推送执行中的 TrajectorySpec 摘要";前端 §13.3 又用 `alpha_output` 来 `trajectoryLine.setPlanned(TrajectorySpec.segments)` 渲染规划轨迹——需要完整 segments。
- 修复:明确 `alpha_output.trajectory` 是**完整 TrajectorySpec**(前端规划层需要),还是**摘要**(前端再发 REST /api/current-pose 拉)。建议改为完整,并去掉 A spec "摘要"措辞。

### A14. 悬浮球快捷键触发与 Chat 输入框焦点耦合
- §12 异常表:"仅 `keyboardInputFocused===false` 时触发快捷键"。但悬浮球物理点击不应被输入框聚焦屏蔽(否则用户在 Chat 打字时无法按悬浮球)。`keyboardInputFocused` 应只屏蔽**键盘全局快捷键**,不屏蔽悬浮球点击。
- 修复:区分"键盘全局快捷键"与"悬浮球点击/扇叶点击"两条触发路径。

### A15. 录音 PCM 实现 MediaRecorder 风险
- A spec 待定项列"前端 PCM 采集具体实现(MediaRecorder vs AudioContext)"。但 MediaRecorder 默认 webm/opus,不能直接出 PCM 16k/16bit/mono。前端没有 AudioWorklet 重采样方案说明。这是实现期硬坑。
- 修复:前端 spec 补一节 PCM 采集(AudioContext + AudioWorklet,16k/16bit/mono,40ms 帧),或允许用 ScriptProcessor 兼容。

### A16. STT 末帧后是否自动触发 β SSE
- A spec §4.4 STT 步骤 9 "最终文本注入 β 上下文,同时开启 β SSE 流式回复"。问题是后端自动发起 SSE 而前端没有发 POST /api/chat/beta?还是 A 把识别结果作为一次 chat POST 自己处理?前端 §4.3 只有"POST /api/chat/beta"。语音自动发送与"可自动发送,可选"(ui.md)有歧义。
- 修复:明确语音末帧后由后端自动注入 β(无需前端再 POST),前端仅订阅已有 SSE;并区分"回填输入框等人按发送"与"自动发送"两种模式开关。

### A17. msgpack venv-B apt 未装但 spec 假设可用
- B spec §12 回填表:`python3-msgpack` Candidate 0.6.2-1 **本机未装**,需 `sudo apt install`。但 §1.2 又把 msgpack 0.6.2 写进"已锁定 ✅"。锁定≠已装。
- 修复:把状态标为"待装",或在.cli 启动脚本前置 `dpkg -l | grep -q python3-msgpack || sudo apt install -y python3-msgpack`。

### A18. `from`/`to` 在 pong 心跳特例的取值未冻结
- 冻结 §3 注 pong "不强约束,联调时对齐(可填 A/heartbeat/省略)"。这不是"冻结",留双侧各自实现会不一致。
- 修复:直接冻结 `to` 在 pong 时取值(如省略或固定 `"heartbeat"`),不要留"联调时定"。

### A19. Pydantic AI 2.0.0 / FastAPI 0.136.3 等版本未回填

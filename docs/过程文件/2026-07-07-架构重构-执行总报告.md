### 文件 1: 总体架构-design.md（重写量最大的文件）

**理由**: 总体架构是整个系统的"契约源头"。所有子系统 spec 都以它为准。求解器被去掉、新链路确立后，如果不把总体架构全文重写，后续各子系统 spec 会出现两个版本的链路描述互相矛盾的情况——例如 B spec 写"给求解器发 trajectory"，A spec 写"给 small_model 发 action"，两份 spec 描述同一条 IPC 链路但 tool 名不同，实现者看糊涂。

#### §2.2 ActionCommand schema（旧版"全链路约束体系"，完全替换）

**改了什么**: 整节从 TrajectorySpec（航点序列+global_constraints+segments+waypoints+三层约束层级）替换为 ActionCommand schema。新 schema 结构：

```json
{
  "task_id": "...",
  "schema_version": 2,
  "actions": [{ "code": "goto", "value": ..., "target": [...], "units": "m", "comment": "..." }],
  "safety_constraints": { "speed_max": 1.5, "ceiling": 2.5, "floor": 0.3, "boundary": [[0,0,0],[5,4,3]] }
}
```

新增"动作编码表"（9 类编码 + value 语义 + target 语义 + 说明），新编码可扩展："加新 code=动作编码表登记+α prompt 更新+小模型输入维度扩展,schema_version 不动,旧小模型遇未知编码 → 兜底 hover"。

`mode` mapping 保留不变。

之后写了一小段 `safety_constraints` 说明：`keep_clear_distance` 等碰撞相关约束废弃（交 ego-planner 阶段2/4 在线避障）。

**为什么这样改**:
- （要求 4）"修改 α 模型的输出，α 模型应该输出 json 格式的动作编码和数值"——这是最硬性的要求。旧 TrajectorySpec 是"航点序列"的概念（基于位置），新 ActionCommand 是"动作意图"的概念（基于语义动作，比如"飞到 (3,2,1)"而非"路径点是 (3,2,1)")。之所以从"航点"改为"动作编码"：
  - α 是 LLM，LLM 更适合产出语义动作（"上升"、"飞往"、"悬停"）而非精确的连续航点。
  - 动作编码是离散值（one-hot），正好满足"小模型需要离散输入"的要求。
  - "动作编码+数值"结构让 α 和小模型的接口清晰：α 说意图，小模型落目标点。
- "动作编码表"定义了 α 和小模型之间共享的编码词汇——两边必须对同一编码的理解一致。这是开放式架构的一部分（要求 6）。
- 编码可扩展：未来加新动作（如 `circle` 绕飞），只需登记新编码+更新 prompt+扩展小模型输入维度，不改总线协议（schema_version 不动）。
- `keep_clear_distance` 废弃是因为碰撞检测从预编场地（求解器）→在线雷达（ego-planner）后，"安全距离"是 ego-planner 内部参数，不再是上层下发的约束。speed_max/ceiling/floor 还在是因为先导没有 ego-planner 做物理限制，软约束必须由上层下发。

#### §2.3 α 动作翻译器（旧版"α 翻译器职责与 loop 仲裁"，完全重写）

**改了什么**:
1. 头部描述从"纯翻译器:…翻译成 TrajectorySpec"改为"纯动作翻译器:…翻译成 ActionCommand(动作编码+数值)"。加了一句"先导用 LLM,远期可换蒸馏小模型 α,接口留好"并区分两类小模型。
2. α loop 仲裁逻辑：伪码中的 `spec = await translate(...)`/`TrajectorySpec` → `cmd = await translate(...)`/`ActionCommand`。下游 `bus.call(to="solver", tool="trajectory")` → `bus.call(to="small_model", tool="generate_goal")`。
3. `current_trajectory_plan` → `current_action_plan`, `.has_remaining()`/`.next_segment()` → `.has_remaining()`/`.next_action()`。
4. 输出描述从"TrajectorySpec(经消息总线发求解器组件)"改为"ActionCommand(经消息总线发 small_model 组件 → 产目标点 → setpoint)"。
5. 接口名 `TrajectoryTranslator` → `ActionTranslator`。
6. 加 `env`（环境数据）作为 translate 入参（小模型需要离散环境输入）。

**为什么这样改**:
- （要求 4）α 输出从 TrajectorySpec 变为 ActionCommand，所以 α 的接口名、loop 逻辑、下游调用必须全部同步。
- "两类小模型区分"是为了防止混淆：蒸馏 α（远期替代 α LLM，在 A 侧）和端侧小模型（α 下游，在 B 侧/机载）是完全不同的组件，不写清楚会导致误把端侧小模型当成 α 的替换。
- 加 `env` 入参：（要求 2）端侧小模型需要离散环境输入作特征，所以 α 翻译时必须把环境也传下去——不管 α 用不用环境数据，传给小模型时不丢。

#### §2.4 β 工具集（旧版"β 工具集"，加一段新工具类）

**改了什么**: 在"数据分析"工具类之后，新增"数据看板驱动"工具类（3 个工具）：

| 工具 | 输入 | 输出 | 说明 |
|---|---|---|---|
| `dashboard_configure(panel_id, spec)` | 看板面板 ID + 展示规格 | 配置回执 | β 决定"展示哪些数据、什么格式" |
| `dashboard_set_filter(panel_id, filter)` | 面板 ID + 筛选条件 | 筛选后数据子集 | β 施加筛选器（时间/类型/阈值） |
| `dashboard_list_panels()` | 无 | 可用面板列表 | 供 β 告知人"可选看板" |

末尾加注："前端看板组件是订阅方,只渲染不下决策。"

**为什么这样改**:
- （要求 7）"前端要提供数据展示看板,具体用哪些形式展示什么格式的数据是确定好的,然后由 β 模型用工具调用,替换实际展示哪些数据(也许提供筛选器功能?)"——这句话有三层意思：
  1. 前端预先确定面板类型和展示形式（如高度时序=ECharts line、3D 轨迹=Scene3D overlay）
  2. β 通过工具调用决定 **在哪个面板展示什么数据**（替换内容）——这是 `dashboard_configure`
  3. β 还能施加筛选器——这是 `dashboard_set_filter`
- "前端只渲染不下决策"是安全边界：β 可以决定展示什么东西，但看板的渲染、数据获取、视图切换全部在前端完成，β 不接触前端渲染逻辑。

#### §3.2 L5 A↔B 消息契约（旧版 381-399 行）

**改了什么**: 消息契约表中：

| 旧 | 新 |
|---|---|
| `A→B call trajectory` TrajectorySpec 下发求解器 | `A→B call action` **ActionCommand 下发(替代旧 trajectory)** α 产出的动作编码,交 B 侧小模型生成目标点 |

2. `reject` 消息字段从 `segmentIndex` → `actionIndex`，原因描述从"求解器拒绝某段"改为"小模型/ego-planner 无法生成可达目标"

**为什么这样改**:

- alpha_output 的内容决定了前端 α 视图渲染什么——原来是 TrajectorySpec（航点+约束），现在是 ActionCommand（动作编码+目标点）。前端 store、3D 渲染、状态栏全都依赖这个格式。
